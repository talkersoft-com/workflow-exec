# Task 0005 — Arcade dictation + recording-nav

## Goal
Model dictation as an event-driven machine resolved by Go, and implement the recording-navigation
behavior with correct async-commit semantics.

## Steps
1. Build the **`dictation` machine** with a short-lived actor **per `job_id`**:
   `idle → recording → pending → confirmed | error`. `⌘D` toggles record/stop. Transitions are
   driven by Go's `dictation:event` (`result → confirmed`, `error → error`) — **never** assumed
   on send. Mic capture stays in the renderer; the WAV is dispatched through the existing IPC.
2. Add setting **`recordingNavBehavior = send (default) | lock`** with guards:
   - Navigate **within** the recording agent → allowed; recording continues.
   - Navigate to **another** agent → **send:** commit dictation to the **original** agent, then
     navigate (toast "Sent to [agent]"); **lock:** block (toast "Locked while recording").
   - **Esc** → cancel + **discard** (toast "Recording cancelled") — the only discard path.
3. **Async-commit:** "commit then navigate" stops recording, dispatches the job for the original
   agent, then navigation proceeds immediately. The per-agent/job actor **outlives navigation**;
   `result`/`error` is applied to the **originating** agent (not the now-current one). The toast
   is the dispatch ack; a later Go `error` surfaces against the originating agent — never dropped.

## Definition of done
The full recording-nav matrix verified end-to-end incl. a real Go round-trip: within-agent nav
keeps recording; cross-agent nav in `send` commits to the original (result/error lands on it even
after navigating away); `lock` blocks; Esc discards. Build green.
