# Task 0005 — Write DATA-MODEL-DELTAS.md

## Goal
Enumerate any database changes (columns, tables, indexes, constraints,
EntityPrefixRegistry entries) needed to support the API, web, and vorch plans.
If nothing is needed, the file must explicitly say so and explain why.

## Target repo
`planning` (writes `cloud-manager/planning/planning/cloud-manager-ansible/DATA-MODEL-DELTAS.md`)

## Required sections

### Problem
One paragraph. The DATA-MODEL.md tables cover the static shape, but the API /
web / vorch plans expose gaps where additional columns or indexes would
materially simplify the implementation. This file enumerates them as a single
migration shopping list so we don't sprinkle migration thinking across three
plans.

### Audit of derived needs
For each plan, list any data-model requirement it surfaces:

- **From API-PLAN**:
  - Does `Playbook` need an `active_revision_id` FK to pin "the current revision
    shown in the editor"? Decide and justify.
  - Does `RoleFile` need a unique index on `(role_id, file_path)` to enforce
    no duplicate paths within a role? (Same question for `PlaybookRoleFile` on
    `(playbook_role_id, file_path)`.)
  - Does the revisioning need an explicit "current revision pointer" or is
    `max(revision_number)` sufficient?
  - Does `AnsibleRole` need a `deleted_at` timestamptz for soft-delete, since
    revision history would otherwise be orphaned?
- **From WEB-PLAN**:
  - Does the file tree need a `RoleFile.parent_path` or similar to render fast,
    or is parsing `file_path` on the fly fine?
  - Does the revision drawer need a `RoleFileRevision.diff_summary` text
    column, or can the UI compute diffs client-side from content?
- **From VORCH-PLAN**:
  - Does `PlaybookRun` need a `log_path` text column to point at the on-disk
    log file (the existing `output` text column may be the wrong place for
    multi-MiB logs)?
  - Does `PlaybookRunTarget` need `started_at` and `completed_at` columns
    (currently only the parent `PlaybookRun` has them)?
  - Does the queue-message contract need a corresponding column like
    `PlaybookRun.queued_message_id` for dedup on redelivery? Justify whether
    idempotency check on `status` alone is enough.
  - Does `PlaybookRun` need a `worker_id` text column for multi-worker
    debugging?
  - Does the worker need a `PlaybookRun.cancel_requested_at` column for the
    cancel signal, or is `status = Cancelled` (a valid target state from
    `Running`) sufficient? Trade-off: a separate column lets the worker
    distinguish "cancel was requested" from "cancel completed", which matters
    if the cancel happens mid-target and the worker needs to finalize current
    target output before flipping to Cancelled.
  - Multi-VM runs make audit of "who triggered this" more interesting. The
    `Audit.created_by` column on `PlaybookRun` already exists — confirm it's
    populated correctly by `RunController.Apply` and the new multi-VM run
    endpoint. Probably no schema change, just verify.

For each item: **decide**. Either "add it, here's the rationale and the column
spec" or "skip it, here's why the existing shape suffices."

### Migration shopping list
For every "add it" decision above, list:
- Schema + table
- Column name + Postgres type + nullability + default
- New index name + columns + uniqueness
- New FK target + onDelete behavior
- EF Core: which entity class file is edited, which `DbContext` config changes
- One-line migration name (e.g. `AddActiveRevisionToPlaybook`)

Group into 1–N coherent migrations (don't make one per column).

### EntityPrefixRegistry additions
List any new entity types being introduced (probably none) and their proposed
prefixes. Confirm no collisions with the existing 13 entries.

### Backfill plan
For any non-nullable column added, describe the backfill (e.g. for
`Playbook.active_revision_id`: at migration time, set to the latest
`PlaybookRevision.id` per playbook, or NULL if none).

### Rollback notes
For each migration, state whether it's reversible without data loss. Flag
anything that isn't.

### If nothing is needed
If after the audit you conclude the existing schema is sufficient for all three
plans, write a one-paragraph "no database changes required" section that lists
each plan and the specific reason the current schema suffices for it.

## Definition of done
`DATA-MODEL-DELTAS.md` exists at
`cloud-manager/planning/planning/cloud-manager-ansible/DATA-MODEL-DELTAS.md`,
either contains a concrete migration shopping list with column-level specs OR
states "no changes required" with per-plan justification.
