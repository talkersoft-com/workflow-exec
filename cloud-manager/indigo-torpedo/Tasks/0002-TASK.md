## Required sections

### Problem
One paragraph. The data model is fully deployed but only `Playbook`,
`VmPlaybookAssignment`, `PlaybookRun`, and `FeatureFlag` have controllers. Every
other ansible-related entity has no API surface. Without it, the UI can't be
built and operators can only manage roles by direct SQL.

### Existing surface (do not redesign)
List the four controllers above with a one-line description each. Note what
they already cover.

### New endpoints
Group by resource. For each resource specify: route, HTTP verbs, request/response
DTOs, validation, error responses, and which existing service/repository layer
it extends. Cover:

- **Ansible roles (global library)** — `/api/v1/ansibleRole`
  - GET (list), GET/{id}, POST (create), PATCH/{id} (update name/desc),
    DELETE/{id} (reject 409 if any `PlaybookGlobalRoleRef` exists referencing it).
- **Role files** — `/api/v1/ansibleRole/{roleId}/file`
  - GET (list — returns file tree as flat array of `{file_path, public_id, modified}`),
    GET/{fileId} (read content), POST (create file with content),
    PATCH/{fileId} (update content — **creates a new `RoleFileRevision` row**
    in the same transaction with monotonically increasing `revision_number`),
    DELETE/{fileId}.
- **Role file revisions** — `/api/v1/ansibleRole/{roleId}/file/{fileId}/revision`
  - GET (list — newest first), GET/{revisionId} (read historic content),
    POST/{revisionId}/restore (creates a new revision whose content matches the
    chosen historic revision — never deletes history).
- **Playbook revisions** — `/api/v1/playbook/{playbookId}/revision`
  - GET (list), GET/{revisionId} (read). No POST — revisions are produced as a
    side effect of `PATCH /api/v1/playbook/{id}` when `content` changes.
  - Modify `PlaybookController.Update` so any `content` change creates a
    `PlaybookRevision` row in the same transaction.
- **Playbook ↔ global role refs** — `/api/v1/playbook/{playbookId}/globalRole`
  - GET (list attached roles), POST (attach — body `{ ansibleRoleId }`,
    rejects duplicates with 409), DELETE/{refId} (detach).
- **Playbook-local roles** — `/api/v1/playbook/{playbookId}/role`
  - GET, GET/{roleId}, POST, PATCH/{roleId}, DELETE/{roleId}.
  - Nested file + revision routes mirror the global-role versions exactly:
    `/api/v1/playbook/{playbookId}/role/{roleId}/file` and
    `/api/v1/playbook/{playbookId}/role/{roleId}/file/{fileId}/revision`.
  - Revision creation is the same trigger: PATCH content → new
    `PlaybookRoleFileRevision`.
- **Playbook run targets** — `/api/v1/run/{runId}/target`
  - GET (list per run), GET/{targetId} (status + stats + output).

- **Multi-VM run trigger (NEW)** — `/api/v1/playbook/{playbookId}/run`
  - POST with body `{ vmIds: string[], varsOverride?: object }`. Creates one
    `PlaybookRun` row plus one `PlaybookRunTarget` per `vmId` in the same
    transaction, snapshots vars onto each target, publishes a single run
    message to the queue, returns `{ runId }` with 202 Accepted.
  - Reject 400 if `vmIds` is empty, any vmId is unknown, or the playbook has
    no resolvable content. If a vm has an existing assignment for this
    playbook, inherit `vars_override` from the assignment and record
    `assignment_id` on the target; otherwise leave `assignment_id` null
    (ad-hoc target).
  - Existing single-VM `POST /api/v1/virtualmachine/{vmId}/playbook/{asgnId}/apply`
    stays — it's the assignment-driven path. The new endpoint is the
    explicit-multi-target path.

- **Run cancellation (NEW)** — `/api/v1/run/{runId}/cancel`
  - POST sets `PlaybookRun.status = Cancelled` if currently `Queued` or
    `Running`. If `Running`, publishes a cancel message (or sets a column the
    worker polls — pick one in the plan body; document the chosen mechanism).
  - Returns 200 with the updated run; 409 if the run is already terminal.

### Cross-cutting

- Every new controller is decorated with `[RequireFeatureFlag("playbooks")]`.
- Every new entity gets a DTO in `src/Models/CloudManager.DTO/Models/` and an
  AutoMapper profile entry under `Services/CloudManager.Data.Services/MappingProfiles/`.
- Public IDs on the wire — use existing prefixes from `EntityPrefixRegistry`:
  `ansr`, `rf`, `rfrev`, `pbrev`, `pgr`, `plr`, `plrf`, `plrfr`, `prt`.
- Validation: reject file paths containing `..` or starting with `/`; cap
  individual file content at a configurable byte limit (suggest 1 MiB).
- Error model: stay consistent with existing controllers (`{ Message: ... }`
  body, HTTP status codes already in use — 400 / 404 / 409 / 500).

### Service-layer hooks

- `RoleFileService.UpdateAsync` and `PlaybookRoleFileService.UpdateAsync` must,
  in the same EF transaction:
  1. Append a revision row with `revision_number = max(existing) + 1`.
  2. If `file_path == "meta/argument_specs.yml"`, invoke
     `ArgumentSpecsTranslator` to re-derive `Playbook.argument_schema` for
     **every playbook that references this role** (global: via
     `PlaybookGlobalRoleRef`; local: just the owning playbook).
- `PlaybookService.UpdateAsync` must append a `PlaybookRevision` row when
  `content` changes, with `revision_number = max(existing) + 1`.

### Materialization API (for vorch-service)

- Add `GET /api/v1/playbook/{id}/materialized?revision={revisionId}` that
  returns:
  - `files: [{ relative_path, content }, ...]` — the full directory tree:
    playbook content + all referenced global roles' files at the role's current
    head + all local-role files.
  - `secret_inputs: [{ extravar_name, vault_path }, ...]` — every input in
    `meta/argument_specs.yml` whose key is annotated with `x-secret-path`
    (vendor extension). Worker uses this manifest to resolve secrets with
    its OWN Vault token and merge them into extra-vars at run time. The API
    does NOT resolve the secret values — it only surfaces the paths.
  - `playbook_public_id` — for Vault prefix construction.
- Document how `playbook_revision_id` on `PlaybookRun` pins which playbook
  revision is materialized; the role files come from current head unless
  `DATA-MODEL-DELTAS.md` introduces per-run role-revision pinning.

### OpenAPI / Swagger

- All new endpoints documented via XML doc comments so the existing Swashbuckle
  generation surfaces them.

### Authorization

- All routes gated by the existing feature flag; no per-user RBAC in this plan
  (deferred — note as out-of-scope).

### Phases

Order from smallest shippable slice to complete feature. Suggested skeleton:
1. AnsibleRole + RoleFile CRUD (no revisions, no schema hook).
2. Add revision write-side: PATCH creates revisions automatically.
3. Add Playbook revisions + revision read-side routes.
4. Add PlaybookGlobalRoleRef and the schema-update hook.
5. Add PlaybookRole / PlaybookRoleFile / PlaybookRoleFileRevision.
6. Add PlaybookRunTarget reads.
7. Add the materialization endpoint for vorch (with `secret_inputs` manifest).
8. Add multi-VM run-trigger endpoint.
9. Add run-cancel endpoint.
