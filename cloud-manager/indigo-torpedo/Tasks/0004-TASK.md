## Required sections

### Problem
One paragraph. `RunController.Apply` enqueues run messages to RabbitMQ today,
but nothing on the worker side consumes them. The existing
`PLAYBOOK-INTEGRATION-PLAN.md` recommends a Python worker
(`cloud-manager-worker/`), but the team is shipping Go elsewhere
(`vorch-service` handles VM lifecycle). This plan resolves the worker-language
question and specifies the consumer.

### Python-vs-Go decision (resolve this explicitly)
- Restate the trade-off: `ansible-runner` is a Python library, so a Python
  worker calls it in-process; a Go worker shells out to the `ansible-runner`
  CLI as a subprocess.
- Recommendation: **Go + subprocess** inside `vorch-service`, for these
  reasons (state each in 1–2 lines):
  - Operational uniformity with the existing Go RabbitMQ consumer
  - One service to deploy/monitor/restart instead of two
  - `ansible-runner` CLI is a stable, documented surface
  - JSON event streaming via `ansible-runner --json` is already structured
- Disposition for `cloud-manager-worker/`: **delete the scaffold** once the Go
  consumer is in place. State this and add a follow-on task: remove
  `cloud-manager-api/cloud-manager-worker/` from the repo when VORCH-PLAN
  Phase 1 ships.

### Queue + message contract
- Queue name: confirm against `IPlaybookRunPublisher` implementation (likely
  `playbook-runs`).
- Message body: minimal — `{ "runId": "run_xxxxx" }`. The worker reads
  everything else from clouddb via the materialization API
  (`GET /api/v1/playbook/{id}/materialized?revision=...`).
- Place the message struct in `vorch-lib/` so the publisher (C#) and consumer
  (Go) can agree by shape (Go struct + JSON tag conventions documented; C# DTO
  on the publisher side; no shared codegen).
- Cancel signal: also subscribe to a `playbook-run-cancel` queue (or fan-out
  exchange) carrying `{ "runId": "run_xxxxx" }`. Worker maintains an in-process
  map of `runId → context.CancelFunc`; on cancel message, cancels the matching
  run's context. The subprocess gets SIGTERM, then SIGKILL after a grace
  period. Final `PlaybookRun.status` is set to `Cancelled`. Either this or a
  DB-poll approach is acceptable — pick one in the plan body.
- ACK strategy: ack only after `PlaybookRun.status` reaches a terminal state
  (Succeeded / Failed / Cancelled). On crash mid-run, the message is
  redelivered and the worker reconciles by checking current run status before
  re-executing.

### Vault token strategy (two distinct tokens per run)

The worker uses TWO different Vault tokens during a run. State this explicitly
in the plan body — it's easy to get wrong.

1. **Worker's own token (long-lived, env-supplied)** — `VAULT_TOKEN` env var.
   Used for:
   - Reading per-VM SSH private keys at the logical path
     `cloudmanager/vm/instances/<vm_pub>/ssh_key` (KV v2 — the API call goes
     to `/v1/cloudmanager/data/vm/instances/<vm_pub>/ssh_key`; the `data/`
     segment is API-path-only, never part of the stored secret path).
   - Resolving `x-secret-path` input values from the materialization manifest's
     `secret_inputs[]` (read-only, with the worker's own auth).
   - Minting the per-run child token via `POST /v1/auth/token/create/playbook-run`.
2. **Per-run child token (short-lived, minted per run)** — minted from the
   `playbook-run` token role established by `ANSIBLE-INSTALL-PLAN.md`. Scoped
   to per-(vm, playbook) Vault prefix via templated policy. Used ONLY by
   ansible-runner as `VAULT_TOKEN` in the subprocess env, so the playbook can
   `vault_write` its outputs back. Revoked when the run finishes.

The orchestrator (cloud-manager-api) does NOT mint these tokens anymore in the
new design — the worker does, because the worker is what calls ansible-runner.
The existing `RunController.GetSecret` (which reads published secrets for the
UI) keeps using the API's own `VAULT_TOKEN` and is unchanged.

### Worker Vault auth method

- **Recommended (Phase 1):** static service token via `VAULT_TOKEN` env var,
  matching the existing cloud-manager-api pattern. Documented in the plan as
  the v1 choice for operational parity.
- **Future migration:** AppRole auth so the worker can rotate its own creds
  without operator intervention. Call this out as a follow-on plan, not a
  blocker for v1.

### Per-message flow (numbered)
1. Pull `runId` from message envelope.
2. Fetch run via API (`GET /api/v1/run/{runId}`) — confirms it exists and
   is in `Queued` state. If already terminal, ack and drop.
3. Mark `PlaybookRun.status = Running, started_at = now()` via API.
4. Resolve the run's playbook + revision via the materialization API. Receive
   `files[]` + `secret_inputs[]` + `playbook_public_id`.
5. Create a tmpfs work directory (`/var/lib/cloud-manager/playbook-runs/<runId>/`).
   Materialize the manifest to disk: write each `{relative_path, content}` as
   files, mkdir parents.
6. Resolve secrets ONCE per run (they're per-playbook, not per-target): for
   each entry in `secret_inputs[]`, use the worker's own token to read the
   value from Vault. Hold the resolved map in memory.
7. For each `PlaybookRunTarget` row:
   a. Pull per-VM SSH private key from Vault (worker's own token), write to
      `env/ssh_key` (0600).
   b. Mint a per-run child token from the `playbook-run` role (worker's own
      token authority), TTL 1h, metadata `{vm_public_id, playbook_public_id}`.
   c. Build `env/extravars` = `{ secrets_prefix, vm_public_id,
      playbook_public_id, ...vars_override_snapshot, ...resolved_secret_inputs }`
      (the resolved secrets from step 6 are merged in here as ordinary
      extra-vars; the playbook should mark tasks consuming them `no_log: true`).
   d. Build `env/envvars` = `{ VAULT_ADDR, VAULT_TOKEN: <child token> }`.
   e. Mark `PlaybookRunTarget.status = Running, started_at = now()`.
   f. Exec `ansible-runner run <workdir> --json` as a subprocess. Stream stdout
      line-by-line into the run log file at
      `/var/log/cloud-manager/playbook-runs/<runId>.log` and update
      `PlaybookRunTarget.{status, stats, output}` periodically.
   g. Check for cancel between targets (and after subprocess exit per target):
      if `PlaybookRun.status == Cancelled` or our context was cancelled, break
      the loop and mark remaining targets as `Cancelled`.
   h. On subprocess exit: list `cloudmanager/vm/instances/<vm_pub>/<pb_pub>/`
      in Vault, write the result to `PlaybookRun.secrets_published`.
   i. Revoke the child token.
8. Aggregate per-target statuses → overall `PlaybookRun.status`:
   - All Succeeded → `Succeeded`
   - Any Failed → `Failed`
   - Any Cancelled and none Failed → `Cancelled`
   - Mixed Succeeded/Failed → `Failed`
9. Set `completed_at`, write final `stats`, update
   `VmPlaybookAssignment.{last_run_id, last_run_status, last_applied_at}`
   for each target's `assignment_id` (if non-null).
10. Shred + delete the workdir.
11. Ack the message.

### Health probe
- Goroutine on `vorch-service` startup, ticks every N minutes (default 5).
- Each tick: exec `ansible-runner --version`. On success, `PATCH /api/v1/admin/feature-flags/playbooks-healthy` (or whatever the existing
  feature-flag endpoint shape is — confirm against `FeatureFlagsController`)
  with `{ healthy: true, last_probed_at: now() }`. On failure: `healthy: false`
  with the error message.
- Also probed at boot so a restart re-evaluates immediately.

### Shared types in vorch-lib
- `PlaybookRunMessage { RunID string }` — wire format for the queue
  (Go struct + JSON tags; C# `PlaybookRunMessage` DTO on the publisher side
  must match field names case-insensitively).
- `MaterializedPlaybook { Files []MaterializedFile }` and
  `MaterializedFile { RelativePath string; Content string }` — response from
  the API materialization endpoint.
- `PlaybookRunStatus` — string constants matching the enum string values
  emitted by the API (`"queued"`, `"running"`, `"succeeded"`, etc.).

### Configuration
- Where settings live (env vars vs config file): match existing vorch-service
  conventions in `vorch-service/config/`.
- New settings needed: `ANSIBLE_RUNNER_BIN` (default `ansible-runner`),
  `PLAYBOOK_WORKDIR_ROOT`, `PLAYBOOK_LOG_DIR`, `VAULT_ADDR`,
  `CLOUD_MANAGER_API_URL`.

### Failure modes + recovery
- Subprocess hangs → kill after configurable timeout (default 2h, matches the
  Vault token TTL ceiling from `ANSIBLE-INSTALL-PLAN.md`).
- API unreachable mid-run → retry with backoff; if still failing after N tries,
  log and leave the message unacked (RabbitMQ will redeliver).
- Worker crash mid-run → see ACK strategy above; idempotent retry checks run
  status first.
- Disk full → fail fast at materialize step, no partial state.

### Phases
1. Bare consumer: subscribe to queue, fetch run, log message, ack. No execution.
2. Materialize playbook to disk from the API; no run yet.
3. Run ansible-runner against a single target with hard-coded SSH key (no Vault).
4. Vault integration: worker's own token reads SSH key + resolves `secret_inputs`.
5. Mint per-run child token, hand to ansible-runner subprocess as VAULT_TOKEN.
6. Multi-target loop + per-target status updates.
7. Cancel subscription + context-cancellation + final status aggregation.
8. Health probe + feature_flags.playbooks.healthy.
9. Delete `cloud-manager-api/cloud-manager-worker/` Python scaffold.
