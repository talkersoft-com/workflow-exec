# Task 0003 — Write WEB-PLAN.md

## Goal
Design the web UI for ansible management in cloud-manager-web as a **dedicated,
self-contained Ansible section** — fully separated from the VM management UI.
Ansible has no business on VM pages. The existing scattered ansible UI (VmPlaybooks
component, ansible sections on VmDetailPage) is replaced entirely by a new
top-level Ansible hub with its own nav, sub-pages, and workflows.

## Required sections

### Problem
One paragraph. The existing ansible UI is sloppy: ansible controls are embedded
inside VM detail pages (`VmPlaybooks` component, ansible sections on
`VmDetailPage`), mixing two unrelated concerns. A VM has no business showing
ansible content. The UI also lacks a YAML editor, role library management, a
dedicated run-trigger workflow, and a proper history view. This plan redesigns
the ansible surface from scratch as a clean, self-contained section of the app.

### Architectural decision — clean separation
State this explicitly. Ansible is managed from the **Ansible section only**.
No ansible UI on VM pages. Specifically:
- The `VmPlaybooks` component is **deleted**.
- Any ansible-related panels, tabs, or buttons on `VmDetailPage` are **removed**.
- A VM assignment is managed from the Ansible side (choose a playbook, attach
  VMs to it) — not from the VM side.
- `PlaybooksPage` and `PlaybookDetailPage` are **relocated or replaced** under
  the `/ansible` prefix.

### Code editor decision
Pick one and justify in 3–5 lines based on bundle size, YAML support, search,
diff, and the existing Vite + React + TS stack:
- **Monaco** (VS Code editor) — heavy bundle (~2 MB gzipped), best YAML/JSON
  support, built-in diff view, familiar to developers
- **CodeMirror 6** — lighter, modular, good YAML mode, no built-in diff
- **textarea + Prism for read-only diffs** — smallest, weakest UX
Recommend Monaco unless a strong case is made otherwise. State the bundle-size
impact and mitigation strategy (dynamic import on editor routes only).

### Nav structure
Add a top-level **"Ansible"** entry to the app nav, gated on
`feature_flags.playbooks.enabled` via the existing `FeatureGate` component.
The Ansible section has an internal sub-nav with these entries:
- **Playbooks** — list and edit playbooks
- **Roles** — global role library
- **Run** — trigger a playbook run
- **History** — all run history

Also add a top-level **"Settings"** entry that is **always visible** (not
flag-gated). Settings hosts feature-flag management — the operator must be
able to enable or disable the `playbooks` flag from the UI without an
out-of-band `curl` call.

### New routes
All routes nested under `/ansible`, all gated via `FeatureGate` with the
`playbooks` flag.

| Route | Page | Purpose |
|-------|------|---------|
| `/ansible` | redirect → `/ansible/playbooks` | Entry point |
| `/ansible/playbooks` | `AnsiblePlaybooksPage` | List all playbooks; create new |
| `/ansible/playbooks/:playbookId` | `AnsiblePlaybookDetailPage` | Playbook overview, attached roles, revision history |
| `/ansible/playbooks/:playbookId/edit` | `AnsiblePlaybookEditorPage` | YAML editor + revision drawer + attached-roles panel |
| `/ansible/playbooks/:playbookId/roles/:roleId` | `AnsiblePlaybookLocalRolePage` | Playbook-local role file tree + editor |
| `/ansible/roles` | `AnsibleRolesPage` | Global role library — list, create, delete |
| `/ansible/roles/:roleId` | `AnsibleRoleDetailPage` | File tree + editor + revision drawer |
| `/ansible/run` | `AnsibleRunPage` | Choose a playbook, select target VMs, set var overrides, trigger run |
| `/ansible/history` | `AnsibleHistoryPage` | All run history; filter by playbook or by VM |
| `/ansible/history/:runId` | `AnsibleRunDetailPage` | Run detail — per-target status, output; live updates during active runs |
| `/settings` | `SettingsPage` | Always visible (not flag-gated) — feature flag toggles, future global config |

### New pages
- **`AnsiblePlaybooksPage`** — replaces `PlaybooksPage`. List + search +
  create. Each row links to `AnsiblePlaybookDetailPage`.
- **`AnsiblePlaybookDetailPage`** — replaces `PlaybookDetailPage`. Shows
  metadata, argument schema, revision timeline, attached global roles,
  local roles list. "Edit YAML" button → `AnsiblePlaybookEditorPage`.
- **`AnsiblePlaybookEditorPage`** — full YAML editor with `RevisionHistoryDrawer`
  and `AttachedRolesPanel`. Save UX as specified below.
- **`AnsiblePlaybookLocalRolePage`** — file tree + editor for a playbook-local
  role; mirrors `AnsibleRoleDetailPage` shape.
- **`AnsibleRolesPage`** — list of global `AnsibleRole` entities; create + delete.
- **`AnsibleRoleDetailPage`** — `RoleFileTree` on left, `RoleFileEditor` on right,
  `RevisionHistoryDrawer` as slide-out.
- **`AnsibleRunPage`** — two-step form: (1) pick a playbook from a searchable
  dropdown, (2) pick one or more VMs from a multi-select list with optional
  per-VM var overrides. "Run" button calls `POST /api/v1/run/apply` (or
  equivalent per API-PLAN). On submit, redirect to
  `AnsibleRunDetailPage` for the new run.
- **`AnsibleHistoryPage`** — paginated table of all `PlaybookRun` records.
  Two filter modes selectable via tabs or dropdown:
  - **By playbook** — group or filter runs by playbook name
  - **By VM** — group or filter runs by target VM
  Each row shows run status, started_at, duration, target count. Click → `AnsibleRunDetailPage`.
- **`AnsibleRunDetailPage`** — replaces `RunDetailPage` for ansible runs.
  Shows overall status + per-target table (`RunTargetsList`). When run status
  is `Running`, auto-polls every N seconds to show live status and streaming
  output per target. Polling stops when run reaches a terminal state.
  Includes the `RunCancelButton` (visible only when status is `Queued` or
  `Running`) so the operator can abort active runs from the UI.
- **`SettingsPage`** — always-visible top-level page at `/settings`. Hosts the
  `FeatureFlagToggle` for every entry returned by `GET /api/v1/feature-flags`
  (today: just `playbooks`; designed to scale as more flags land). Shows the
  current `enabled` state, `healthy` indicator, and `last_probed_at` per flag.
  Toggling calls `PATCH /api/v1/admin/feature-flags/{key}` and refetches the
  flags list on success. This is the operator's in-UI control to enable or
  disable the ansible section without using `curl`.

### New components
- `RoleFileTree` — collapsible file-path navigator. Props: `files[]`,
  `selectedPath`, `onSelect`. Shared between global and local role pages.
- `RoleFileEditor` — wraps chosen editor. Props: `path`, `content`, `dirty`,
  `onSave`. PATCH on save; success invalidates revision list.
- `RevisionHistoryDrawer` — slide-out listing revisions newest-first with
  content preview. "Restore" button calls POST `/restore`. Used on role file
  pages and playbook editor.
- `AttachedRolesPanel` — tabs: "Global roles" (attach/detach via picker) and
  "Local roles" (list with links to local role editor).
- `RunTargetsList` — table of `PlaybookRunTarget` rows with status badge,
  stats summary, collapsible output. Supports live refresh when run is active.
- `VmMultiSelect` — searchable multi-select of VMs for the run trigger form.
  Shows VM name + host. Supports per-VM var override expansion.
- `RunHistoryFilters` — tab/dropdown switcher for "by playbook" vs "by VM"
  filtering on `AnsibleHistoryPage`.
- `AttachedVmsPanel` — reverse-direction view used on `AnsiblePlaybookDetailPage`
  and `AnsiblePlaybookEditorPage`. Lists every VM the playbook is attached to
  via `VmPlaybookAssignment` (vm name, last run status, last applied at). This
  is the from-the-playbook-side view of the many-to-many relationship; the
  VM-side view does NOT exist anymore (deleted per the clean-separation
  decision above). This panel is how an operator answers "which VMs is this
  playbook deployed on?" without leaving the Ansible section.
- `RunCancelButton` — used on `AnsibleRunDetailPage`. Visible only when run
  status is `Queued` or `Running`. Calls `POST /api/v1/run/{id}/cancel` and
  triggers a re-fetch. Disabled with a tooltip when the run is already
  terminal.
- `FeatureFlagToggle` — used inside `SettingsPage`. Renders a labeled switch
  per flag, plus the `healthy` and `last_probed_at` indicators next to each
  flag so the operator can see "enabled but broken" states distinctly. Toggle
  calls `PATCH /api/v1/admin/feature-flags/{key}`.

- Any ansible-related panels, tabs, or sections in `VmDetailPage` — **remove**.
- `PlaybooksPage.tsx` and `PlaybookDetailPage.tsx` — **replace** with the new

### New Redux slices
- `ansiblePlaybooksSlice` — list/get/create/update/delete; caches revision list.
- `ansibleRolesSlice` — global role library; caches file tree and per-file content.
- `roleFileRevisionsSlice` — keyed by `(roleId, fileId)`; revision lists + historic content.
- `playbookRevisionsSlice` — keyed by `playbookId`.
- `playbookLocalRolesSlice` — keyed by `playbookId`.
- `runTargetsSlice` — keyed by `runId`; supports polling updates.
- `ansibleRunsSlice` — run list for history page; filter params as slice state.
- `featureFlagsSlice` — fetches `GET /api/v1/feature-flags` on app load and
  on a polling interval (so the UI tracks `healthy` state). Powers both the
  `FeatureGate` checks across the Ansible section and the `FeatureFlagToggle`
  on `SettingsPage`. If a slice already exists for `FeatureGate`, extend it
  rather than duplicating.

### API client (`services/api.ts`) extensions
Add one typed method per endpoint from `API-PLAN.md`. All new endpoints use the
same `fetch` helpers. New TS interfaces in `src/types/ansible.ts` covering:
`AnsibleRole`, `RoleFile`, `RoleFileRevision`, `PlaybookRevision`,
`PlaybookGlobalRoleRef`, `PlaybookLocalRole`, `PlaybookLocalRoleFile`,
`PlaybookLocalRoleFileRevision`, `PlaybookRunTarget`, `PlaybookRun`.

### Save UX
- Editor is dirty when content differs from the loaded version.
- "Save" button disabled when not dirty.
- Cmd/Ctrl-S triggers save.
- On save: optimistic editor state update → network call → on success invalidate
  and re-fetch revision list. On failure: toast error, keep dirty flag.
- Browser-leave prompt when dirty.

### Phases
1. `SettingsPage` + `FeatureFlagToggle` (operator can flip the `playbooks` flag from the UI before anything else lights up).
2. Ansible nav section + empty sub-pages scaffold (routes wired, nothing in them).
3. Remove `VmPlaybooks` from `VmDetailPage`; redirect old `/playbooks` routes to `/ansible/playbooks`.
4. `AnsiblePlaybooksPage` + `AnsiblePlaybookDetailPage` (read-only, no editor).
5. `AnsibleRolesPage` + `AnsibleRoleDetailPage` (read-only file tree, no editor).
6. Wire up Monaco behind dynamic import; `RoleFileEditor` save flow + `RevisionHistoryDrawer`.
7. Playbook YAML editor + `AttachedRolesPanel` + revision drawer.
8. `AttachedVmsPanel` on `AnsiblePlaybookDetailPage` and `AnsiblePlaybookEditorPage`.
9. Playbook-local roles editor (reuses phases 6–7 components).
10. `AnsibleRunPage` — trigger form with `VmMultiSelect`.
11. `AnsibleHistoryPage` + `AnsibleRunDetailPage` with filter tabs.
12. Live polling on active runs in `AnsibleRunDetailPage` + `RunCancelButton`.
