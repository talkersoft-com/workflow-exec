# Task 0006 — Migration, deploy, results, ship

## Steps
1. Apply the EF migration (`vdef`/`sref`/`varev` + retrofit + re-key) to the live DB BEFORE
   deploying the API:
   ```bash
   cd /home/todd/workspace/cloud-manager/vm-infra/cloud-manager/cloud-manager-api/src/Models/CloudManager.Entities && \
     dotnet ef database update \
       --connection "Host=localhost;User ID=cloudmanager;Password=P@ssw0rd!;Port=5432;Database=clouddb;SslMode=Prefer;Trust Server Certificate=true"
   ```
2. Deploy: `python3 .cicd/deploy-cloud-manager.py --target api`, then `--target mcp`
   (mcp deploy builds dist only — it does NOT restart the running MCP server). Web is NOT
   deployed — this workflow must not change it (deck.md). A failed deploy is a stop
   condition — write a FIX file under `Retro/`, resolve, retry.
3. Smoke: `cloud_health_check`; create one plain `vdef` + one secret `vdef` with a
   `cloudmanager/…` secretRef through the deployed API (then delete them); one cleartext
   write to the secret-classified name returns 409; one `requiredVars` call returns a
   correct missing list against a known-gap fixture.
4. Write `Results/RESULT.md`: outcome, branch, PR table (TBD until ship), phase summary,
   deploy targets + smoke-test attempt counts + HTTP codes, and — because mcp dist was
   rebuilt — this exact operator instruction near the top:

   > **Operator action required:** Restart cloud-manager-mcp via `/mcp` in Claude Code to pick up the new dist.

5. Write `Retro/LESSONS.md` (what would have helped; FIX file list — or one line if clean).
6. Ship (see `../deck.md`):
   ```
   hv_ship  deck: "cloud-manager"
            message: "feat: variable registry + secret refs + required enforcement (P4 — vdef/sref/varev)"
            title:   "Ansible Epic P4: typed variables, secrets as refs, required-var enforcement"
            stage:   "exec"
   ```
7. Fill the PR table in `Results/RESULT.md` from the hv_ship output (amend on the same
   branch; push the amendment via hv_ship again, which will push-only). Per deck behaviour
   the fill may land as a follow-up PR — verify actual PR state with `gh` if the ship
   output omits PR lines.

Done when: migration applied, api + mcp deployed with passing smoke tests, RESULT.md
(with operator /mcp instruction) and LESSONS.md written, hv_ship opened PRs for every repo
ahead of main.
