# Task 0001 — Entities + migration (cloud-manager-api repo)

## Steps
1. RESEARCH (read, no changes): how P1 stores `gvar`/`hvar` records and values, how P2
   stores `vblk` blocks and `defines`/`consumed_by` edges (incl. the `var:` pseudo-id
   form on `EntityEdge`), and how existing prefixes/public-ids are registered. Note the
   forward-looking C-VAR comment in `EntityEdge.cs`.
2. Entities in schema `ansible` (contracts §1):
   - `VariableDefinition` (`vdef`): Name; Scope `global|playbook|role|inventory`;
     OwnerPublicId nullable; Kind `required|default|internal`; Sensitivity `plain|secret`;
     DefaultValue jsonb nullable; Description. Unique index on (name, scope, owner).
   - `SecretRef` (`sref`): FK → VariableDefinition (Cascade); VaultPath — the API-form
     `cloudmanager/data/…` string porch already understands.
   - `VariableEvent` (`varev`): append-only; event types per contracts §6 (`created`,
     `updated`, `deleted`, `classified_secret`, `secret_ref_set`, `secret_ref_removed`).
3. DB-level invariant: CHECK constraint `sensitivity = 'secret' ⇒ default_value IS NULL`
   (mirrored later by service validation in Task 0002). Secret values exist only in Vault.
4. P1 retrofit: nullable `variable_definition_id` FK column on the P1 var tables
   (`gvar`/`hvar`) so inventory values can link to typed definitions without breaking
   existing rows.
5. P2 re-key migration: convert `var:` pseudo-id edges to `vdef_…` ids where the name
   matches a registered definition; unmatched names stay pseudo (still resolvable by name).
   The migration is idempotent and safe on a DB with zero `vdef` rows.
6. Register the `vdef`/`sref`/`varev` prefixes with the existing public-id machinery;
   internal Guids never leak (project convention).
7. EF migration added; `dotnet ef migrations script` output reviewed for the CHECK and the
   unique index.
8. Build: `python3 .cicd/build-cloud-manager.py --target api` must pass.

Done when: entities + migration compile, CHECK and unique index present in the migration,
P1 retrofit column and P2 re-key migration in place, prefixes registered, api build clean.
