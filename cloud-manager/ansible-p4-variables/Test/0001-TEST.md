db_provision_fresh or equivalent) — `vdef`/`sref`/`varev` tables exist in schema `ansible`.
TC-002 CHECK enforced: direct INSERT of a `vdef` row with `sensitivity='secret'` AND a
non-null `default_value` is rejected by the database. TC-003 unique (name, scope, owner):
second INSERT with the same triple fails; same name under a different owner succeeds.

append-only at the service surface — no Update/Delete members exposed. TC-006 P1 retrofit:
`variable_definition_id` exists on the `gvar`/`hvar` tables, nullable, and existing rows
survive the migration with NULL. TC-007 re-key: a fixture `var:` pseudo-id edge whose name
matches a `vdef` is converted to the `vdef_…` id; a non-matching name stays pseudo; running
the migration against zero `vdef` rows is a no-op. TC-008 public-id prefixes `vdef_`/`sref_`/
`varev_` resolve via the existing helpers; no Guid appears in any new DTO. TC-009
`--target api` build passes cleanly.
