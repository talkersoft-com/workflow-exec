# Task 0005 — Live E2E, deploy, results, ship

## Goal
Prove the marketplace on the live system (real VMs on the hypervisor), deploy, write results,
ship.

## Steps
1. **Migrate the live DB FIRST** (deploy fragment order): `dotnet ef database update` against
   `clouddb`, then run the seeds.
2. **Deploy**: `python3 .cicd/deploy-cloud-manager.py --target api` then `--target mcp` (web is
   NOT in this plan). Record smoke-test attempts/HTTP codes for RESULT.md.
3. Enable the `marketplace` feature flag (`cloud_feature_flag_update_enabled key:"marketplace"`).
4. **E2E degenerate case**: `cloud_marketplace_provision` of `jammy` on the live host → VM
   provisions, gets an IP, zero assignments, VmEvents show instantiate + chain_completed (empty
   chain). Destroy the test VM afterward.
5. **E2E full case**: `cloud_marketplace_provision` of `postgres-jammy` → VM provisions; sequencer
   auto-enqueues the pg-14-jammy run at position 0 WITHOUT any manual apply; run reaches
   Succeeded (`cloud_run_get`); VmEvents show chain_step → chain_completed. Keep or destroy the VM
   per operator preference — note the choice.
6. **Regression**: create a plain VM via `cloud_vm_create` (no blueprint), attach/apply a playbook
   directly — identical behavior to pre-change. Verify the marketplace flag OFF hides the new
   endpoints while direct attach still works.
7. Write `Results/RESULT.md` (incl. PR table from hv_ship output placeholders, deploy evidence,
   the operator /mcp reload note for the MCP dist) and `Retro/LESSONS.md`.
8. `hv_ship deck:"cloud-manager" stage:"exec"` with the message/title from `../deck.md`; confirm
   PRs; record them in RESULT.md.

## Definition of done
Both e2e cases pass on live VMs; regression green; deployed; PRs confirmed.
