# Task 0003 — BlueprintRunSequencer

## Goal
VMs instantiated from a blueprint get their playbook chain applied automatically, in blueprint
order, after provisioning completes — with stop-on-failure semantics. API-side only.

## Steps (cloud-manager-api)
1. Locate the two existing writeback sites (PLAN open-question default #2 — hook the consumers
   directly): (a) where orchestration status updates land the VM at OrchestrationStatus=Completed;
   (b) where PlaybookRun status updates land at Succeeded/Failed.
2. `BlueprintRunSequencer` service:
   - `OnVmProvisionCompleted(vmId)`: if VM.BlueprintId == null → no-op. Else compute next step.
   - `OnRunStatusChanged(runId, status)`: resolve the run's VM target(s); for blueprint VMs,
     Succeeded → enqueue next; Failed/Cancelled → stop chain, record VmEvent
     `blueprint_run_chain_failed` (payload: position, assignmentId, runId).
   - Next-step derivation (stateless, per PLAN): BlueprintPlaybooks ordered by Position, matched to
     the VM's assignments by PlaybookId; next = first whose assignment LastRunStatus != Succeeded;
     none left → VmEvent `blueprint_run_chain_completed`.
   - Enqueue via the existing `IPlaybookRunService.EnqueueAsync(assignmentId)` — no new run
     machinery.
3. Hook calls are best-effort try/catch (the VmEvent pattern): a sequencer error must never break
   the writeback path that vorch/porch depend on.
4. Record VmEvent `blueprint_run_chain_step` on each enqueue (payload: position, assignmentId).
5. Unit tests for the next-step derivation (succeeded-prefix, failed-stop, empty-blueprint,
   non-blueprint VM no-op). Build `--target api` green.

## Definition of done
Build + unit tests green; manual smoke deferred to Task 0005 (needs the live hypervisor).
