# Test 0002 — API surface

Pre-req: scratch DB from Test 0001, API running against it, `marketplace` flag ENABLED (flip via
feature-flag endpoint/MCP for the test; record that you re-disable after if policy requires).

## TC-001 — Feature flag gates everything
With the flag disabled: every `/api/v1/blueprint*` and `/api/v1/marketplace*` route returns the
standard flag-gated response (match the `playbooks` flag behavior).
Pass: gated. Fail: any route serves content while disabled.

## TC-002 — Blueprint CRUD + lifecycle
Create → Draft; publish → Published; archive → Archived; re-publish → Published; PATCH name on a
Published blueprint succeeds; DELETE on Published fails with a clear error; archive then DELETE
succeeds (soft: row remains with deleted_at, name reusable). Every transition appended a
BlueprintEvent (verify via SQL: created/published/archived/updated events in order, append-only).

## TC-003 — Ordered playbook management
Attach two playbooks (positions 0,1); attach a third with no position → lands at 2; reorder third
to 0 → positions renumber atomically (0,1,2 with no duplicate-position state visible); duplicate
attach of the same playbook → 409/conflict error; detach middle → remaining renumber or stay gap-
free per implementation, but unique(position) holds.

## TC-004 — Marketplace listing
Only Published blueprints appear; counts and image summary correct for seeded `jammy` (0 playbooks)
and `postgres-jammy` (1).

## TC-005 — Instantiate (MVP semantics)
Instantiate `postgres-jammy` with a host + name + memory override:
- VM created with blueprint defaults except the override; `blueprintId` set; MAC/IP allocated.
- Exactly one VmPlaybookAssignment exists matching the blueprint's playbook with copied vars.
- An orchestration provision message was triggered (orchestration command/event recorded).
- Response carries virtualMachineId + runPlan with the assignment id.
Instantiate `jammy`: VM created, zero assignments, runPlan empty.
Instantiate a Draft blueprint: rejected.

## TC-006 — Direct-attach regression
`cloud_vm_playbook_attach/list/update/detach` against a NON-blueprint VM behave exactly as before
(same shapes, same statuses). Pass: byte-identical behavior.
