# Task 0003 — Run-time resolution snapshot (cloud-manager-api repo)

## Steps
1. Entity `ResolutionSnapshot` (`rsnap`, schema `vm`, contracts §1): FK to `PlaybookRun`;
   per-target resolution report stored as jsonb in the SAME C-RES shape the preview returns,
   secrets masked (the engine masks — the snapshot stores only what the engine emits).
   Public id prefix `rsnap_`. Retention is lifecycle-coupled to the run row (PLAN.md open
   question 3 default) — cascades with run deletion, no independent pruning job.
2. EF migration adding the table. Do NOT modify `PlaybookRun.VarsSnapshot` — it stays as-is;
   `rsnap` is additive provenance alongside it.
3. Capture at enqueue: when a run is enqueued, resolve each target's context through
   `IVariableResolutionService` and persist one report per target. Capture is best-effort
   protective of the run path (mirror the audit-event convention: failure to snapshot must
   not block the enqueue — try/catch + log), but a successful enqueue in normal operation
   always has snapshots.
4. Read endpoint: `GET api/v1/run/{runId}/resolution` → per-target C-RES reports. Gated
   `ansible-studio`. 404 for unknown run; empty collection (not 404) for a run that has no
   snapshots (e.g. pre-feature historical runs).
5. Integration tests: trigger a run (flag on) → snapshots exist per target, jsonb matches
   C-RES, secrets masked in the stored jsonb (SQL-level string search for the fixture
   cleartext finds nothing); read endpoint round-trips; historical run without snapshots →
   200 + empty.
6. Build: `python3 .cicd/build-cloud-manager.py --target api` must pass. Migration is
   APPLIED to the live DB only in Task 0006 — keep it pending here unless local integration
   tests need a local apply.

Done when: `rsnap` entity + migration exist, snapshots captured at enqueue per target,
read endpoint returns them, masking verified at the storage layer, api build clean.
