# Task 0001 — Precedence engine (cloud-manager-api repo)

## Steps
1. RESEARCH (read, no changes): how P1's inventory layers (`gvar`/`hvar`, group nesting),
   P2's `vblk` variable blocks, P4's `vdef` typing (sensitivity, requiredness — C-VAR §7),
   assignment/blueprint overrides (jsonb, `$secretRef`-aware), and porch's current extravars
   assembly are stored and queried today. Note `PlaybookRun.VarsSnapshot` — it stays; this
   plan adds provenance alongside, never replaces it.
2. Define the precedence table as DATA: an ordered list of layer descriptors mapped to
   Ansible's documented ~22-level order. Only levels representable in our model participate;
   the table itself records which levels are out of scope (e.g. ansible.cfg-level sources
   porch doesn't use) so unmodeled levels are explicit, not silent (PLAN.md open question 2
   default: explicit unmodeled markers).
3. Implement `IVariableResolutionService.Resolve(context)` — context = playbookId
   (+revision), inventoryId, hostId, optional extraVars. Pure service, no controller
   dependencies. Fold layers host-up: role defaults → inventory group vars by nesting depth
   → host vars → play vars → assignment override → blueprint override → extra vars.
   Per variable produce: winner (precedence level + layer + source record public id),
   overridden chain (same shape), state (`resolved | undefined | missingRequired` —
   required kinds from C-VAR with no winner are `missingRequired`).
4. Masking is ENGINE-level (contracts §8 invariant): any variable whose `vdef` says
   `sensitivity: "secret"` — or whose winning value is a `$secretRef` — reports
   `effectiveValue: "***"` with the ref's public id in provenance. The engine never returns
   cleartext on any path; controllers/MCP cannot leak what they never receive.
5. Group-nesting flatten is iterative, relying on P1's acyclicity guarantee. Layer inputs
   are record queries batched per context — no N+1 over hosts (epic §11 groundwork for
   Task 0005's perf gate).
6. Unit suite: fold ordering (each adjacent layer pair beats the one below), overridden
   chain completeness, undefined vs missingRequired states, secret masking on every path
   (direct secret `vdef`, `$secretRef` winner, secret overridden by plain), unmodeled-level
   markers, and C-VAR fixtures taken verbatim from contracts §7.
7. Public-id conventions: provenance carries public ids only (`hvar_…`, `vblk_…`, …) —
   internal Guids never leak (project convention).
8. Build: `python3 .cicd/build-cloud-manager.py --target api` must pass.

Done when: engine resolves a real playbook+inventory+host context with correct winner /
overridden / state per variable, secrets masked engine-side, unit suite green, api build clean.
