# Task 0004 — Write path: ack consumer (`teardown_succeeded` / `teardown_failed`)

## Goal
The `OrchestrationCompletionQueue` consumer in `CloudManager.AMQP.Consumer` translates each vorch destroy ack into exactly one `teardown_succeeded` or `teardown_failed` event.

## Files
- `cloud-manager-api/src/Services/CloudManager.AMQP.Consumer/<the destroy-ack consumer>` (the handler that subscribes to `OrchestrationCompletionQueue`).

## Steps
1. Locate the existing destroy-ack handler. Add `IVmEventService` dependency via DI.
2. On message receipt, resolve the VM by public_id from the ack body. Use `.IgnoreQueryFilters()` so soft-deleted VMs still match.
3. Inspect the ack body's success field:
   - success → `RecordEventAsync(vmId, VmEventType.TeardownSucceeded, ackBody, actor: "vorch-service", occurredAt: ackBody.timestamp ?? UtcNow)`
   - failure → `RecordEventAsync(vmId, VmEventType.TeardownFailed, ackBody, actor: "vorch-service", occurredAt: ackBody.timestamp ?? UtcNow)`
4. Exactly one event per ack — guard against double-publishing by inspecting message identifier if the consumer is at-least-once.
5. Build clean.

## Notes
- Do not modify vorch. Use the ack format as-is.
- The payload jsonb stores the entire ack body so the UI can pretty-print and operators can inspect the vorch error message.

## Definition of done
Build clean. A destroy operation that vorch acks successfully produces a `teardown_succeeded` event row. A failing ack produces `teardown_failed` with the vorch error visible in `payload`.
