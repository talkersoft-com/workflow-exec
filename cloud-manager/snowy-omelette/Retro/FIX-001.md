# FIX-001 — Phase 11 E2E test prompt routes mismatch implementation

## What I observed

The Phase 11 E2E verification prompt assumes REST routes that do not exist in the
current API build:

- Prompt says `POST /api/v1/VirtualMachine` — real route is
  `POST /api/v1/virtualmachine/create/{HostPublicId}` and the request body is
  `{ "virtualMachine": {...}, "vmi": "vmi_..." }`, not a flat object with
  `hostId` and `virtualMachineImageId`.
- Prompt says `GET /api/v1/VirtualMachine` (list) — real route is
  `GET /api/v1/virtualmachine/list`.
- Prompt says `DELETE /api/v1/VirtualMachine/{id}` — that returns 405 (only
  PATCH is allowed at that path). The actual destroy endpoint is
  `DELETE /api/v1/orchestration/destroy/{publicId}` on `OrchestrationController`.
- Prompt says the response shape from list uses key `virtualMachines` with
  items keyed by `publicId` — actual item key is `id`.

Tests 1-3 still passed once I used the real routes/bodies, but the prompt
should be updated so future runs don't waste time re-discovering this.

## Secondary observation — `destroy_requested` timing

The prompt step 4 expects `destroy_requested` to appear within ~2s of the
DELETE call. In reality the DELETE only publishes a `destroy-vm` AMQP message;
`destroy_requested` is written inside the consumer's success branch right
before `teardown_succeeded` (both fired by `DeleteVirtualMachineAsync` +
`RecordEventAsync` in `ConsumerService.cs`). So both events show up together
~15s later when vorch acks. Working as designed, but the prompt suggests
`destroy_requested` is the most recent before vorch ack — which is not what
happens.

## Suggested prompt fix

Update Phase 11 task and any E2E recipe to:

1. Replace VM-create call with the create/{HostPublicId} form and the
   `{virtualMachine, vmi}` request body.
2. Replace list call with `/api/v1/virtualmachine/list`.
3. Replace delete call with `DELETE /api/v1/orchestration/destroy/{publicId}`.
4. Drop the "destroy_requested before ack" intermediate step, OR move the
   destroy_requested write to happen synchronously in the destroy endpoint
   (before publishing the AMQP message) so the audit shows intent immediately.

## Was the audit feature itself broken?

No. All three tests passed end-to-end. The events table, the GET events
projection, retry-teardown happy path, and the 400 guard all work correctly.
