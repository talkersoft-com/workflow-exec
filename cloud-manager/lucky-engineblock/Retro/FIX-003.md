# FIX-003 — Sequencer fires before guest SSH is ready (chain failed on fresh VM)

## Failure
Phase 0005 TC-003: instantiating `postgres-jammy` (vm_PZSZ5PCC9M) provisioned fine and the
sequencer auto-enqueued the position-0 run (run_Z8F0XQAV5E) immediately on the create-vm
Success ack — but the run failed with `UNREACHABLE: ssh: connect to host 10.0.150.63 port 22:
No route to host`. vorch acks "Success" when the hypervisor domain is up, before cloud-init has
brought up sshd. A human operator always waited before applying manually; the sequencer did not.
Port 22 on the same VM answered a couple of minutes later — pure readiness race.
(Stop-on-failure itself worked: `blueprint_run_chain_failed` was recorded and position 1 was
never enqueued — that part of the test passed.)

## Fix
`BlueprintRunSequencer.OnVmProvisionCompletedAsync` no longer advances inline. It schedules a
background task (fresh DI scope) that polls TCP :22 on the VM's recorded IP (re-read each
attempt, since vorch writes the IP back asynchronously) every 10s for up to 5 minutes, then
starts the chain. On timeout it records `blueprint_run_chain_failed` with reason
`ssh_unreachable_timeout`. No vorch/porch change. Known limitation (documented in LESSONS): an
API restart during the wait window drops the pending chain start; the operator can apply
manually.

## Status
Resolved — re-tested with a fresh instantiate after redeploy (see RESULT.md Phase 5).
