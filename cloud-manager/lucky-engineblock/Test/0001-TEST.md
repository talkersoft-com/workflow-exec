# Test 0001 — Data model

## TC-001 — Build passes
`python3 .cicd/build-cloud-manager.py --target api`
Pass: exit 0. Fail: any compile error.

## TC-002 — Migration round-trips on a scratch database
Provision/point at a scratch DB (db-toolkit `clouddb_local` or equivalent — NEVER the live
`clouddb` for this test). `dotnet ef database update` → verify tables
`marketplace.blueprints`, `marketplace.blueprint_playbooks`, `marketplace.blueprint_events` and
column `vm.virtual_machines.blueprint_id` exist (db_get_table_schema / db_list_tables). Then
`dotnet ef database update <previous-migration>` → verify they are gone. Re-apply forward.
Pass: clean both directions. Fail: any error or leftover artifacts.

## TC-003 — Constraints enforced
On the scratch DB via SQL: inserting two BlueprintPlaybooks with the same (blueprint, playbook) or
the same (blueprint, position) fails; two non-deleted Blueprints with the same name fails; a
soft-deleted Blueprint's name can be reused.
Pass: all three rejections + the reuse succeed as described.

## TC-004 — Seeds idempotent and correct
Run the seed twice against the scratch DB.
Pass: exactly one `jammy` (0 playbooks, Published) and one `postgres-jammy` (1 playbook at
position 0, Published); one `marketplace` feature flag row (disabled); second run changes nothing.

## TC-005 — Prefixes registered
Insert a Blueprint/BlueprintPlaybook/BlueprintEvent via EF on the scratch DB.
Pass: PublicIds generated with `bp_`, `bpp_`, `bpev_` prefixes.
