# Task 0003 — Run sequencer

## Goal
A new `BlueprintRunSequencer` service auto-runs a blueprint VM's playbooks in blueprint order once
provisioning completes, advancing on success and stopping on failure — composed entirely from
existing API-side machinery. No vorch/porch change, no schema change, no new queue infrastructure.

## Repo
`cloud-manager-api`

## Design (from PLAN.md — read it first)
- Trigger source (plan default): hook the AMQP status-writeback consumer directly — the place where
  VM OrchestrationStatus and playbook-run status transitions already materialize. Fall back to a
  lightweight poller ONLY if the consumer path proves awkward; if so, write a FIX file explaining
  why.
- When a VM with `BlueprintId != null` reaches `OrchestrationStatus=Completed` → enqueue the run
  for the assignment matching the blueprint's Position 0 (existing PlaybookRun enqueue path).
- When a sequenced run reaches `Succeeded` → enqueue the next position.
- Ordering state is DERIVED, not stored: VM.BlueprintId → BlueprintPlaybooks ordered by Position,
  matched to the VM's assignments by PlaybookId; next = first whose LastRunStatus != Succeeded.
- Failure stops the chain and logs a VmEvent `blueprint_run_chain_failed`; the user can fix and
  `apply` manually from there.
- VMs with `BlueprintId == null` are never touched by the sequencer.
- Blueprint with zero playbooks (jammy): sequencer does nothing after provision — terminal state
  immediately.

## Steps
1. Locate the orchestration status-writeback consumer and the playbook-run status update site.
2. Implement `BlueprintRunSequencer` (service + DI registration) invoked from those two
   transitions; guard non-blueprint VMs and re-entrancy (a status update arriving twice must not
   double-enqueue).
3. Add `blueprint_run_chain_failed` (and any needed sequencer event types) to `VmEventType`;
   record events best-effort.
4. Build: `python3 .cicd/build-cloud-manager.py --target api` — must pass cleanly.

## Definition of done
Sequencer wired to both transitions, derived-order logic correct, stop-on-failure + event logged;
api build green. Run `Test/0003-TEST.md`.
