# FIX-001 — Playwright verification hit the wrong origin

## Failure
First verification run navigated to `http://localhost:3000/blueprints` and the page rendered
"Marketplace disabled" with console errors (`SyntaxError: Unexpected token '<'` on API/SignalR
calls), despite `cloud_feature_flag_list` reporting `marketplace` enabled and healthy.

## Diagnosis
- The web bundle calls the API with a relative base (`VITE_API_BASE_URL=` is empty in the repo
  `.env`), so all `/api/v1/*` requests go to the page origin.
- `localhost:3000` is the raw `serve -s` static backend (systemd `cloud-manager-web`). It has no
  API proxy; its SPA fallback returns `index.html` (HTTP 200, HTML) for `/api/v1/*`, which the
  app fails to parse → feature flags load fails → FeatureGate falls back to "disabled".
- The real deployed origin is `https://ubuntu-server.talkersoft.com`: it serves the same bundle
  AND proxies `/api/v1/*` to the API on `localhost:5250` (verified:
  `curl https://ubuntu-server.talkersoft.com/api/v1/featureflags` returns the flag JSON;
  `curl http://localhost:3000/api/v1/featureflags` returns HTML).

## Fix
No code or deploy change — the deploy (rsync to `/opt/cloud-manager-web`, HTTP 200 smoke) was
correct. Re-run the Playwright verification against `https://ubuntu-server.talkersoft.com`.

## Status
Resolved — re-run performed against the correct origin (see RESULT.md).
