# Task 0002 — Editor decorations: badges, gutter markers, theme tokens (cloud-manager-web repo)

## Steps
1. RESEARCH (read, no changes): how the P7 editor renders the `decorations` prop (C-ED
   compartment); the SCSS design-token system (`--color-*` custom properties under
   `[data-theme]`) and how the P7 theme consumes it.
2. Wire `useVarDecorations` output into the editor pages' `decorations` prop
   (`AnsiblePlaybookEditorPage`, `AnsibleRoleDetailPage` — the C-ED consumers). The editor
   component itself is NOT edited; data flows through the existing prop only.
3. Visual treatment per kind:
   - `secret`: lock badge on the identifier + a distinct secret color token (new `--color-*`
     tokens as needed, defined for BOTH `[data-theme]` values — no hardcoded hex).
   - `required`: required badge; when resolution data (Task 0003+) reports `missingRequired`
     for the current context, a bold gutter dot on the line. Until a context exists, the badge
     renders without the missing amplification (plan open question 2 default: missingness is
     contextual).
   - `vaultRef`: vault icon on `$secretRef` values and `vault_*` lookups.
   - Unregistered: ordinary template coloring, plus the quiet tooltip hint from Task 0001's
     surfaced names.
4. Component tests: each kind renders its badge/marker; theme flip (`[data-theme]`) resolves
   all decoration colors from tokens; a document with an empty registry renders with zero
   decorations and no errors (graceful degradation).
5. Build: `python3 .cicd/build-cloud-manager.py --target web` must pass.

Done when: secret/required/vaultRef decorations render in both editor pages via the C-ED
`decorations` prop, correct under both themes, with graceful empty-registry behavior; web
build clean. No P7 editor-component file is modified.
