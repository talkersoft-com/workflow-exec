# Task 0008 — E2E verify

## Goal
A fresh `cloud_playbook_run` succeeds end-to-end with no manual SQL or Vault CLI ops, and the postgres password lands at `cloudmanager/data/playbook-secrets/<runId>/<host>/postgres`.

## Steps

1. **State check.** `cloud_vm_list` — confirm `vm_GY68JPPKDN` (or another running Jammy VM) exists. If not, `cloud_vm_create` one with the jammy VMI (`vmi_2QMK0FHWXG`). Wait until SSH responds (poll `nc -zv -w 3 <ip> 22`).
2. **Trigger.** `cloud_playbook_run playbookId="pb_PG14JAMMY01" vmIds=["<vm_public_id>"]` — capture the returned `runId`.
3. **Poll.** `cloud_run_get` every few seconds until status is `succeeded` or `failed`. If failed, dump `cloud_run_target_get` for the per-host stats and stop.
4. **Vault check.** From this dev box:
   ```bash
   export VAULT_ADDR=https://ubuntu-server.talkersoft.com:8200
   export VAULT_TOKEN=<orchestrator token from /kvm-automator/config-server.yaml>
   vault kv get cloudmanager/playbook-secrets/<runId>/<host>/postgres
   ```
   The output should show `host`, `port: 5432`, `db: app`, `user: appuser`, `password: <generated>`. The orchestrator policy update from Phase 5 must grant this read.
5. **No-manual-ops check.** Confirm:
   - No `psql` UPDATE on `vm.playbooks.content` happened during this phase
   - No `vault policy write` or `vault write auth/token/roles/...` happened during this phase
   - The playbook content's git-equivalent (the seed file from Phase 6) is unchanged from what's running

## Definition of done
Status=succeeded; Vault read returns the secret; no manual ops were performed. The cloud-manager-web Run detail page (browser, http://localhost:3000) surfaces the Vault path in the run output.
