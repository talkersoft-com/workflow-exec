# Task 0002 — VAULT_ADDR FQDN

## Goal
Porch's `env/envvars` JSON emits the configured FQDN VAULT_ADDR (`https://ubuntu-server.talkersoft.com:8200`) instead of `https://127.0.0.1:8200`. The TLS cert validates against the FQDN, so `validate_certs: false` becomes unnecessary.

## Files
- `vorch-service/vorch-service/internal/ansible/playbook_run_exec.go` — the `vaultAddrOrEmpty(h.Vault)` call and the `envvars` map construction.
- `vorch-service/vorch-service/internal/vault/client.go` — the `Client.Address()` method (already returns `cl.c.Address()` which should be the configured URL).

## Investigation

1. Audit `vaultAddrOrEmpty` — does it return `h.Vault.Address()`, or does it hardcode/default to `127.0.0.1`? The current `envvars` output (`/kvm-automator/playbook-runs/<recent-run>/env/envvars`) emits `127.0.0.1:8200` even though `/kvm-automator/config-server.yaml` has `vault.address: https://ubuntu-server.talkersoft.com:8200`. Something is short-circuiting the configured value.
2. If `vaultAddrOrEmpty` is correct, the issue is upstream: how `h.Vault` was initialized. Check `vorch-service/cmd/porch/main.go` (or equivalent) where the vault Client is constructed — the `vaultapi.Config.Address` it receives might be wrong.

## Edits

```yaml
# Before — env/envvars for a fresh run
{"VAULT_ADDR": "https://127.0.0.1:8200", ...}

# After
{"VAULT_ADDR": "https://ubuntu-server.talkersoft.com:8200", ...}
```

## Build + deploy

Same as Task 0001 — rebuild porch, restart systemd. Then trigger a sample playbook run and inspect `env/envvars` to confirm the FQDN.

## Definition of done
`env/envvars` from a fresh run shows the FQDN VAULT_ADDR. `curl -sk https://<that-url>/v1/sys/health` from this box and from inside a fresh VM both succeed with TLS validation (no `-k` needed after this — the cert is for that FQDN).
