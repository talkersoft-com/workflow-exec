# Task 0009 — Teardown + rebootstrap verify

## Goal
A full `bootstrap-server.py` cycle from a wiped workspace ends with a green playbook run, with the bootstrap-worker step doing the Vault policy + playbook import automatically.

## Steps

This phase is the "would this work on a fresh box?" verification. Treat the dev-box as a customer install.

1. **Snapshot the working state.** Capture a list of: current playbooks (via `cloud_playbook_list`), current Vault policies (`vault policy list`), current Vault token roles. Stash for comparison after.
2. **Run the driver from the operator's other machine OR run the worker directly.**
   - `python3 bootstrap-worker.py` from this box (no SSH, just exec locally per the local-only design)
   - It will:
     - Back up the current workspace
     - Clone fresh workflow-configuration
     - `hv init hive-deck-pro`
     - `hv init cloud-manager`
     - Rebuild hv binary
     - Build MCP dists
     - **NEW** — call `bootstrap-policies.py` (Phase 5 script)
     - **NEW** — call `import-playbooks.py` (Phase 6 script)
     - Verify
3. **Trigger a fresh playbook run** against a newly-provisioned VM (because the old VM might still be in libvirt but doesn't exist in the new DB). Expect status=succeeded.
4. **Compare snapshots.** Vault policies + roles match what was there before (modulo policy text being normalized). Playbook content matches what was exported in Phase 6.

## Definition of done
Bootstrap completes without operator intervention (no manual `vault policy write`, no manual `psql` UPDATE, no manual `cloud_playbook_create`). A subsequent `cloud_playbook_run` against a fresh VM succeeds end-to-end.

## Note
This phase is destructive — it backs up and replaces `/home/todd/workspace`. If aborted partway, the previous workspace is recoverable from `~/workspace-backups/workspace-<timestamp>`. Document in `Retro/LESSONS.md` if anything in the bootstrap chain didn't behave as expected.
