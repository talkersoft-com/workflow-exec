# Fix 002 — `npm run typecheck` OOMs (pre-existing)

## Failed test
Task 0001, TC-001 — `npm run typecheck` (`tsc --noEmit`) crashes with "JavaScript heap out of memory" at the default ~4GB node heap.

## Root cause
Pre-existing, not caused by this change: stashing the resolver wiring and re-running typecheck on the clean `gloppy-snapper` tree OOMs identically. The database-toolkit type program is too large for the default heap (node v22.22.2, tsc via `npm run typecheck`).

## Fix applied
Retry typecheck with `NODE_OPTIONS=--max-old-space-size=8192`. If it passes, TC-001 is satisfied (with this flag noted). If it still OOMs, typecheck is recorded as broken-before-and-after this workflow, TC-001 falls back to `npm run build` + the dist smoke tests, and the repo owner should fix typecheck in a separate change.

## Outcome
Partial pass, pre-existing breakage documented. With `NODE_OPTIONS=--max-old-space-size=8192` tsc completes instead of OOMing, but reports numerous pre-existing type errors (database-lib/database-manager.ts, factory.ts, cdk-command-logs.ts, cdk-component-operations.ts, database-infrastructure-operations.ts, several TS2589 "excessively deep" in tool entry points). Zero errors in the files this workflow touches (`database-config-loader.ts`, `vault-resolver.ts`). TC-001's typecheck clause is therefore evaluated as "no new errors introduced"; repo-wide typecheck repair is out of scope and should be its own plan. TC-001's build clause still applies in full.
