# Fix 003 — TC-006 failure was a wrong-loader integration, not a stale MCP server

## Failed test
Task 0001, TC-006 — `db_test_connection pg-test` kept returning "Database connection requires
host, port, username, and rootPassword" after the rebuild. Initial diagnosis blamed a stale MCP
server process awaiting operator `/mcp` reload. Disproved: a **freshly spawned** server from the
rebuilt dist (driven over stdio JSON-RPC) failed identically.

## Root cause (three stacked issues)
1. **Wrong loader.** The toolkit has two config-loading stacks. Task 0001 wired
   `resolveVaultEntries` into `tools/shared/database-config-loader.ts`, but the MCP tools
   (`database-connection-tester`, `database-lister`, `execute-sql-query`, …) import
   `tools/shared/config-provider.ts` (`AutoConfigManager`), which reads
   `~/.hv/toolkit/db/config/database.json` — a different loader AND a different config file than
   the `.eng-data-tools` one the first wiring targeted.
2. **Config shape mismatch.** The live `~/.hv` pg-test entry used `credentialProvider: "vault"` +
   `vmId` — the contract of a *third*, already-built mechanism
   (`database-lib/connection-providers/`, `VaultCredentialProvider` + `SshTunnelProvider`) that the
   factory never triggers because `factory.ts:54` checks only the legacy `connectionProvider`
   field, not the new split fields. The resolver wired by this task expects `vaultPath`.
3. **Environment.** The pg-test SSH tunnel had been bound to local 5432, colliding with clouddb's
   configured address (masked as a clouddb auth failure); and the connection tester deliberately
   connects as an admin user (`usernameOverride || 'postgres'`), which the entry didn't set.

## Fix applied
- `tools/shared/config-provider.ts`: import `resolveVaultEntries`, apply it once in
  `loadDatabaseConfig()` (covers both test-mode and MCP-mode paths) — per PLAN.md's "apply to all
  config load paths used by the MCP tools".
- `~/.hv/toolkit/db/config/database.json` pg-test entry: added `vaultPath` (resolver contract),
  `usernameOverride: "appuser"` (tester admin convention, same as clouddb/appdb),
  `host: "localhost"`, `port: 15432` (entry-wins tunnel override on a non-colliding port).
- Tunnels: pg-test moved to `localhost:15432 → 10.0.150.61:5432`; `localhost:5432 →
  ubuntu-server localhost:5432` restored for clouddb.
- TC-003 amended: changed files under `tools/` are `database-config-loader.ts` AND
  `config-provider.ts`; `vault-resolver.ts` remains untouched.

## Verification
Fresh-server stdio smoke: pg-test `db_test_connection`, `db_list_databases`,
`db_execute_sql_query` (SELECT version()) all PASS; clouddb connection and
`db_get_config_identifiers` regressions PASS.

## Follow-up (out of scope, recorded)
`database-lib/database/factory.ts:54` should also trigger provider resolution on the new
`credentialProvider`/`tunnelProvider` fields — that path auto-resolves credentials AND tunnels per
connection and would make the load-time resolver + manual tunnel setup unnecessary for entries
like pg-test. Candidate for a small follow-up plan.
