# Task 0003 — Investigate double-encoded per-VM SSH secret

## Goal
Map the full producer/consumer graph of Vault secret `cloudmanager/vm/instances/<vm-id>/ssh` and write a recommendation on its shape. No code change in this task.

## Background
The secret has a single field `ssh_key` whose value is itself a JSON string (`{"private_key":"...","created_at":"..."}` with escaped newlines), forcing consumers to double-parse. Changing the shape without mapping all readers risks breaking working consumers.

## Target repo(s)
`workflow-plans` (findings doc only). Code is read-only everywhere in this task.

## Steps
1. Find the writer: grep `vorch-lib` (and `vorch-service`) for the secret path pattern (`vm/instances`, `/ssh`, `ssh_key`) and the marshalling that produces the nested JSON. **The deck convention forbids vorch changes** — if the writer is vorch-lib, the recommendation must flag that explicitly.
2. Find every reader:
   - `cloud-manager-cli` `internal/vaultclient` (used by `cm ssh`)
   - porch (search `vorch-service` and the rest of the workspace — porch was carved out of vorch-service)
   - scripts under `cloud-manager-api/scripts/`
   - any other hit on the path pattern across the workspace
3. Do NOT fetch the secret's contents from Vault — the shape is already documented in PLAN.md; reason from code. Never reproduce private-key material in the findings doc.
4. Write the findings doc: `../../../workflow-plans/cloud-manager/beaming-messenger/FINDINGS-0003-ssh-secret-shape.md` containing:
   - Writer: repo + file:line + the exact marshalling step that double-encodes
   - Readers: each with repo + file:line + how it parses (single vs double decode)
   - Option (a): normalize the shape — one coordinated change covering writer, a migration for existing secrets, and every reader; list the blast radius (repos touched, vorch flag, migration risk)
   - Option (b): document the double-encoded shape as the contract; list the blast radius (ongoing consumer complexity)
   - An explicit recommendation between (a) and (b); if (a), state that the work belongs in a separate plan (default assumption per PLAN.md, given the vorch no-change convention)

## Definition of done
Findings doc exists with writer + complete reader map (file:line evidence), both options with blast radius, and an explicit recommendation. Zero code changes anywhere; vorch-lib/vorch-service untouched.
