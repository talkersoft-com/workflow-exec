# Task 0002 — Investigate postgres VM provisioning gap

## Goal
Establish, with evidence, how PostgreSQL got onto the pg-test VM (`vm_SMAZ3C2YVZ`, 10.0.150.61) — playbook-owned or hand-installed — and produce a findings doc recommending (and only if playbook-owned, implementing) network-access configuration for newly provisioned postgres VMs.

## Background
The VM came up with PostgreSQL 14 bound to 127.0.0.1 and a localhost-only pg_hba, unreachable even from the KVM host. Fixed manually on 2026-06-10 (`listen_addresses='*'`, scram entries for 10.0.150.0/24 and 10.0.144.140/32). Observed once; cause unknown. Do NOT codify anything until ownership is established.

## Target repo(s)
`workflow-plans` (findings doc). Conditionally the repo owning the postgres playbook/role — only if ownership is confirmed; update `../deck.md` if so.

## Steps
1. Via cloud-manager-mcp, inspect what is attached to and has run against the VM:
   - `cloud_vm_playbook_list` for `vm_SMAZ3C2YVZ`
   - `cloud_playbook_run_list` filtered to that VM (run history around the VM's creation)
   - `cloud_playbook_list` / `cloud_playbook_role_list` / `cloud_ansible_role_list`; for anything postgres-flavored, pull role files with `cloud_playbook_role_file_list` / `cloud_playbook_role_file_get` and look for postgres install/config tasks
2. Grep the workspace repos (read-only) for postgres provisioning automation (cloud-init, image build steps under `os-images/`, scripts) that might have installed postgres outside the playbook system.
3. Write the findings doc: `../../../workflow-plans/cloud-manager/beaming-messenger/FINDINGS-0002-postgres-provisioning.md` containing:
   - What installed postgres on the VM, with evidence (playbook/role names + file references, run IDs, or the absence trail that proves hand-install)
   - If a playbook/role owns postgres setup: a scoped recommendation for `listen_addresses` + pg_hba (scram-sha-256 only) limited to the VM network and the KVM host source IP — and answer the open question: subnet-only (10.0.150.0/24) vs subnet + KVM host (10.0.144.140/32); tunnel traffic observed arriving from the latter
   - If hand-installed: document that, and recommend whether a postgres role should exist (recommendation only — do not invent a new playbook in this workflow)
4. **Only if** a playbook/role owns postgres setup: implement the recommended change, idempotent, affecting newly provisioned VMs only — never retroactively reconfiguring existing VMs. Update `../deck.md` repo table.

## Definition of done
Findings doc exists and answers the ownership question with concrete evidence; pg_hba scope proposal included; code change present only if ownership was confirmed, and then idempotent + scoped to new VMs.
