# Task 0004 — Results, Retro, Ship

## Goal
Record the run's outcome, then ship the deck so main has the resolver wiring and the findings docs.

## Target repo(s)
`workflow-exec` (this folder), then the deck via `hv_ship`.

## Steps
1. Write `Results/RESULT.md` — outcome (SHIPPED/PARTIAL/FAILED), branch `gloppy-snapper`, phase summary, PR table (URLs filled from hv_ship output afterward is acceptable for the URLs only — the file itself must exist BEFORE the ship call or it lands on the next branch).
2. Write `Retro/LESSONS.md` — what would have helped, one line if nothing went wrong; list any FIX files written.
3. Verify all boxes in `Orchestrate/ORCH.md` for Tasks 0000–0003 are checked.
4. Run `hv_status deck: "cloud-manager"` — expected dirty repos are exactly those in `../deck.md`'s table (database-toolkit, workflow-plans, workflow-exec, plus the conditional 0002 repo if used).
5. Ship:
   ```
   hv_ship  deck: "cloud-manager"
            message: "database-toolkit: wire vault-resolver into config loaders; pg-test provisioning + ssh-secret findings"
            title:   "Vault-backed db connections: wire the resolver; provisioning + ssh-secret findings"
   ```
6. `hv_list_pulls deck: "cloud-manager"` — record PR URLs and merge status in `Results/RESULT.md`.
7. After all PRs are merged: `hv_next deck: "cloud-manager"` to transition the deck to a fresh branch.
8. Remind the operator: the database-toolkit MCP must be reloaded via `/mcp` wherever it runs, since dist changed.

## Definition of done
RESULT.md and LESSONS.md committed in the shipped PR; PRs open (or merged) for exactly the repos in deck.md's table; PR URLs recorded; deck transitioned with `hv_next` after merge.
