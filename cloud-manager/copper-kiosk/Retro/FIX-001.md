# FIX-001 — Chain panel stuck on "Pending" while the run was Queued (Task 0004 / TC-001)

## Symptom
Instantiated `postgres-jammy` (vm_V3KMJ0BX30) and stayed on the VM detail page. The backend
sequencer enqueued `run_V9AV4BMRNX` (status Queued at 16:23:23Z), but the run-chain panel kept
showing the step as **Pending** — TC-001 requires the queued/running phase to be visible.

## Root cause
The panel derived step state from `PlaybookAssignment.lastRunStatus`
(`GET /virtualmachine/{id}/playbook`). The API only writes `lastRunStatus`/`lastRunId` on the
run-target's terminal/progress writeback (`PlaybookRunTargetService.cs:115`), NOT when the
sequencer enqueues the run. So the assignment stays `None` through the entire Queued (and
potentially Running) phase, and the panel would have jumped Pending → Succeeded.

## Fix (web only, existing endpoints)
Derive live step state from `GET /api/v1/run?vmId={publicId}` (`api.runs.list`) — the same
endpoint AnsibleHistoryPage uses. The newest run per `playbookId` carries `status`
(Queued/Running/Succeeded/Failed/Cancelled) and the run id for the failure link. The
assignment list is still used to resolve the instantiate-time `runPlan` (assignment ids →
playbook ids); `lastRunStatus` remains only as a fallback when no run row exists yet.

## Re-test
Re-run TC-001 against the same VM (run still in flight) — panel must show Queued/Running then
Succeeded with no manual reload.
