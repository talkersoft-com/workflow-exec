# Task 0004 — VM provenance + live run-chain panel

## Goal
On the VM detail page, a marketplace-born VM shows where it came from and how its chain is going —
live.

## Steps
1. Provenance: when `vm.blueprintId` is set, show "Created from blueprint <name>" linking to
   `/blueprints/{id}`. Absent for null (admin-created VMs unchanged).
2. Run-chain panel: the blueprint's ordered playbooks, each with derived state
   pending → queued → running → succeeded/failed, composed ONLY from existing endpoints
   (blueprint detail for the order; run/target endpoints + assignment lastRunStatus for state).
   Live updates over the same SignalR mechanism RunDetailPage uses — no polling loops.
3. Failure UX: a failed step marks where the chain stopped and links to `/ansible/runs/{runId}`.
4. If the panel CANNOT be composed from existing endpoints, that is a Phase 0005 finding — do not
   add or modify API endpoints.
5. Build green.

## Definition of done
Instantiating postgres-jammy and sitting on its VM detail page shows the chain advance to
Succeeded without a manual refresh.
