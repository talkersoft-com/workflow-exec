# FIX-001 — Postgres playbook apply was a zero-target no-op

## Symptom
During Task 0003 E2E, after provisioning the Postgres VM `fw-e2e-pg` (vm_1M83BPB0SP,
10.0.150.95) I attached `pb_PG14JAMMY01` (assignment `asgn_315ZF0ES39`) and triggered it with
`cloud_vm_playbook_apply(vmId, assignmentId)` → `run_4YP0CWY6JR`.

The run reported **`final status=succeeded`**, but a port probe showed `5432` still blocked
from the (trusted) hypervisor. porch.log revealed the truth:

```
run_4YP0CWY6JR has zero targets; marking succeeded (no-op)
run_4YP0CWY6JR final status=succeeded
```

The playbook never executed on the VM — a false success. The cloud-init ufw baseline *was*
present (22 open, all else blocked), but the per-service `5432` rule from the Postgres
playbook was never applied because the playbook body never ran.

## Root cause
`cloud_vm_playbook_apply` resolved the assignment to **zero run-targets** for this VM, so
porch short-circuited to a no-op success. The assignment-derived targeting did not enumerate
the VM as a host (likely because the VM run-target was not registered/healthy at apply time).

## Fix
Re-run the Postgres playbook with **explicit** VM targeting via
`cloud_playbook_run(playbookId=pb_PG14JAMMY01, vmIds=[vm_1M83BPB0SP])`, which builds the
run-target list directly from `vmIds` instead of the assignment, then poll the run to
`Succeeded` and confirm `5432` is reachable from the on-subnet/trusted hypervisor.

## Verification after fix
- `cloud_run_get` / `cloud_run_target_list` show one target (the pg VM) and status Succeeded.
- `nc -z 10.0.150.95 5432` from the hypervisor (source 10.0.144.140, a trusted /32) → OPEN.
- `nc -z 10.0.150.95 5672/15672/80` → blocked (only 22 + 5432 open).
