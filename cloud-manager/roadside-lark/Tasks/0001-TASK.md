# Task 0001 — `cm-seed-secrets` seeder script

## Goal
Author the authoritative, idempotent seeder in `vm-infra/secret-management`
(`scripts/cm-seed-secrets`). It is the ONLY thing that sets the controller's
Postgres/RabbitMQ passwords and renders the systemd credential files.

## Behaviour (per secret)
For DBPASSWD and RABBITMQ_PASSWORD:
1. **value** — read the current password from Vault (`cloudmanager/controller/postgres`,
   `cloudmanager/controller/rabbitmq`). If present, reuse it. Else generate a 32-char random
   (`ascii_letters + digits`).
2. **apply** — Postgres: `ALTER ROLE cloudmanager WITH PASSWORD '...'`. RabbitMQ:
   `rabbitmqctl change_password cloudmanager '...'`.
3. **store** — write the value back to its Vault path (KV v2).
4. **inject** — `systemd-creds encrypt --name=<NAME> - /etc/cloud-manager/creds/<name>.cred`.

For VAULT_TOKEN: do NOT generate. Read the live token (from `/etc/cloud-manager/api.env`
or Vault), and `systemd-creds encrypt --name=VAULT_TOKEN` it to
`/etc/cloud-manager/creds/vault-token.cred`.

## Requirements
- Idempotent: re-running reuses stored values; output identical creds; never clobbers.
- Vault auth via the existing controller token (env `VAULT_TOKEN` / `VAULT_TOKEN_FILE`).
- Creds dir `/etc/cloud-manager/creds/` created mode 0700, files 0600, root-owned.
- Mirror the reuse-or-generate Vault block from `cloud-manager-api/seed/playbooks/pg-14-jammy.yaml`.
- No secret value printed to stdout/stderr or logs.

## Definition of done
Script exists, is executable, and a dry run on the controller produces the three `.cred`
files; a second run reuses (no password change). Build step: N/A (shell). Run `Test/0001-TEST.md`.
