# Task 0003 — systemd unit wiring (LoadCredentialEncrypted) + vorch/porch RabbitMQ cred

## Goal
Ship unit drop-ins (in `vm-infra/secret-management`) that inject the creds, and remove every
plaintext secret from the units. Handle the RabbitMQ three-touch-point explicitly.

## Steps
1. **cloud-manager-api** drop-in:
   - `LoadCredentialEncrypted=VAULT_TOKEN:/etc/cloud-manager/creds/vault-token.cred`
   - `LoadCredentialEncrypted=DBPASSWD:/etc/cloud-manager/creds/db-password.cred`
   - `LoadCredentialEncrypted=RABBITMQ_PASSWORD:/etc/cloud-manager/creds/rabbitmq-password.cred`
   - Remove the plaintext `Environment=DBPASSWD=...` and `Environment=RABBITMQ_PASSWORD=...` lines.
2. **vorch_service** + **porch** drop-ins:
   - `LoadCredentialEncrypted=VAULT_TOKEN:/etc/cloud-manager/creds/vault-token.cred`
   - `Environment=VAULT_TOKEN_FILE=%d/vault-token` (Go client already honours this).
   - Remove the plaintext token from `config-server.yaml` (vorch) and `porch.service`.
3. **RabbitMQ for vorch/porch (the fiddly part):** vorch/porch read RabbitMQ creds from
   `config-server.yaml` today. Choose ONE and document it in deck.md:
   - (preferred) `LoadCredentialEncrypted=RABBITMQ_PASSWORD:...` + have vorch/porch read the
     cred file path, OR
   - have `cm-seed-secrets` write the rotated RabbitMQ password into `config-server.yaml`.
   Whichever — it must end with NO plaintext RabbitMQ password in any file.

## Definition of done
Drop-ins committed; after a daemon-reload + restart, `systemctl cat` shows no secret values and
the decrypted files appear under each service's `$CREDENTIALS_DIRECTORY` at 0400. Run `Test/0003-TEST.md`.
