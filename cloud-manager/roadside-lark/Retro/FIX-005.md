# FIX-005 — porch.service plaintext VAULT_TOKEN not stripped (quoted form missed)

## Task / Test
Task 0004 cutover / Task 0003 TC-002 (`systemctl cat … | grep hvs.` → no matches).

## Symptom
After the cutover, `systemctl cat porch` still showed:
```
Environment="VAULT_TOKEN=hvs.CAESILMFv47…"
```
The API and vorch units, `api.env`, and `config-server.yaml` were stripped
correctly; only porch retained the plaintext token.

## Root cause
Two compounding issues:
1. The strip used `sed '/^Environment=VAULT_TOKEN=/d'`, but porch.service writes
   the line **quoted**: `Environment="VAULT_TOKEN=…"`. The `"` after `=` meant the
   anchored pattern never matched, so the line survived.
2. Because the base unit still exported `VAULT_TOKEN`, `credenv.HydrateVaultToken`
   (which defers to an already-set `VAULT_TOKEN`) used the plaintext env value and
   ignored the cred — so porch was not actually sourcing the token from the
   credential. (Functionally OK only because the value is the same unrotated token.)

## Fix
Strip both quoted and unquoted forms:
```sh
sudo sed -i '/^Environment="\?VAULT_TOKEN=/d' /etc/systemd/system/porch.service
```
Updated `cm-cutover.sh` to use this pattern, then `daemon-reload` + `restart porch`.
With no `VAULT_TOKEN` in the env, `credenv` now reads
`$CREDENTIALS_DIRECTORY/VAULT_TOKEN` (the cred).

## Retest
`systemctl cat porch | grep hvs.` → no matches; porch `active` and able to mint the
per-run Vault token (proven by the end-to-end VM provision in Task 0005).
