# FIX-002 — `%d` specifier does not expand inside `Environment=` (vorch/porch)

## Task / Test
Task 0003 — unit wiring. Test 0003 TC-001 (creds materialize + service reads them).

## Symptom
A transient unit with
`Environment=VAULT_TOKEN_FILE=%d/VAULT_TOKEN` exposed the **literal** string
`%d/VAULT_TOKEN` to the process (not the credentials path), so the file was
unreadable:
```
VAULT_TOKEN_FILE=%d/VAULT_TOKEN readable=
cat: %d/VAULT_TOKEN: No such file or directory
```
The credentials themselves materialized correctly at `$CREDENTIALS_DIRECTORY`
(`-r--------`, 0400), so `LoadCredentialEncrypted=` is fine — only the `%d`
reference was wrong.

## Root cause
systemd does not perform `%d` specifier expansion inside `Environment=` *values*
(`%d` is resolved for command lines / a limited set of settings, not arbitrary
env values). The path was therefore passed verbatim.

## Fix
Stop depending on `%d`. systemd always exports `$CREDENTIALS_DIRECTORY` to a
service that has any `LoadCredential*=`, so:
- `internal/credenv` now derives the cred path from `$CREDENTIALS_DIRECTORY`
  (`$CREDENTIALS_DIRECTORY/VAULT_TOKEN`, `$CREDENTIALS_DIRECTORY/RABBITMQ_PASSWORD`)
  when the explicit `VAULT_TOKEN_FILE` / `RABBITMQ_PASSWORD_FILE` env is unset.
- The `Environment=…_FILE=%d/…` lines are removed from the vorch/porch drop-ins;
  the explicit `*_FILE` envs remain *supported* (for tests/overrides) but are no
  longer required.

## Retest
Transient unit with only `LoadCredentialEncrypted=` (no `*_FILE`): the built Go
helper resolves the token/password from `$CREDENTIALS_DIRECTORY`. Expect the
token to read back starting `hvs.`.
