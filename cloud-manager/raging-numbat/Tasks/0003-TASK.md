# Task 0003 — Write path: internal events

## Goal
Record `created`, `destroy_requested`, and `ip_address_updated` events from the existing `VirtualMachineService` methods.

## Files
- `cloud-manager-api/src/Services/CloudManager.Data.Services/VirtualMachineService.cs`

## Steps
1. Inject `IVmEventService` into `VirtualMachineService`.
2. `CreateVirtualMachineAsync` — after `SaveChangesAsync`, call `_events.RecordEventAsync(vm.Id, VmEventType.Created, new { vm.Name, vm.IpAddress, vm.MacAddress, vm.HostId }, actor: vm.CreatedBy)`.
3. `DeleteVirtualMachineAsync` — after stamping `DeletedAt`, call `_events.RecordEventAsync(vm.Id, VmEventType.DestroyRequested, new { vm.PublicId, vm.Name }, actor: "admin")` (use the same actor literal the existing code uses for CreatedBy until proper auth lands — see PLAN open question).
4. `UpdateIpAddressAsync` — after persisting, call `_events.RecordEventAsync(entity.Id, VmEventType.IpAddressUpdated, new { ip_address = ipAddress }, actor: "vorch-service")`.
5. Build clean.

## Notes
- All three writes happen *after* the underlying state change is persisted, so a failed event write never blocks the primary action.
- `Created` / `DestroyRequested` payloads are jsonb — include the public-id-safe identifiers the timeline view will need; do not embed UUIDs.

## Definition of done
Build clean. Creating a VM persists a `created` event row. Soft-deleting persists `destroy_requested`. IP write-back persists `ip_address_updated`.
