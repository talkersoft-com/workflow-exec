# Task 0011 — E2E verify

## Goal
Prove the full audit / retry-teardown loop end-to-end against a real VM.

## Steps

### Happy path (create → destroy → succeeded)
1. Deploy cloud-manager-api (`dotnet ef database update` then restart). Deploy cloud-manager-mcp and cloud-manager-web.
2. Create a VM via the API (e.g. `cloud_vm_create`).
3. `GET /api/v1/vm/{publicId}/events` → expect 1 row with `event_type=created`.
4. Soft-delete via the API (`DELETE /api/v1/vm/{publicId}`).
5. `GET /events` → expect `created`, `destroy_requested` (in that order, occurred_at DESC: destroy_requested first).
6. Wait for vorch's ack. `GET /events` → expect `teardown_succeeded` to appear within a few seconds.
7. Confirm in the cloud-manager-web VM detail page: timeline shows three events; Retry teardown button is NOT visible.

### Sad path (failed teardown → retry → success)
1. Pick (or stage) a VM whose vorch destroy will fail (e.g. libvirt domain already gone).
2. Soft-delete via the API.
3. `GET /events` → expect `destroy_requested` then `teardown_failed`.
4. In the web UI: timeline shows the failure; `Retry teardown` button IS visible.
5. Click `Retry teardown`. Confirm timeline shows `retry_teardown_requested` immediately, then a follow-up `teardown_succeeded` or `teardown_failed` after vorch acks.
6. POST `/retry-teardown` on a healthy VM (last event = `created`) → expect 400.

## Definition of done
Both paths complete. Timeline reflects all events in order. Retry button visibility matches the documented rule. The 400 guard rejects illegal retries.
