# Task 0005 — Retry-teardown service + endpoint

## Goal
Add `VirtualMachineService.RetryTeardownAsync` and `POST /api/v1/vm/{publicId}/retry-teardown`. The endpoint rejects with 400 unless the VM's most recent terminal event is `teardown_failed`.

## Files
- `cloud-manager-api/src/Services/CloudManager.Data.Services/Interface/IVirtualMachineService.cs`
- `cloud-manager-api/src/Services/CloudManager.Data.Services/VirtualMachineService.cs`
- `cloud-manager-api/src/CloudManager.API/Controllers/<VirtualMachineController>.cs`

## Steps

### Service
```csharp
Task<VmEvent> RetryTeardownAsync(string publicId, string actor);
```
1. Resolve the VM by public_id (`.IgnoreQueryFilters()` — VM is soft-deleted at this point).
2. Look up the most recent terminal lifecycle event for the VM (`teardown_succeeded` or `teardown_failed`).
3. If the most recent terminal event is not `teardown_failed` → throw `InvalidOperationException("VM is not in a teardown-failed state; retry-teardown is not applicable.")`. The controller maps this to 400.
4. Record `retry_teardown_requested` event with the actor.
5. Publish a fresh `destroy-vm` message to RabbitMQ using the exact same producer code the original destroy uses.
6. Return the new event row.

### Endpoint
```
POST /api/v1/vm/{publicId}/retry-teardown
→ 200 with the new VmEvent DTO on success
→ 400 with "VM is not in a teardown-failed state" on guard failure
→ 404 if the VM does not exist
```

## Notes
- The 400 guard is what makes retry-teardown safe. Operators cannot retry against healthy VMs by accident.
- The `actor` value comes from the request context (until proper auth lands, the controller passes `"admin"` matching project convention).

## Definition of done
Build clean. `POST /retry-teardown` returns 400 on a VM whose last event is `created` or `teardown_succeeded`. Returns 200 + new event on a VM whose last terminal event is `teardown_failed`, and produces a fresh `destroy-vm` message visible in vorch's log.
