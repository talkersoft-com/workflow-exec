# Task 0012 — Playwright + Vault audit verification

## Goal
Walk the A.6 / B.5 / C.5 acceptance scenarios from PLAN.md against the deployed system. Capture screenshots + Vault audit log snippets.

## Target repo(s)
None (verification only).

## Steps

### A.6 — Vault scoped-token end-to-end

1. Enable a transient Vault file-audit device if not already enabled (root token required):
   ```
   curl -sk -H "X-Vault-Token: <root>" \
        -X POST -d '{"type":"file","options":{"file_path":"/tmp/vault-audit.log"}}' \
        https://ubuntu-server.talkersoft.com:8200/v1/sys/audit/file
   ```
   If you can't enable audit (no root token in scope), use `journalctl -u cloud-manager-api` and the `playbook_runs` DB row as evidence instead.

2. Navigate to `/ansible/run` via Playwright. Pick `pg`, target `postgres-test`, click Run.

3. Inspect:
   - **DB**: `SELECT public_id, vault_secrets_prefix, vault_run_token IS NOT NULL AS has_token FROM vm.playbook_runs ORDER BY created DESC LIMIT 1;`
     - Expect: `vault_secrets_prefix` = `cloudmanager/data/playbook-secrets/<runId>`, `has_token` = `t`.
   - **API journal**: `sudo journalctl -u cloud-manager-api --since "2 minutes ago" | grep -iE "vault|hvs"` — token literals should appear as `hvs.<REDACTED>`.

4. Wait for terminal status. Re-query the run row; `vault_run_token` should be NULL (revoked).

5. **Token works during run, fails after**:
   - During the run, grab the token from API logs (it'll be redacted — fallback: use a dev override to print once for verification; document the override removal).
   - Alternative: don't grab the token; instead, verify via Vault audit log entries `token/create` at start and `token/revoke` at end.

6. Capture: `Results/A6-vault-audit.md` with the relevant audit log lines and DB row snapshots.

### B.5 — Requirements enforcement

1. Navigate to `/ansible/playbooks/pb_EMEJCH08GG` (pg). Click "+ Add requirement".
2. Modal: select `community.postgresql`, min version `4.0.0`, blank max. Submit.
3. Confirm the row appears with status badge "Installed" (because the collection IS installed on the controller from lilac-redpanda).
4. Navigate to `/ansible/collections`. Click Remove on `community.postgresql`. Wait until Removed.
5. Back to `/ansible/playbooks/pb_EMEJCH08GG`. Banner now red: "Missing required collections for the current controller." with Fix link. Screenshot: `Results/B5-banner-unmet.png`.
6. Navigate to `/ansible/run`. Pick `pg`, target `postgres-test`. Run button disabled with inline message naming `community.postgresql`. Screenshot: `Results/B5-run-blocked.png`.
7. Back to Collections, Install `community.postgresql`. Wait for Installed.
8. Run page: button re-enabled, no error. Capture: `Results/B5-run-enabled.png`.

2. `/ansible/collections`: controller dropdown shows `ubuntu-server`. Refresh — selection persists. Screenshot: `Results/C5-selector.png`.
3. Optionally, INSERT a stub second controller row:
   ```
   PGPASSWORD='P@ssw0rd!' psql -h localhost -U cloudmanager -d clouddb -c \
     "INSERT INTO bare_metal.hosts (id, public_id, name, description, network_configuration_id, is_controller, created, created_by) \

      FROM bare_metal.hosts WHERE name = 'ubuntu-server';"
   ```
