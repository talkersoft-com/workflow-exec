# Task 0002 — CloudManager.Vault.Client project + redaction enricher

## Goal
A new C# project under `cloud-manager-api/src/Services/CloudManager.Vault.Client/` exposing `IVaultClient` with token/policy CRUD. Plus a Serilog enricher (or equivalent) that redacts `hvs.<token>` strings from all log output.

## Target repo(s)
`cloud-manager-api`

## Steps

1. **Create the new project**:
   ```
   cd vm-infra/cloud-manager/cloud-manager-api/src/Services
   dotnet new classlib -n CloudManager.Vault.Client -o CloudManager.Vault.Client
   dotnet sln ../CloudManager.sln add CloudManager.Vault.Client/CloudManager.Vault.Client.csproj
   ```

2. **Add reference from CloudManager.API**:
   ```
   cd ../CloudManager.API
   dotnet add reference ../Services/CloudManager.Vault.Client/CloudManager.Vault.Client.csproj
   ```
   Plus add a reference from `CloudManager.Data.Services` so `PlaybookRunService` can call it.

3. **Write `IVaultClient.cs`** with the surface from PLAN.md:
   ```csharp
   public record VaultPolicy(string Name);
   public record VaultChildToken(string ClientToken);

   public interface IVaultClient
   {
       Task<VaultPolicy> WritePolicyAsync(string name, string hclBody);
       Task DeletePolicyAsync(string name);
       Task<VaultChildToken> CreateChildTokenAsync(string[] policies, TimeSpan ttl, string displayName, bool noParent, bool renewable);
       Task RevokeTokenAsync(string token);
   }
   ```

4. **Write `VaultClient.cs`** — HTTP implementation:
   - Constructor takes `HttpClient` (via `IHttpClientFactory`), `VaultAddress`, `RootToken` (from `IConfiguration` keys `Vault:Address` and `Vault:Token` — fed by `VAULT_ADDR` + `VAULT_TOKEN` env vars at startup).
   - All HTTP calls authenticate with `X-Vault-Token: <root token>`.
   - JSON request bodies use `System.Text.Json` with snake_case for Vault's expectations (e.g. `display_name`, `no_parent`).
   - On non-2xx: throw `VaultException(int statusCode, string body)`.

5. **`VaultException` class** in the same project, plain Exception subclass.

6. **DI registration**:
   - New `ConfigureServices.cs` in the Vault.Client project with:
     ```csharp
     services.AddHttpClient<IVaultClient, VaultClient>(c => { /* no base addr — set per-call */ });
     ```
   - Call from `Program.cs` in CloudManager.API.

7. **Config wiring** in `Program.cs`:
   - Read `VAULT_ADDR` + `VAULT_TOKEN` env vars and bind to `Vault:Address` and `Vault:Token` config sections.

8. **Serilog redaction enricher**:
   - Check whether the API already uses Serilog (look at `Program.cs` for `UseSerilog` / `AddSerilog`). If yes: add a `LogEventEnricher` that scans every property value for `hvs\.[A-Za-z0-9_\-]+` and replaces with `hvs.<REDACTED>`. Register it.
   - If the API uses the default Microsoft.Extensions.Logging: add an `ILoggerProvider` wrapper that intercepts log messages and runs the same regex. Document the choice in `Retro/CHANGE-0002.md`.

9. **Build**:
   ```
   cd vm-infra/cloud-manager/cloud-manager-api/src && dotnet build CloudManager.sln -c Release
   ```
   Expect 0 errors.

10. Write `Retro/CHANGE-0002.md`.

## Definition of done
- New `CloudManager.Vault.Client.csproj` builds clean and is referenced by CloudManager.API + CloudManager.Data.Services.
- `IVaultClient` registered in DI.
- Redaction layer present — grep proves it (`grep -nE "hvs\\\\." src/CloudManager.API/Program.cs` or equivalent).
- A quick smoke test: `curl https://ubuntu-server.talkersoft.com:8200/v1/sys/health -ki` works (Vault reachable; not API code, just env sanity).
- `Retro/CHANGE-0002.md` exists.
