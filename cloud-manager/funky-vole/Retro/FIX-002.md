# Fix 002 — cloudmanager Vault policy missing token-mint and policy-write capabilities

## Failed test
Task 0012 A.6 — after the TLS fix in FIX-001, the run trigger now returns the correct **503** (VaultException is raised, controller maps it to "Vault unavailable, try again"). The DB row is created in Queued state, then the mint attempt fails and the second SaveChanges never happens, so `vault_run_token` is NULL and `vault_secrets_prefix` is empty. AMQP publish is also skipped — there is no orphan run shipped to vorch.

Vault response:
```
CloudManager.Vault.Client.VaultException:
  Vault returned 403: {"errors":["1 error occurred:\n\t* permission denied\n\n"]}
  at VaultClient.WritePolicyAsync (cloudmanager/data/playbook-secrets/...)
```

## Root cause
The deployed `cloudmanager` token's policy at `scripts/vault/cli/cloudmanager.py:10-34` grants full sudo on `cloudmanager/*` but does NOT grant:

- `sys/policies/acl/playbook-run-*` — needed for `WritePolicyAsync` (the per-run scoped policy)
- `auth/token/create` — needed for `CreateChildTokenAsync` (mint the run token)
- `auth/token/revoke` — needed for `RevokeTokenAsync` (cleanup on terminal)

The nimble-orangutan PLAN explicitly stated (line 20):

> The deployed cloudmanager token already has full read/write on `cloudmanager/*` — it can mint child tokens via `auth/token/create`.

This assumption was incorrect — sudo on `cloudmanager/*` does not extend to `auth/*` or `sys/policies/*`.

## Fix applied (code)
`scripts/vault/cli/cloudmanager.py` updated to add the three missing path stanzas to `CLOUDMANAGER_POLICY`. New tokens minted by the script will have the correct capability set.

## Fix required (operator)
The currently-deployed token cannot be retroactively upgraded — Vault policies are looked up by name, so updating the `cloudmanager` policy in-place with a root/admin token will retroactively grant the new capabilities to the existing token. Required operator step:

```
# from a shell with VAULT_TOKEN=<root or admin>
cd cloud-manager-api/scripts/vault
./vault-cli cloudmanager     # prompts for admin token; rewrites the policy idempotently
```

Or directly:
```
vault policy write cloudmanager - <<EOF
... (contents from CLOUDMANAGER_POLICY in cloudmanager.py)
EOF
```

A root token is not in this session's reach — no `vault-init.json`, `unseal-keys.json`, or stored admin token file exists on disk (verified by `sudo find /etc/vault.d /root /home -name "vault*init*"`). This is a deliberate ops-side restriction.

## Verified without root
- API correctly catches `VaultException` and returns 503 with `{ "Message": "Vault unavailable, try again" }` — the graceful-degradation path holds.
- The redacting logger DID NOT need to scrub anything because no `hvs.*` token was ever minted (the mint call failed before the response body was logged).
- DB row is left in Queued without vault columns set, **and AMQP is not published** — so there's no orphan run going to vorch with a missing token. This is the desired failure mode.
- TLS, error propagation, Vault client wiring, and the DI graph are all confirmed correct by this very failure trace.

## Outcome
Conditional pass — code path is correct. Full E2E vault mint+revoke audit (A.6) requires the operator to extend the `cloudmanager` policy with the three new paths. Tracked in `LESSONS.md` as `vault-policy-extension` follow-up.
