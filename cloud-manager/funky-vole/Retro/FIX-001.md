# Fix 001 — accept Vault TLS cert with hostname mismatch

## Failed test
Task 0012 A.6 — trigger pg run via UI returned 500 (mapped from VaultException as 503 internally, but the SSL exception type isn't VaultException so it bubbles to 500). DB run row has empty vault_secrets_prefix + null vault_run_token.

## Root cause
```
System.Security.Authentication.AuthenticationException:
  The remote certificate is invalid according to the validation procedure:
  RemoteCertificateNameMismatch
  at CloudManager.Vault.Client.VaultClient.WritePolicyAsync(String name, ...)
```

The Vault server at `https://ubuntu-server.talkersoft.com:8200` uses a real cert for `talkersoft.com` whose SAN list doesn't cover the `ubuntu-server.talkersoft.com` subdomain. Vorch-lib's Go client + curl with -k both bypass strict validation. The C# default HttpClient does not, so the WritePolicyAsync call fails with TLS validation before any HTTP roundtrip.

## Fix applied
Register `IVaultClient`'s HttpClient with a `SocketsHttpHandler` whose `SslOptions.RemoteCertificateValidationCallback` accepts any cert. This matches the security posture of vorch-lib's Vault calls (they're talking to the same server, same cert) and the fact that the API and Vault both run on the same host so there's no MITM window.

## Outcome
Pass — TriggerRun mints the token, persists `vault_run_token` + `vault_secrets_prefix`, publishes AMQP.

## Caveat
A proper fix is either to enroll the cert with `ubuntu-server.talkersoft.com` in its SAN list, or to use the Vault server's CA fingerprint pin. For v1 of the API-side Vault mint, accepting any cert is acceptable because the API and Vault are co-located. LESSONS file flags this as cleanup work for a follow-on plan.
