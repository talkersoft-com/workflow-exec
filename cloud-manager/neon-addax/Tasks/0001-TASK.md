# Task 0001 — Migration: AddAnsibleConstraintsAndTargetTimestamps

## Goal
Land the three schema changes from `DATA-MODEL-DELTAS.md` as a single EF Core
migration. These are prerequisites for Phase 1+ implementation tasks.

## Target repo
`cloud-manager-api`

## Reference
- `cloud-manager/planning/planning/cloud-manager-ansible/DATA-MODEL-DELTAS.md` — full spec
- `cloud-manager/vm-infra/cloud-manager/cloud-manager-api/src/Models/CloudManager.Entities/Models/` — entity classes
- `cloud-manager/vm-infra/cloud-manager/cloud-manager-api/src/Models/CloudManager.Entities/CloudManagerDbContext.cs` — model config

## Steps

1. Edit `src/Models/CloudManager.Entities/Models/PlaybookRunTarget.cs` — add:
   ```csharp
   public DateTime? StartedAt { get; set; }
   public DateTime? CompletedAt { get; set; }
   ```
2. Edit `CloudManagerDbContext.OnModelCreating` (or equivalent fluent-config location):
   - Add `entity.HasIndex(e => new { e.RoleId, e.FilePath }).IsUnique();` on the `RoleFile` config (or `modelBuilder.Entity<RoleFile>().HasIndex(...)`).
   - Add `entity.HasIndex(e => new { e.PlaybookRoleId, e.FilePath }).IsUnique();` on the `PlaybookRoleFile` config.
   - Configure the new `StartedAt` / `CompletedAt` columns with snake_case mapping (`started_at`, `completed_at`) consistent with the existing convention.
3. From `src/Models/CloudManager.Entities/`, run:
   ```
   dotnet ef migrations add AddAnsibleConstraintsAndTargetTimestamps --output-dir Migrations
   ```
   Confirm the generated migration creates the two unique indexes and the two columns; no other changes should appear in the diff (`dotnet ef migrations script <prev> AddAnsibleConstraintsAndTargetTimestamps` to inspect).
4. Apply against `clouddb` (localhost:5432):
   ```
   dotnet ef database update
   ```
5. Verify via the database-toolkit MCP that:
   - `vm.playbook_run_targets` has `started_at` and `completed_at` (timestamp with time zone, nullable).
   - Two unique indexes exist (`ix_role_files_role_id_file_path_unique` and `ix_playbook_role_files_playbook_role_id_file_path_unique`, or whatever names EF generated — accept EF's naming if it's stable).
6. Run `dotnet build` from the solution root — must exit 0.

## Definition of done
Migration file exists under `Migrations/`. Migration is applied to clouddb.
`PlaybookRunTarget.cs` has the two new properties. `dotnet build` passes.
The two unique indexes and two columns are present in the live schema (verified
via db_get_table_schema).
