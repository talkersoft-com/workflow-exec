# Task 0003 — API Phase 2: revision write-side + revision read routes

## Goal
PATCH on a role file auto-creates a new `RoleFileRevision` row in the same EF
transaction. Read-side routes expose the revision history and a "restore"
endpoint that creates a new revision matching a historic one.

## Target repo
`cloud-manager-api`

## Reference
- `cloud-manager/planning/planning/cloud-manager-ansible/API-PLAN.md` — sections
  "Role file revisions" and "Service-layer hooks → `RoleFileService.UpdateAsync`"

## What to build

### DTOs
- `RoleFileRevision.cs` — `{ publicId, revision_number, content, change_summary, created, created_by }`
- `RoleFileRevisionSummary.cs` — `{ publicId, revision_number, change_summary, created, created_by }`

### Service updates
- `RoleFileService.UpdateAsync`:
  1. Open EF transaction (or rely on SaveChanges-in-one-shot if all writes are tracked).
  2. Compute `revision_number = (max RoleFileRevision.revision_number for this file) + 1`. Use `?? 0` for first revision.
  3. Insert new `RoleFileRevision` row with `role_file_id`, `revision_number`, `content` = NEW content, `change_summary` (optional from request body, default `"edit"`).
  4. Update `RoleFile.content` and audit fields.
  5. Commit.
- New `IRoleFileRevisionService` + `RoleFileRevisionService` — list per file (newest first), get by publicId, **restore**: load the historic revision, then call `RoleFileService.UpdateAsync` with that content + `change_summary = "restore from rev N"`. This re-enters the auto-revision path and creates a NEW revision (never deletes history).

### Controller
- `RoleFileRevisionController.cs` at `[Route("api/v1/ansibleRole/{roleId}/file/{fileId}/revision")]` with `[RequireFeatureFlag("playbooks")]`:
  - GET (list, newest first)
  - GET/{revisionId}
  - POST/{revisionId}/restore — body `{ change_summary? }` → returns the new `RoleFileRevisionDto`

### Request body extension
- Extend the existing `RoleFileController` `UpdateRoleFileRequest` to optionally accept `change_summary`. If absent, default to `"edit"`.

## Steps
1. Build the DTOs + service updates + new controller.
2. Register `IRoleFileRevisionService` in `ConfigureServices.cs`.
3. `dotnet build` — must exit 0.
4. Smoke-test:
   - Create a role + file (from Task 0002).
   - PATCH the file 3 times with different content + change_summary values → each PATCH returns 200; database has 3 `role_file_revisions` rows with `revision_number` 1, 2, 3.
   - GET `…/file/{fileId}/revision` → array of 3, newest first.
   - GET `…/revision/{revisionId}` for revision 1 → returns the original content.
   - POST `…/revision/{revisionId}/restore` for revision 1 → returns a NEW revision (number 4) whose content equals revision 1's.
5. Confirm no revision row is created if PATCH content is unchanged.
6. Clean up test rows.

## Definition of done
PATCH creates a new revision row every time content actually changes, never
when unchanged. Restore creates a new revision (never deletes). All four
revision routes respond correctly. `dotnet build` passes.
