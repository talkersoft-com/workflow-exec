# Task 0009 — API Phase 8: multi-VM run-trigger endpoint

## Goal
Add `POST /api/v1/playbook/{id}/run` for triggering a multi-VM run. Creates
one `PlaybookRun` + N `PlaybookRunTarget` rows in one transaction and
publishes a single message to `playbook-runs`.

## Target repo
`cloud-manager-api`

## Reference
- `cloud-manager/planning/planning/cloud-manager-ansible/API-PLAN.md` — section "Multi-VM run trigger"

## What to build

### Request DTO
- `TriggerPlaybookRunRequest`: `{ vmIds: string[], varsOverride?: object }`. `vmIds` is a list of VM publicIds. `varsOverride` is a JSON object (serialized as `string` server-side for the DB column).

### Service updates
- New method on `IPlaybookRunService` (or a new `IPlaybookMultiRunService` if the existing one feels overloaded): `TriggerMultiVm(string playbookPublicId, IList<string> vmPublicIds, string varsOverrideJson)`.
  1. Validate: `vmIds` non-empty, every VM resolves, playbook exists with non-empty `content`. Throw `InvalidOperationException` (→ controller 400) on any failure.
  2. In one EF transaction:
     - Create `PlaybookRun` with `playbook_id`, `playbook_revision_id` = latest revision, `status = Queued`.
     - For each VM: look up `VmPlaybookAssignment` for (vm, playbook). If present, snapshot the assignment's `vars_override` onto the target's `vars_override_snapshot` and set `assignment_id`. If not present, use the request's `varsOverride` (or `{}`) and `assignment_id = null`.
     - Insert `PlaybookRunTarget` rows with `status = Queued`.
     - Commit.
  3. Publish `{ runId }` via `IPlaybookRunPublisher.EnqueueRun(runId)`.
  4. Return the created `PlaybookRun`.

### Controller
- New method on `PlaybookController`:
  ```csharp
  [HttpPost("{id}/run")]
  public async Task<IActionResult> TriggerRun(string id, [FromBody] TriggerPlaybookRunRequest req)
  ```
  Returns 202 Accepted with `{ runId: publicId }`. 400 on validation. 404 on missing playbook.

## Steps
1. Build the request DTO + service method + controller method.
2. `dotnet build` — must exit 0.
3. Smoke-test:
   - Create a playbook with content.
   - POST `/api/v1/playbook/{p}/run` with `{ vmIds: [vm1, vm2], varsOverride: { db_name: "test" } }` → 202 with `runId`.
   - GET `/api/v1/run/{runId}` → exists, status `Queued`.
   - GET `/api/v1/run/{runId}/target` → 2 target rows, each with `vars_override_snapshot = {"db_name":"test"}` and `assignment_id = null`.
   - Confirm via RabbitMQ management UI (or by checking queue depth) that one message landed on `playbook-runs`.
   - Repeat with an existing `VmPlaybookAssignment` for one of the VMs → that target inherits the assignment's `vars_override` and carries the `assignment_id`.
4. Error cases:
   - Empty `vmIds` → 400.
   - Unknown VM publicId → 400.
   - Playbook with empty content → 400.
5. Clean up test rows.

## Definition of done
Endpoint creates correct rows + publishes message. Single-VM apply
(`PlaybookAssignmentController.Apply`) is unchanged and still works.
`dotnet build` passes.
