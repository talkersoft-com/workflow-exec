# Task 0002 — API Phase 1: AnsibleRole + RoleFile CRUD

## Goal
Land the smallest meaningful slice of the new API surface: full CRUD on
`AnsibleRole` and `RoleFile` with no revisions and no argument-specs hook.
Operators can create global roles and edit their files; history is lost on
save (added in Task 0003).

## Target repo
`cloud-manager-api`

## Reference
- `cloud-manager/planning/planning/cloud-manager-ansible/API-PLAN.md` — sections
  "Ansible roles (global library)" and "Role files"
- `src/CloudManager.API/Controllers/PlaybookController.cs` — mirror this pattern

## What to build

### DTOs (`src/Models/CloudManager.DTO/Models/`)
- `AnsibleRole.cs` — `{ publicId, name, description, created, modified, fileCount }`
- `RoleFile.cs` — `{ publicId, roleId, file_path, content, modified }` (renamed `roleId` is the role's publicId, not internal Guid)
- `RoleFileSummary.cs` — `{ publicId, file_path, modified }` (list response)

### AutoMapper profiles (`src/Services/CloudManager.Data.Services/MappingProfiles/`)
- `AnsibleRoleProfile.cs`
- `RoleFileProfile.cs`

### Services (`src/Services/CloudManager.Data.Services/`)
- `IAnsibleRoleService` + `AnsibleRoleService` — list / get by publicId / create / update / delete. Delete throws `InvalidOperationException` if any `PlaybookGlobalRoleRef` row references it (controller catches → 409).
- `IRoleFileService` + `RoleFileService` — list per role / get / create / update / delete. Path validation: extract a static helper `RoleFilePathValidator.Validate(string path, int contentBytes)` that throws `InvalidOperationException` on rule violation (reject `..`, leading `/` or `~`, backslashes, content > 1 MiB).
- Register both in `ConfigureServices.cs`.

### Controllers (`src/CloudManager.API/Controllers/`)
- `AnsibleRoleController.cs` at `[Route("api/v1/[controller]")]` with `[RequireFeatureFlag("playbooks")]`. Verbs: GET (list), GET/{id}, POST (Register), PATCH/{id}, DELETE/{id}.
- `RoleFileController.cs` at `[Route("api/v1/ansibleRole/{roleId}/file")]` with `[RequireFeatureFlag("playbooks")]`. Verbs: GET (list), GET/{fileId}, POST, PATCH/{fileId}, DELETE/{fileId}.
- Error model matches existing controllers: `{ Message: "..." }`, 400/404/409/500. XML doc comments on every method.

## Steps
1. Create the DTOs + profiles + services + controllers per the spec above.
2. Register the new services in `ConfigureServices.cs`.
3. `dotnet build` — must exit 0.
4. Smoke-test happy paths:
   - `POST /api/v1/ansibleRole` with `{"name":"test","description":"x"}` → 201
   - `GET /api/v1/ansibleRole` → list includes the new role
   - `POST /api/v1/ansibleRole/{publicId}/file` with `{"file_path":"tasks/main.yml","content":"---\n"}` → 201
   - `PATCH /api/v1/ansibleRole/{publicId}/file/{filePublicId}` with `{"content":"---\n- name: test\n"}` → 200
   - `DELETE /api/v1/ansibleRole/{publicId}` → 204
5. Smoke-test error paths:
   - File-path `"../etc/passwd"` → 400
   - Content > 1 MiB → 400
   - Duplicate file_path within role → 400 (DB-side unique index from Task 0001 catches the race)
   - Delete role with attached `PlaybookGlobalRoleRef` → 409 (skip if no refs exist yet; revisit in Task 0005)
6. Clean up test data: `DELETE /api/v1/ansibleRole/{publicId}` for any temp rows.

## Definition of done
Both controllers exist, registered, and respond correctly to the happy and
error paths above. `dotnet build` passes. New DTOs use `publicId` on the wire,
not internal Guids.
