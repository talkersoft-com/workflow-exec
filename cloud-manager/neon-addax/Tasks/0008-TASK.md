# Task 0008 ŌĆö API Phase 7: materialization endpoint

## Goal
Single endpoint that the worker hits to fetch a self-contained playbook
snapshot (playbook content + all referenced global roles + all local roles
+ a manifest of `x-secret-path` inputs to resolve).

## Target repo
`cloud-manager-api`

## Reference
- `cloud-manager/planning/planning/cloud-manager-ansible/API-PLAN.md` ŌĆö section "Materialization API (for vorch-service)"

## What to build

### DTO
- `MaterializedPlaybook.cs`:
  ```csharp
  public class MaterializedPlaybook {
      public string PlaybookPublicId { get; set; }
      public string PlaybookRevisionId { get; set; }
      public List<MaterializedFile> Files { get; set; }
      public List<SecretInputRef> SecretInputs { get; set; }
  }
  public class MaterializedFile { public string RelativePath { get; set; } public string Content { get; set; } }
  public class SecretInputRef { public string ExtravarName { get; set; } public string VaultPath { get; set; } }
  ```
  Wire-format names: `playbook_public_id`, `playbook_revision_id`, `files`, `secret_inputs`, `relative_path`, `content`, `extravar_name`, `vault_path` (snake_case per existing convention).

### Service
- `IPlaybookMaterializationService` + impl with one method `Materialize(string playbookPublicId, string? revisionPublicId)`. Logic:
  1. Resolve the playbook. If `revisionPublicId` given, use that revision's content; otherwise use current `Playbook.content`.
  2. Build `files[]`:
     - Add the playbook content under a top-level path (default `site.yml` ŌĆö the worker writes it as the entry-point file).
     - For every global role referenced via `PlaybookGlobalRoleRef`, add every `RoleFile` under `roles/{role.name}/{file.file_path}` using the role's CURRENT head content.
     - For every `PlaybookRole` owned by the playbook, add every `PlaybookRoleFile` under `roles/{role.name}/{file.file_path}`. If a name collides with a global role, the local role wins (matches ansible convention).
  3. Build `secret_inputs[]`:
     - Find every file in `files[]` whose `relative_path` matches `roles/*/meta/argument_specs.yml`.
     - Parse each as YAML, walk the `argument_specs.<entrypoint>.options.*` tree.
     - For every option with an `x-secret-path` annotation, emit `{ extravar_name: option_name, vault_path: <annotation value> }`.
     - Deduplicate by `extravar_name` (last wins; document this).
  4. Return the DTO.
- Use any reasonable YAML parser already in the project (YamlDotNet or similar ŌĆö check existing deps).

### Controller
- New method on `PlaybookController`:
  ```csharp
  [HttpGet("{id}/materialized")]
  public async Task<IActionResult> Materialize(string id, [FromQuery] string? revision = null)
  ```
  Returns the `MaterializedPlaybook` DTO.

## Steps
1. Build the DTO + service + controller method.
2. Register `IPlaybookMaterializationService` in `ConfigureServices.cs`.
3. `dotnet build` ŌĆö must exit 0.
4. Smoke-test:
   - Create a global role with a `tasks/main.yml` and a `meta/argument_specs.yml` containing one option with `x-secret-path: cloudmanager/data/test/secret`.
   - Create a playbook, attach the role, give the playbook some content.
   - GET `/api/v1/playbook/{publicId}/materialized` ŌåÆ response includes `site.yml`, the role's tasks file, the argument_specs file, and `secret_inputs[]` with the one entry.
   - GET with `?revision={older-revision-publicId}` ŌåÆ response uses that revision's playbook content but current role file content.
5. Clean up.

## Definition of done
Endpoint returns the correct file tree + secret_inputs manifest for both
current-revision and historic-revision queries. `dotnet build` passes.
