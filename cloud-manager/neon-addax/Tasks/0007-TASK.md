# Task 0007 — API Phase 6: PlaybookRunTarget reads

## Goal
Expose per-run target rows for the UI's `AnsibleRunDetailPage`. Read-only; the
worker writes these rows during execution.

## Target repo
`cloud-manager-api`

## Reference
- `cloud-manager/planning/planning/cloud-manager-ansible/API-PLAN.md` — section "Playbook run targets"

## What to build

### DTOs
- `PlaybookRunTarget.cs` — `{ publicId, runId, vmId, vmName, assignmentId?, status, stats, output, started_at?, completed_at? }`. Joins to VM for `vmName`. Includes the new `started_at`/`completed_at` from Task 0001's migration.
- `PlaybookRunTargetSummary.cs` — `{ publicId, vmId, vmName, status, started_at?, completed_at? }` (for list views; omits the bulky `stats`/`output`)

### Service
- `IPlaybookRunTargetService` + impl — list per run, get by target publicId. Internal write methods used by the worker's API calls (for status / stats / output updates) — but those are internal to the service for now; if there's no auth distinction between API consumers, expose them on the same controller for the worker to call.

### Controller
- `PlaybookRunTargetController` at `[Route("api/v1/run/{runId}/target")]` with `[RequireFeatureFlag("playbooks")]`:
  - GET (list summaries)
  - GET/{targetId} (full)
  - PATCH/{targetId} `{ status?, stats?, output?, started_at?, completed_at? }` — write-side for the worker. (Note: in a future workflow, this should be scoped to a worker-only credential; for now flag-gated only.)

## Steps
1. Build DTO + service + controller.
2. Register `IPlaybookRunTargetService` in `ConfigureServices.cs`.
3. `dotnet build` — must exit 0.
4. Smoke-test:
   - Insert a synthetic `PlaybookRun` + 2 `PlaybookRunTarget` rows via SQL (the multi-VM trigger that creates them lands in Task 0009 — for now hand-insert for testing).
   - GET `/api/v1/run/{runPublicId}/target` → list of 2 with summary fields.
   - GET `/api/v1/run/{runPublicId}/target/{targetPublicId}` → full row with stats + output (empty for now).
   - PATCH the target with `{"status":"running","started_at":"2026-05-27T20:00:00Z"}` → 200; row updated.
   - PATCH again with `{"status":"succeeded","completed_at":"2026-05-27T20:05:00Z","stats":{"ok":5,"failed":0}}` → 200.
5. Clean up the synthetic rows.

## Definition of done
List + detail + PATCH endpoints respond correctly with the new
`started_at`/`completed_at` columns serialized. `dotnet build` passes.
