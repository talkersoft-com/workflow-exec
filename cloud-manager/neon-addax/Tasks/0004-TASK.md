# Task 0004 — API Phase 3: PlaybookRevision auto-create + read routes

## Goal
PATCH on `Playbook.content` auto-creates a `PlaybookRevision` row. Read-side
routes expose the revision history of a playbook.

## Target repo
`cloud-manager-api`

## Reference
- `cloud-manager/planning/planning/cloud-manager-ansible/API-PLAN.md` — section "Playbook revisions"
- `src/CloudManager.API/Controllers/PlaybookController.cs` — existing controller to modify

## What to build

### DTOs
- `PlaybookRevision.cs` — `{ publicId, revision_number, content, change_summary, created, created_by }`
- `PlaybookRevisionSummary.cs` — `{ publicId, revision_number, change_summary, created, created_by }`

### Service updates
- `PlaybookService.UpdateAsync`:
  1. Detect whether incoming `Content` differs from stored value (use the actual current value, not the request body's other fields).
  2. If unchanged or null: no revision row.
  3. If changed: in same EF transaction, insert `PlaybookRevision` with `playbook_id`, `revision_number = (max + 1) ?? 1`, `content` = NEW content, `change_summary` (from request, default `"edit"`).
  4. Update `Playbook.content` and audit fields.
  5. Commit.
- New `IPlaybookRevisionService` + `PlaybookRevisionService` — list per playbook, get by publicId. No write methods (revisions are write-through from Playbook PATCH).

### Controller updates
- Modify `PlaybookController`'s `UpdatePlaybookRequest` to accept an optional `change_summary` field (default `"edit"` in the service).
- New `PlaybookRevisionController` at `[Route("api/v1/playbook/{playbookId}/revision")]`:
  - GET (list, newest first)
  - GET/{revisionId}

## Steps
1. Build DTOs + service + controller per the spec.
2. Register `IPlaybookRevisionService` in `ConfigureServices.cs`.
3. `dotnet build` — must exit 0.
4. Smoke-test:
   - Create a playbook with content `"---\nhosts: all\n"`.
   - PATCH content to `"---\nhosts: all\ntasks: []\n"` → 200; one `playbook_revisions` row exists.
   - PATCH content to the SAME value again → 200; still only one revision (no duplicate).
   - PATCH content again with a new value → 200; two revisions now.
   - GET `…/revision` → array of 2, newest first.
   - GET `…/revision/{revisionId}` → full historic content returned.
5. Clean up.

## Definition of done
Playbook PATCH on content creates a revision once per actual change.
Revision routes respond correctly. `dotnet build` passes.
