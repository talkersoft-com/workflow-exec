# Task 0010 — API Phase 9: run-cancel endpoint + IPlaybookRunCancelPublisher

## Goal
Add `POST /api/v1/run/{runId}/cancel`. Queued runs flip directly to
`Cancelled`. Running runs publish a `{ runId }` message to a new
`playbook-run-cancel` queue for the worker to react to.

## Target repo
`cloud-manager-api`

## Reference
- `cloud-manager/planning/planning/cloud-manager-ansible/API-PLAN.md` — section "Run cancellation"
- `src/Services/CloudManager.AMQP.Publisher/` — pattern for the existing `IPlaybookRunPublisher`

## What to build

### Publisher
- `IPlaybookRunCancelPublisher` + impl in `src/Services/CloudManager.AMQP.Publisher/`. Method: `EnqueueCancel(Guid runId)`. Publishes `{ "runId": "run_xxx" }` to a queue named **exactly** `playbook-run-cancel` (the vorch implementation will subscribe by this name).
- Register in DI.

### Service updates
- New method on `IPlaybookRunService`: `Cancel(string runPublicId)`. Logic:
  - Load run. If `status` is terminal (`Succeeded`, `Failed`, `Cancelled`) → throw `InvalidOperationException` (→ controller 409).
  - If `status == Queued`: update `status = Cancelled`, `completed_at = now()`, save. Return updated run.
  - If `status == Running`: publish `{ runId }` via the new cancel publisher. Do NOT flip status here — the worker will write the final `Cancelled` once it finalizes the in-progress target. Return the run unchanged (status still `Running`).

### Controller
- New method on `RunController`:
  ```csharp
  [HttpPost("{runId}/cancel")]
  public async Task<IActionResult> Cancel(string runId)
  ```
  Returns 200 with the (possibly-updated) `PlaybookRunDto`. 404 if not found. 409 if already terminal.

## Steps
1. Build the publisher + service method + controller method.
2. Register the publisher in `ConfigureServices.cs`.
3. `dotnet build` — must exit 0.
4. Smoke-test:
   - Queued run: trigger via Task 0009's endpoint, immediately cancel → status flips to `Cancelled`, `completed_at` set.
   - Already-cancelled run: cancel again → 409.
   - Running run (simulate by directly UPDATEing `playbook_runs.status` to `running` for a test row): cancel → 200, message lands on `playbook-run-cancel` queue (verify via RabbitMQ management UI), status stays `Running` in DB until worker writes the final state.
   - Succeeded run: cancel → 409.
5. Clean up test rows.

## Definition of done
- Cancel endpoint behaves correctly for all 4 starting states (Queued, Running, Succeeded, Failed).
- A message lands on `playbook-run-cancel` for Running runs.
- `dotnet build` passes.
