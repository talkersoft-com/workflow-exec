# Test 0001
TC-001 mode round-trip: `POST api/v1/playbook/{pid}/run` with no `mode` persists `apply` and
the run DTO shows `mode: "apply"`; with `"mode": "check"` the DTO shows `check`; with
`"mode": "bogus"` → 400. TC-002 trigger parity: a trigger request body WITHOUT `mode` produces
a run record and queue message byte-identical to pre-change behavior (no new required fields,
`PlaybookRunMessage` untouched). TC-003 lint trigger: `POST api/v1/playbook/{pid}/lint` →
202 + runId; a `{"runId"}` message lands on `playbook-lints`; lint run status is one of the
five existing strings at every stage — grep migrations/DTOs for any sixth status string, zero
hits. TC-004 lint findings: worker-auth POST of an ansible-lint JSON fixture persists `plint`
findings bound to the playbook revision and marks the run terminal; unauthenticated POST →
401/403. TC-005 chunks: ingest N chunks per two targets with interleaved Seq; GET with
`afterSeq` returns only later chunks in Seq order per target; a >32KB chunk is rejected.
TC-006 events: `playbook_check_completed` and `playbook_lint_completed` are valid additive
`vm_events` types; existing event types unchanged. TC-007 public ids only: every new DTO id
matches `^(plint|logc|run)_`; no Guids. TC-008 `--target api` build passes cleanly.
