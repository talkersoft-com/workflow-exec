# Task 0006 — Migration, deploy, results, ship

## Steps
1. Apply the EF migrations (`Mode` column, `plint`, `logc`) to the live DB BEFORE deploying
   the API:
   ```bash
   cd /home/todd/workspace/cloud-manager/vm-infra/cloud-manager/cloud-manager-api/src/Models/CloudManager.Entities && \
     dotnet ef database update \
       --connection "Host=localhost;User ID=cloudmanager;Password=P@ssw0rd!;Port=5432;Database=clouddb;SslMode=Prefer;Trust Server Certificate=true"
   ```
2. Deploy API + MCP: `python3 .cicd/deploy-cloud-manager.py --target api`, then
   `--target mcp` (mcp deploy builds dist only — it does NOT restart the running MCP
   server). Web is NOT deployed — this workflow must not change it (deck.md).
3. Deploy porch (vorch workflow deploy fragment):
   ```bash
   cd ~/workspace/cloud-manager/vm-infra/cloud-manager/vorch-service
   ./scripts/deploy-app.sh stage
   ```
   **Restart caution:** before `sudo systemctl restart porch`, check for active work —
   `cloud_playbook_run_list` filtered to Running/Queued and no VM with
   orchestrationStatus=InProgress (`cloud_vm_list`). Wait or cancel deliberately; never
   restart blind. `vorch_service` needs a restart ONLY if shared `internal/` or vorch-lib
   changes altered its binary — they did (vorch-lib message addition), so restart both with
   the same caution. Then `systemctl is-active vorch_service porch` and
   `journalctl -u porch -n 20 --no-pager` clean. A failed deploy is a stop condition —
   write a FIX file under `Retro/`, resolve, retry.
4. Smoke (through the deployed stack): `cloud_health_check`; check-mode run on a TEST VM →
   terminal, no mutation, `playbook_check_completed` event; lint trigger → terminal with
   findings; chunk read for the check run returns ordered chunks; one apply-mode run still
   Succeeds with the familiar tail output; `ansible-studio` health shows the lint probe.
5. Write `Results/RESULT.md`: outcome, branch, PR table (TBD until ship), phase summary,
   deploy targets + smoke attempts + HTTP codes, the ansible-lint provisioning follow-up
   note (Task 0005), the vorch-lib version-tag bump note (Task 0003), and — because mcp
   dist was rebuilt — this exact operator instruction near the top:

   > **Operator action required:** Restart cloud-manager-mcp via `/mcp` in Claude Code to pick up the new dist.

6. Write `Retro/LESSONS.md` (what would have helped; FIX file list — or one line if clean).
7. Ship (see `../deck.md`):
   ```
   hv_ship  deck: "cloud-manager"
            message: "feat: porch check mode, ansible-lint, chunked log streaming (P6 — plint/logc)"
            title:   "Ansible Epic P6: porch --check, lint, log chunk streaming (additive-only)"
            stage:   "exec"
   ```
8. Fill the PR table in `Results/RESULT.md` from the hv_ship output (amend on the same
   branch; push the amendment via hv_ship again, which will push-only). Per deck behaviour
   the fill may land as a follow-up PR — verify actual PR state with `gh` if the ship
   output omits PR lines.

Done when: migrations applied, api + mcp + porch deployed with passing smoke tests,
RESULT.md (with operator /mcp instruction) and LESSONS.md written, hv_ship opened PRs for
every repo ahead of main.
