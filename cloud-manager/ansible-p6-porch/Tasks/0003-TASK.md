# Task 0003 — Porch: lint consumer (vorch-lib + vorch-service repos)

## Steps
1. RESEARCH (read, no changes): the collection-install consumer in porch — queue
   declaration, manual ack, terminal-status check on redelivery, nack behavior — this task
   mirrors it exactly. Also the existing playbook materialization helper.
2. vorch-lib: add NEW `PlaybookLintMessage{ RunId }` struct (JSON tag `runId`) and the
   `playbook-lints` queue name const. NOTHING else in vorch-lib changes — no edits to
   existing structs, JSON tags, queue names, or status strings (deck.md "must NOT change").
   ```bash
   cd ~/workspace/cloud-manager/vm-infra/cloud-manager/vorch-lib
   go build ./... && go test ./...
   ```
   vorch-lib is consumed via the go.mod `replace` directive — rebuild the service to pick it
   up; never edit the replace path. Note in the task output that the version tag should be
   bumped when shipped.
3. Porch: new `playbook-lints` consumer mirroring the collection-install pattern — manual
   ack, on redelivery check the lint run's status via API first and ack-without-work if
   already terminal (crash recovery contract).
4. Handler: fetch the lint run, materialize the playbook with the existing helper into a
   workdir, run `ansible-lint --format json project/`, parse findings (an empty findings
   array is a SUCCESS with zero findings; a non-zero lint exit due to findings is not a
   handler failure), POST findings to `POST api/v1/playbook/{pid}/lint/{runId}/result`,
   which marks the run terminal. `ansible-lint` binary missing → run fails with a clear
   message (health probe in Task 0005 is the guard). Any new log lines route through
   `internal/redact` — Vault material never reaches lint logs or findings.
5. Emit the additive `playbook_lint_completed` event on terminal.
6. Unit tests: message decode, findings JSON parse (fixture with rule id, severity, path,
   line, message), exit-code semantics, redelivery-terminal skip.
7. Build both binaries + full suites of BOTH modules (message struct added → vorch rule):
   vorch-lib `go build ./... && go test ./...`; vorch-service
   `go build ./... && go test ./... && go vet ./...`.
8. e2e: trigger `POST api/v1/playbook/{pid}/lint` on a playbook with a known lint finding;
   poll until terminal; verify `plint` findings match ansible-lint's output for that
   revision and the `playbook_lint_completed` event exists.

Done when: lint run round-trips queue → porch → findings → terminal against the live stack,
existing vorch-lib surface byte-identical, both modules' suites green.
