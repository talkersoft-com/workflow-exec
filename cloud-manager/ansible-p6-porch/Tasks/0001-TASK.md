# Task 0001 — API: run mode + plint/logc entities and endpoints (cloud-manager-api repo)

## Steps
1. RESEARCH (read, no changes): how `PlaybookRun` is persisted and exposed today (run DTOs,
   trigger endpoint, worker-auth PATCH routes), how the collection-install flow publishes to
   its queue from the API side, and how `vm_events` types are declared (contracts §6). Note
   the worker-auth mechanism used by porch PATCHes — the new chunk/lint ingest routes reuse
   it unchanged.
2. Run mode: add nullable/defaulted `Mode` column to `PlaybookRun` (`vm` schema, additive
   migration, default `apply`). `POST api/v1/playbook/{pid}/run` body gains optional
   `"mode": "apply" | "check"`; invalid values → 400. Persist and expose `mode` in run DTOs
   (run get/list). **No message change** — porch fetches the run by id and reads its fields.
3. Lint trigger: `POST api/v1/playbook/{pid}/lint` → 202 + runId. Creates a lint run record
   reusing the five status strings (`queued running succeeded failed cancelled` — no new
   strings), publishes `{"runId"}` to NEW queue **`playbook-lints`** following the existing
   queue-publish pattern (collection-installs is the model). Gate behind the
   `ansible-studio` feature flag per contracts §11.
4. Lint results: `PlaybookLintResult` entity (`plint` public-id prefix, `vm` schema,
   additive migration): per playbook revision, jsonb findings in ansible-lint's JSON shape
   (rule id, severity, path, line, message). Worker-auth
   `POST api/v1/playbook/{pid}/lint/{runId}/result` persists findings and marks the lint
   run terminal. Read path exposes findings for the playbook revision.
5. Log chunks: `PlaybookRunLogChunk` entity (`logc` prefix, `vm` schema, additive
   migration): RunId, TargetId, Seq (monotonic per target), Content (≤32KB, reject larger),
   CreatedAt. Worker-auth ingest `POST api/v1/run/{runId}/log/chunk`; read
   `GET api/v1/run/{runId}/log/chunk?afterSeq=&max=` returning chunks ordered by Seq for
   tail-follow consumers.
6. Events: register additive `vm_events` types `playbook_check_completed` and
   `playbook_lint_completed` (contracts §6).
7. Public-id conventions: `plint_…` / `logc_…` public ids in all DTOs; internal Guids never
   leak (project convention).
8. Unit/integration tests: mode round-trip (default apply, explicit check, invalid → 400),
   lint 202 + queue publish, findings persistence per revision, chunk Seq ordering +
   afterSeq paging + 32KB cap, existing run trigger byte-identical when `mode` omitted.
9. Build: `python3 .cicd/build-cloud-manager.py --target api` must pass. Do NOT apply the
   migration to the live DB in this task — that is Task 0006 step 1.

Done when: migrations created (not applied), all endpoints respond per contract in tests,
run DTOs carry `mode`, lint publish reaches `playbook-lints`, api build clean.
