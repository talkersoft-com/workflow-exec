# Task 0004 — Porch: chunked log streaming (vorch-service repo)

## Steps
1. RESEARCH (read, no changes): the existing line-scanner loop in the run pipeline
   (`playbook_run_exec.go` area) — where output is accumulated, truncated to the last 16KB,
   redacted, and PATCHed per target every 2s.
2. Beside that loop — not replacing it — flush redacted output as sequenced chunks: buffer
   per target; flush when the buffer reaches 32KB or 2s elapses, whichever first; each flush
   POSTs `{ targetId, seq, content }` to `POST api/v1/run/{runId}/log/chunk` with Seq
   monotonic per target starting deterministically (document 0 or 1 and test it). Final
   flush on target terminal so no tail is lost.
3. **The existing 16KB-tail PATCH behavior is kept byte-identical** — same payload fields,
   same cadence, same truncation. Chunks are additive alongside (older consumers unaffected).
4. Redaction: chunk content passes through the same `redact.ScrubString` path as the tail
   PATCH — no new unredacted route to the API. Chunk POST failures must not fail the run:
   log (redacted) and continue; the tail PATCH remains the source of truth for status.
5. Unit tests: flush triggers (32KB boundary, 2s timer, terminal flush), Seq monotonicity
   per target under interleaved multi-target output, redaction applied to chunk content,
   chunk-POST failure tolerance (run still completes).
6. Build both binaries + full suite:
   ```bash
   cd ~/workspace/cloud-manager/vm-infra/cloud-manager/vorch-service/vorch-service
   go build ./... && go test ./... && go vet ./...
   ```
7. e2e: run a playbook producing >16KB of output on a TEST VM; verify
   `GET api/v1/run/{runId}/log/chunk` reassembles the FULL ordered output (beyond the 16KB
   tail), `afterSeq` paging tails correctly mid-run, and the run's tail-PATCH `output` field
   still shows the familiar last-16KB tail.

Done when: full run output is reconstructable from ordered chunks on a real run, tail PATCH
unchanged, tests green.
