# Task 0002 — Porch: check mode (vorch-service repo)

## Steps
1. RESEARCH (read, no changes): `cmd/porch` + `internal/ansible` run pipeline — where the
   run is fetched by id over HTTP, where the `ansible-runner run … --json` command line is
   assembled, and where per-target status PATCHes happen. Confirm (as PLAN.md did) that
   `--check` appears nowhere yet.
2. Read `mode` from the fetched run. When `mode == "check"`, append `--cmdline "--check"`
   to the ansible-runner invocation. Everything else — Vault two-token strategy, inventory,
   redaction, per-target status writes, cancellation — is identical code paths, not copies.
   Unknown/absent mode behaves as `apply`.
3. Emit the additive `playbook_check_completed` event on terminal success of a check-mode
   run (via the existing API event path porch already uses).
4. Unit tests: command-line assembly with mode apply/check/absent (exactly one `--check`
   under check, zero otherwise); status/redaction paths unchanged by mode.
5. Build both binaries + full test suite (shared `internal/` packages — vorch rule):
   ```bash
   cd ~/workspace/cloud-manager/vm-infra/cloud-manager/vorch-service/vorch-service
   go build ./... && go test ./... && go vet ./...
   ```
6. e2e against a real VM (porch must be redeployed for this — coordinate with Task 0006 if
   deferring; otherwise deploy to stage per the deploy fragment, checking for in-flight runs
   before restarting porch): pick/create a TEST VM, run a known-good mutating playbook with
   `mode: "check"`, verify the run reaches a terminal status, the mutation did NOT happen on
   the target, `mode: "check"` is visible in the run DTO, and a `playbook_check_completed`
   event exists. Then an apply-mode run of the same playbook behaves exactly as before.

Done when: check-mode run completes against a real VM with no target mutation, apply path
untouched, both binaries build, tests green.
