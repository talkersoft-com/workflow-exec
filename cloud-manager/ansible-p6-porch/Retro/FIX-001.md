# FIX-001 — Migration accidentally applied to live clouddb during Task 0001

## Failure
Task 0001 step 9 forbids applying the migration to the live DB (that is Task 0006 step 1).
While preparing the scratch-DB test harness I ran
`DBHOST=… DATABASE=clouddb_test … dotnet ef database update` from
`src/Models/CloudManager.Entities`, expecting the runtime env-var connection logic
(`CloudManagerDbContext.RetrieveConnectionString`) to be honored. It was not: design-time EF
uses `CloudManagerDbContextFactory` (`src/Models/CloudManager.Entities/DbContextFactory.cs`),
which reads `appsettings.json` / falls back to a **hardcoded live connection**
(`Server=localhost;Database=clouddb;User Id=cloudmanager;…`). The new migration
`20260612062109_AddPorchExecutionUpgrades` was therefore applied to live `clouddb`, and the
scratch `clouddb_test` stayed empty.

## Detection
Post-migration verification queried `clouddb_test` → zero tables; querying live `clouddb`
showed `vm.playbook_runs.mode` present.

## Fix
1. Roll the live DB back one migration:
   `dotnet ef database update 20260612044815_AddYamlDocuments` (same factory → live clouddb,
   which is exactly the DB that must be reverted). The rolled-back objects were seconds old and
   empty (new column + two new empty tables), so the rollback is loss-free.
2. Verify `mode` column and `playbook_lint_results` / `playbook_run_log_chunks` are gone from
   live `clouddb`.
3. For all scratch-DB operations use an explicit connection override:
   `dotnet ef database update --connection "Host=localhost;Database=clouddb_test;…"` —
   never rely on env vars for design-time EF commands in this repo.

## Lesson
In cloud-manager-api, `dotnet ef` ALWAYS targets live clouddb unless `--connection` is passed
explicitly. Task 0006's intentional live migration can use the default; every test apply must
pass `--connection`.
