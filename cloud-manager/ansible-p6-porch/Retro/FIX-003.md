# FIX-003 — porch deploy fragment broken: build-app.sh targets the module root (no Go files)

## Failure
Task 0006 step 3: `./scripts/deploy-app.sh stage` failed with
`no Go files in …/vorch-service/vorch-service`. `scripts/build-app.sh` builds
`packages=( "${gitroot}/vorch-service" )` — the module ROOT — but since the porch split the
binaries live under `cmd/vorch-service` and `cmd/porch`; the root has no `.go` files, so
`go build` (and therefore every deploy via this script) fails. The script also `rm -rf`s the
artifact dir inside `build_package`, which would wipe the first binary when a second package
is built.

## Fix
Patched `scripts/build-app.sh` in the vorch-service repo (ships with this PR):
- `packages=( cmd/vorch-service cmd/porch )` under the module dir — both binaries built.
- Moved `rm -rf/mkdir` of the artifact dir out of `build_package` so multiple packages
  coexist in one artifact dir.
Then re-ran `./scripts/deploy-app.sh stage` → both binaries + stage config copied to
`/kvm-automator`.

## Lesson
The deploy fragment had bit-rotted against the cmd/ repo layout introduced when porch was
split out — nothing in CI builds via `build-app.sh`, so the breakage was invisible until a
real deploy. Exercising the full deploy script (not just `go build ./...`) is part of the
porch task definition for good reason.
