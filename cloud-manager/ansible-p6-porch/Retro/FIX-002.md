# FIX-002 — `--cmdline --check` as two argv tokens breaks ansible-runner argparse

## Failure
Task 0002 e2e (Test 0002 TC-002): the first check-mode run against pg-test failed in ~1s.
Target output showed ansible-runner usage text ending in:
`ansible-runner run: error: argument --cmdline: expected one argument`.

`buildRunnerArgs` appended `"--cmdline", "--check"` as two argv elements. Python argparse
(which ansible-runner uses) refuses an option value that itself starts with `--` when passed
as a separate token — it parses `--check` as a new flag and reports `--cmdline` got no value.

## Fix
Pass the pair as a single `=`-joined token: `--cmdline=--check`. Updated `buildRunnerArgs`
and the unit tests (`runner_args_test.go`) to assert the single-token form, then re-ran the
e2e.

## Lesson
When forwarding option values that start with `--` to argparse-based CLIs, always use the
`--opt=value` single-token form. The plan text `--cmdline "--check"` describes shell quoting,
not argv structure.
