# Task 0003 — Porch wires consumers + healthprobe

## Goal
Make `cmd/porch/main.go` a real entry point: open AMQP, register the four ansible consumers, start the healthprobe goroutine. Verify the healthprobe target endpoint exists API-side; if not, fix it in the same plan.

## Target repo(s)
`vorch-service` (primary); `cloud-manager-api` (conditional, only if healthprobe endpoint is missing)

## Steps
1. In `cmd/porch/main.go`:
   - Read config (env vars and/or `config-server.yaml`) using the same loader vorch uses.
   - Open AMQP connection via `internal/amqp`.
   - Register consumers via `internal/ansiblesub`:
     - `playbook-runs` → `internal/ansible.NewPlaybookRunHandler()`
     - `playbook-run-cancel` → `internal/ansible.NewPlaybookCancelHandler(handler.Registry)`
     - `collection-installs` → `internal/ansible.NewCollectionInstallHandler()`
     - `collection-removes` → `internal/ansible.NewCollectionRemoveHandler()`
   - Start healthprobe goroutine.
   - Block on a select / signal handler the same way vorch does.
2. Match vorch's logging style: `log.Printf("Started Service at %s", time.Now())` at boot. (Actual log lines stay generic — they'll be cleaned up in Phase 0005's redaction work.)
3. **VERIFY** the healthprobe endpoint match. Open `internal/ansible/.../probe.go` and find the PATCH URL + body. Then in `cloud-manager-api`:
   ```bash
   grep -rnE "admin/feature-flags|FeatureFlags.*PATCH|HttpPatch" src/CloudManager.API/Controllers/
   ```
   Pass: there is a route accepting `PATCH` for the feature-flag health write with the body the probe sends. Record the verification result in `Retro/CHANGE-0003.md`.
4. If the verification FAILS (no matching route, or shape mismatch), add a small change to `FeatureFlagsController`:
   - Route: `PATCH /api/v1/FeatureFlags/{key}/health` (or whichever route the probe is already calling — match the probe, not vorch's preferred shape, to minimize cross-repo edits).
   - Body: `{ healthy: bool, lastProbedAt: DateTime }`.
   - Updates `feature_flags.healthy` and `last_probed_at`. No auth changes beyond the existing controller's.
   - This is a separate sub-change in the same `cloud-manager-api` PR.
5. Run `go build ./cmd/porch ./cmd/vorch-service` — both succeed.
6. If API was edited: `dotnet build src/CloudManager.API/CloudManager.API.csproj -c Release` exits 0.

## Definition of done
`cmd/porch/main.go` wires four AMQP consumers + healthprobe goroutine. Healthprobe endpoint verification recorded. If endpoint missing, API PATCH route added. Both Go binaries build cleanly. (Not yet deployed — Phase 0006 owns deploy scripts.)
