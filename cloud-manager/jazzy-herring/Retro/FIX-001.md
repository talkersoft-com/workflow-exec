# Fix 001 — ansible-runner not on PATH for porch + ProtectHome blocked /home venvs

## Failed test
Phase 7 (cutover), TC-005-adjacent. After porch started, healthprobe logged: `healthprobe: ansible-runner --version failed: exec: "ansible-runner": executable file not found in $PATH`.

## Root cause
Two layered problems:

1. **PATH didn't include the install location.** ansible-runner was installed by `pipx install` into `/home/cloudmanager/.local/bin/ansible-runner`. The porch systemd unit (`User=root`, `HOME=/tmp`) has the default minimal PATH, which doesn't include any user-local pipx bin.

2. **ProtectHome=yes blocked /home AND /root.** Even after installing ansible-runner as root via `sudo pipx install ansible-runner` (which lands at `/root/.local/share/pipx/venvs/ansible-runner/bin/ansible-runner`), the porch unit could not see it — `ProtectHome=yes` makes both `/home/` and `/root/` inaccessible to the process.

## Fix applied
- `sudo pipx install ansible-runner` to land a system-wide-ish copy under `/root/.local/...`.
- Symlink at `/usr/local/bin/ansible-runner` pointing to a thin wrapper script (`/usr/local/bin/ansible-runner.real` — kept as a real file because the pipx-managed shebang chain dereferences through /root).
- Added `Environment="ANSIBLE_RUNNER_BIN=/usr/local/bin/ansible-runner"` to `porch.service`.
- Changed `ProtectHome=yes` → `ProtectHome=no` on `porch.service` (same trust profile as the retired Python `cloud-manager-worker`, which ran with full /home access).
- `daemon-reload` + restart porch.

## Outcome
Pass. Healthprobe writes `feature_flags.playbooks.healthy=true` + `last_probed_at=2026-05-28 21:42:51-04` to the DB on first tick. `systemctl is-active porch` → `active`.

## Follow-up
A cleaner long-term fix is to ship ansible-runner via the OS package manager (apt) or vendor a self-contained tarball under `/opt/`, so neither pipx nor /home access is required. Tracked as `[[follow-up-ansible-runner-system-install]]`.
