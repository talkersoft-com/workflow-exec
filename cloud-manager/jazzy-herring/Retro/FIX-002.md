# Fix 002 — ansible-runner needs explicit `-p` + `--project-dir` (and inventory generation is a pre-existing gap)

## Failed test
Phase 9, TC-001 / TC-004. First pg-on-postgres-test trigger ended with run.status=4 (Failed). Log file showed:
```
The -p option must be specified when not using -m or -r
```

## Root cause #1 (fixed): bad ansible-runner invocation
`internal/ansible/playbook_run_exec.go:260` was calling:
```go
cmd := exec.CommandContext(ctx, bin, "run", workdir, "--json")
```
ansible-runner's `run` subcommand requires one of `-p PLAYBOOK | -m MODULE | -r ROLE`. The materialize endpoint writes `site.yml` at the workdir root (`MaterializedFile { RelativePath = "site.yml" }`), not under the default `project/` subdir.

## Fix #1 applied
```go
cmd := exec.CommandContext(ctx, bin, "run", workdir,
    "--project-dir", workdir, "-p", "site.yml", "--json")
```
After this fix, ansible-runner finds and runs site.yml. Run status flips from Failed (4) to Succeeded (3).

## Root cause #2 (PRE-EXISTING, NOT fixed in this plan): no inventory
After fix #1, the run reaches status=3 but the playbook log shows:
```
[WARNING]: No inventory was parsed, only implicit localhost is available
[WARNING]: provided hosts list is empty, only localhost is available.
"event": "playbook_on_no_hosts_matched"
```
The pg playbook is written for `hosts: all` with `inventory_hostname == postgres-test`. Without an inventory file at `<workdir>/inventory/hosts`, ansible-runner has nothing to target. The playbook reports success because `playbook_on_no_hosts_matched` is not a failure event — but no postgres install happens and no secret is written to Vault.

This gap is **inherited from vorch's pre-existing playbook execution path**, not introduced by this carve-out. The Python `cloud-manager-worker` (now retired) generated the inventory at run time, drawing the VM's IP from libvirt and writing `inventory/hosts` with `<vm-name> ansible_host=<ip> ansible_user=cloudmanager ansible_ssh_private_key_file=env/ssh_key`. That logic was never ported into vorch's `playbook_run_exec.go` — vorch never actually ran a playbook end-to-end in production, because the Python worker won the round-robin half the time and silently wrote the inventory itself.

## What now
Out-of-scope for charmed-panda. The plan was "carve ansible out of vorch into porch + retire Python worker"; it did NOT scope "fix the inventory gap". Documented for a follow-up plan: `[[follow-up-porch-inventory-generation]]`.

What still needs work (filed as the follow-up):
- Generate `inventory/hosts` with `<target.VMName> ansible_host=<ip> ansible_user=<user> ansible_ssh_private_key_file=env/ssh_key` for each target before exec.
- Source the VM IP — likely a new GET `/api/v1/vm/{id}` field, or a libvirt-side query through vorch (which is the only thing that knows VM IPs reliably in this stack).
