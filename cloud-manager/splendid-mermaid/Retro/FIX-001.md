# FIX-001 — Source-VM validation must tolerate a soft-deleted source VM

## Phase / Task
Task 0004 (E2E), exercising the Task 0002 (P1) resolver against the deployed system.

## Failure observed
E2E happy path — provision the templated `{vm_public_id}` blueprint `bp_HVTW8NWCZZ`
(`splendid-mermaid-consumer`) supplying the Postgres VM `pg-test123` (`vm_W28X31Y8SX`) as the
binding's `vm_public_id` source — returned **422** instead of injecting:

```
POST /api/v1/marketplace/bp_HVTW8NWCZZ/instantiate
{"hostId":"host_KW7YKFF3YM","name":"sm-consumer-probe",
 "secretBindingParams":[{"secretBindingId":"sb_CTHR021ZR5","params":{"vm_public_id":"vm_W28X31Y8SX"}}]}
→ HTTP 422 {"message":"Secret binding 'VM postgres credential' could not be resolved",
            "secretBinding":"VM postgres credential","vaultPath":null}
```
(No VM row was created — the P0 resolve-before-create guarantee held; this is purely a
resolution-rejection bug, not an orphan.)

## Root cause
`SecretBindingResolver.ComputeVaultPathAsync` validates the supplied source VM with
`_context.VirtualMachines.GuidByPublicIdAsync(sourceVmPublicId)`. `VirtualMachine` carries a global
soft-delete query filter (`CloudManagerDbContext` line ~1021: `HasQueryFilter(e => e.DeletedAt == null)`),
so the lookup excludes soft-deleted VMs. `pg-test123` (`vm_W28X31Y8SX`) is soft-deleted in the deployed
DB (its row exists with `deleted_at` set), so the lookup returned null and the resolver rejected a
perfectly valid source.

A secret legitimately **outlives** its source VM: the value lives in Vault at
`cloudmanager/data/vm/instances/vm_W28X31Y8SX/postgres` independent of the VM's lifecycle. The
already-shipping concrete-path binding `sb_08HX9DZ2E7` hardcodes that exact path and resolved fine
for `db-consumer-1` even though the VM is soft-deleted. The templated form supplying the same id must
behave identically. Requiring the source VM to be *live* is too strict and contradicts
"soft-delete authoritative" (which governs the binding/secret, not a path component).

## Fix
Validate the supplied source VM with `IgnoreQueryFilters()` so a real-but-soft-deleted source VM is
accepted; only a never-existed public id is rejected (clean 422). The path substitution continues to
use the public_id string (paths are keyed by public id), matching the concrete-path behavior.

```csharp
var sourceVmGuid = await _context.VirtualMachines
    .IgnoreQueryFilters()
    .Where(v => v.PublicId == sourceVmPublicId)
    .Select(v => (Guid?)v.Id)
    .FirstOrDefaultAsync();
```

## Verification after fix
Rebuild + redeploy api, then re-run the happy path → expect 200 + create-vm command + injection;
re-run the clean-422 (no param) and regression cases. Recorded in RESULT.md.
