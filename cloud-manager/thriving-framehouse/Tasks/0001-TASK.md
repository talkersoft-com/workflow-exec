# Task 0001 — API: persist + return the resolved Vault path

**Repo:** `cloud-manager-api` (.NET)
**Plan:** Design §#3; phase 0001.

## Goal
Store the **actual** resolved Vault path on the `vm.vm_secret_bindings` usage row at provision
time, and return that stored value from the read endpoint — so cross-VM templated bindings report
the source VM's true path instead of a value recomputed against the listing VM's id.

## Steps

### 1. Migration (one, reversible)
- Add a nullable `resolved_vault_path` (text) column to `vm.vm_secret_bindings`.
- Use EF Core: add the property to the `VmSecretBinding` entity + its configuration, then
  `dotnet ef migrations add AddResolvedVaultPathToVmSecretBindings`
  (from `src/Models/CloudManager.Entities`).
- Confirm the generated `Up` adds the column and `Down` drops it (reversible).
- Do **not** apply to the live DB here — application happens in Task 0003 before deploy.

### 2. Populate at provision
- `SecretBindingResolver` already computes the concrete vault path per binding. Thread that
  resolved path through to the site where the `VmSecretBinding` usage row is inserted, and store
  it in `resolved_vault_path`.
- Keep the existing `{virtual_machine_id, secret_binding_id, cred_name}` writes unchanged; this
  is an added field, not a rewrite.

### 3. Read endpoint
- `GET /api/v1/vm/{publicId}/secret-binding` returns the **stored** `resolvedVaultPath`.
- Fall back to the current recompute **only when the stored column is null** (rows written before
  this change), so old rows still return a best-effort value.
- camelCase JSON; the value is a Vault **path**, never a secret value.

### 4. Build
- `python3 .cicd/build-cloud-manager.py --target api` — must pass clean.

## Definition of done
Migration is reversible and additive; provision populates `resolved_vault_path` from the resolver;
read endpoint returns the stored value and recomputes only on null; api builds clean.
`Test/0001-TEST.md` passes.
