## TC-001 — Migration applied to live DB
After `dotnet ef database update` against clouddb, inspect `vm.vm_secret_bindings`
(db-toolkit `db_get_table_schema` or psql).
Pass: `resolved_vault_path` (text, nullable) column exists.
Fail: column missing or wrong type.

## TC-002 — Both targets build clean
Run `python3 .cicd/build-cloud-manager.py --target all`.
Pass: api + mcp build with no errors.
Fail: any build error.

## TC-003 — API deploy smoke-tests green
Run `python3 .cicd/deploy-cloud-manager.py --target api`.
Pass: service restarts and the `localhost:5250/api/v1/Host` poll returns HTTP 200 within the window.
Fail: deploy aborts or smoke poll never returns 200 → stop condition, write Retro/FIX.

## TC-004 — E2E Pass A: provision through MCP without guard error
With the MCP reloaded (`/mcp`), call `cloud_marketplace_provision` for a consumer VM from a
templated `{vm_public_id}` binding, passing
`secretBindingParams:[{secretBindingId, params:{vm_public_id: <source vm public id>}}]`.
Pass: the call succeeds — no camelCase guard error; provisioning is enqueued.
Fail: the snake_case guard error returns, or the call is rejected for the param keys.

## TC-005 — E2E Pass B: read endpoint returns the true source path
After the consumer provisions, call `cloud_vm_secret_binding_list` (and/or
`GET /vm/{publicId}/secret-binding`) for the consumer VM.
Pass: `resolvedVaultPath` is the **source** VM's path (`…/<source_vm_public_id>/postgres`),
read from the stored column — NOT the consumer's id; no secret value in output.
Fail: path contains the consumer VM's id, or recomputes instead of using the stored value.
