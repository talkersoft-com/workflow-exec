# Task 0001 — Migration + entity + DTO for VM ip_address

## Goal
Add `ip_address VARCHAR(45) NULL` to `vm.virtual_machines`. Wire it through the EF entity, the DTO, and the AutoMapper profile. Existing rows survive with NULL.

## Target repo(s)
`cloud-manager-api`

## Steps
1. Add `public string? IpAddress { get; set; }` to `src/Models/CloudManager.Entities/Models/VirtualMachine.cs`.
2. Map the column in `CloudManagerDbContext.cs` inside the `VirtualMachine` entity config block:
   ```csharp
   entity.Property(e => e.IpAddress).HasColumnName("ip_address").HasMaxLength(45).IsRequired(false);
   ```
3. Add the DTO field on `src/Models/CloudManager.DTO/Models/VirtualMachine.cs` (or whatever the VM DTO is named in this codebase):
   ```csharp
   public string? IpAddress { get; set; }
   ```
4. Update the AutoMapper profile under `src/Services/CloudManager.Data.Services/MappingProfiles/` to map `IpAddress` both directions.
5. Generate the EF migration:
   ```
   cd src/Models/CloudManager.Entities
   dotnet ef migrations add AddVmIpAddress --output-dir Migrations --context CloudManagerDbContext
   ```
6. Review the generated `<timestamp>_AddVmIpAddress.cs` — should contain `migrationBuilder.AddColumn<string>(...)` for `ip_address` with `nullable: true` and `maxLength: 45`. No data changes in the migration itself; backfill happens in Phase 5.
7. `dotnet build src/CloudManager.API/CloudManager.API.csproj -c Release` — exit 0.

## Definition of done
Migration file exists. `dotnet build` clean. New IpAddress property on entity + DTO. AutoMapper handles both directions. Migration NOT applied to DB yet — Phase 10 does that.
