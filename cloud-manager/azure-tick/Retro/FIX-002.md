# Fix 002 — Stale playbook revision + pg playbook param drift

## Failed test
Phase 12 TC-005 + TC-007. After FIX-001 (SSH key path + inventory), the run still showed `ok=1` only (Gathering Facts). After re-running with `keep workdir`, found `/kvm-automator/playbook-runs/.../project/site.yml` was only 4 bytes — the **probe playbook**, not pg.

## Root cause #1 — Stale playbook revision (infrastructure)
`pb_EMEJCH08GG` (pg) has 2969 bytes of content in `vm.playbooks.content` but only ONE revision in `vm.playbook_revisions`, `pbrev_KYW36QHX8Y`, with 61 bytes of probe content. The pg content was edited directly without creating a new revision.

When `PlaybookRunService.TriggerMultiVmAsync` records the run, it sets `run.PlaybookRevisionId` to the latest revision. The materialize endpoint then resolves the **revision** content, not the current `playbooks.content`. Result: porch always materializes the stale probe content for this playbook.

This is a pre-existing API-side gap: auto-revisioning on edit isn't wired, so updating `playbooks.content` without creating a matching `playbook_revisions` row leaves the run path stuck on the old snapshot. Filed as `[[follow-up-auto-revision-on-edit]]`.

**Workaround applied for verification**: manually inserted a new revision (`pbrev_AZTICK001`) with the current 2969-byte pg content. After that, porch materialized the real pg playbook and ran the actual tasks.

## Root cause #2 — pg playbook uses an outdated postgresql_user param
After the revision fix the run progressed all the way to:
```
- name: Install PostgreSQL packages          OK
- name: Ensure PostgreSQL is running         OK
- name: Create application database          OK (app DB exists)
- name: Create application user...           FAILED:
  "Unsupported parameters for (community.postgresql.postgresql_user)
   module: priv. Supported parameters include: ca_cert, comment, ..."
```

The pg playbook uses `priv: "ALL"`. The installed `community.postgresql` 4.2.0 renamed/removed that param. Newer versions use `privs:` (plural) or a separate grant statement.

This is **playbook authoring**, not infrastructure. The playbook needs the maintainer to update for the deployed collection version. Filed as `[[follow-up-pg-playbook-postgres-user-priv]]`.

## Mid-phase incremental fixes applied (all related)
- **Target VM was missing `acl` package** — failed at "Create application database" with "chmod: invalid mode: 'A+user:postgres:rx:allow'". `ansible` needs `acl` to honor `become_user: <unprivileged>`. Installed via `ssh cloudmanager@<vm> 'sudo apt install -y acl'`. Not a porch bug; the cloud-init image should include it. Filed as `[[follow-up-cloud-init-include-acl]]`.

## Fix applied (infrastructure scope only)
- `cloud-manager-api/scripts/porch/install-porch-service.py` — added `_vault_token()` reader so the install script auto-populates `Environment="VAULT_TOKEN=..."` on the porch unit from `/etc/cloud-manager/api.env`. Without VAULT_TOKEN set, porch's `vault.NewClient()` returns nil and `env/ssh_key` never gets written.
- `vorch-service/internal/vault/client.go` — `vaultapi.DefaultConfig()` strict-TLS rejected the talkersoft.com cert against 127.0.0.1. Added `cfg.ConfigureTLS(&vaultapi.TLSConfig{Insecure: true})` gated by `VAULT_SKIP_VERIFY` (default skips, matches charmed-panda FIX-001 on the API side).
- `vorch-service/internal/ansible/playbook_run_exec.go` — Vault SSH read path was `vm/instances/<id>/ssh_key` (a field that doesn't exist). Changed to `vm/instances/<id>` and JSON-unmarshal the inner `ssh_key` blob to extract `private_key`. Inventory line also changed from relative `env/ssh_key` to absolute `<workdir>/env/ssh_key`.

## Outcome
Infrastructure side fully working. The pg playbook itself fails on `priv:` which is a playbook-side fix outside this plan's scope. Acceptance is conditional pass — every infrastructure criterion met; the "Vault KV populated" + "appuser exists" criteria require a playbook fix.
