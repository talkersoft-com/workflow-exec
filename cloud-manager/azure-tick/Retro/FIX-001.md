# Fix 001 — Vault SSH-key path + relative ssh_key in inventory

## Failed test
Phase 12 TC-005 — first pg-on-postgres-test trigger ended Failed. ansible-runner log showed:
```
"msg": "Failed to connect to the host via ssh: ...
no such identity: env/ssh_key: No such file or directory
cloudmanager@10.0.150.55: Permission denied (publickey)."
```

## Root causes (two layered, fixed together)

### #1: Vault SSH-key path was wrong (pre-existing in vorch's code, carried into porch)
`internal/ansible/playbook_run_exec.go:165` was reading `cloudmanager/vm/instances/<publicId>/ssh_key`, but vorch-lib's `StoreSSHKey` writes the JSON blob at `cloudmanager/vm/instances/<publicId>` with the field name `ssh_key`. The inner JSON has `private_key`, `public_key`, `created_at`. The old read silently returned nil, so `env/ssh_key` never got written. The result: ansible-runner had no key to present.

Verified via direct Vault read (with the cloudmanager token):
```
GET /v1/cloudmanager/data/vm/instances/vm_189KV047V2
data.ssh_key = "{\"private_key\":\"-----BEGIN OPENSSH...\", \"public_key\":\"ssh-ed25519 ...\", \"created_at\":\"2026-05-24...\"}"
```

### #2: Inventory used a relative ssh_key path
Inventory line had `ansible_ssh_private_key_file=env/ssh_key`. ansible-runner cd's into `<workdir>/project/` before executing the playbook, so `env/ssh_key` resolves to `<workdir>/project/env/ssh_key` — wrong directory. The real file is at `<workdir>/env/ssh_key`.

## Fix applied
1. `playbook_run_exec.go` Vault read: `vm/instances/%s/ssh_key` → `vm/instances/%s`. Then extract the `ssh_key` field, JSON-unmarshal into `{PrivateKey string}`, and use `.PrivateKey`. Old `value` and `private_key` field fallbacks kept for any other secret shape that might predate this convention.
2. `playbook_run_exec.go` inventory: `ansible_ssh_private_key_file=env/ssh_key` → `ansible_ssh_private_key_file=<workdir>/env/ssh_key` (absolute).
