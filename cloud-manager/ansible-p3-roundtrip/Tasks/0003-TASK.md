# Task 0003 — Structured-edit write path (cloud-manager-api)

## Steps
1. RESEARCH (read, no changes): P2's `play`/`ptask` record shapes and public ids; the exact
   existing revision-cutting path the new endpoints must reuse (so `pbrev`/`rfrev` history,
   P2 re-decomposition, and events all fire exactly as a raw-text save would).
2. Add the PATCH endpoints P2 deferred:
   - `PATCH api/v1/playbook/{pid}/play/{playId}` — structured field changes on a play
   - `PATCH api/v1/playbook/{pid}/task/{ptaskId}` — structured field changes on a task
3. Flow per request: validate fields → build a recompose edit list (set/insert/delete at
   record paths, anchored on the `ydoc` span map) → sidecar `/recompose` splices → the new
   YAML text is saved through the EXISTING revision-cutting path. No parallel save path.
4. Span drift: if the revision changed since the edit list was built, surface the sidecar's
   409 to the client (refetch semantics) — do not rebase server-side.
5. Conventions: camelCase, public_ids only, flag-gated `ansible-studio`, auth-scoped.
6. Build: `python3 .cicd/build-cloud-manager.py --target api`

Done when: build passes; a structured edit to a play/task produces a new revision whose YAML
differs ONLY in the intended span (comments, key order, styles intact); history, events, and
re-decomposition all fired through the normal path.
