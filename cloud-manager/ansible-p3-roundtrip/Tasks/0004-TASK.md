# Task 0004 — P2 decomposer integration (cloud-manager-api)

## Steps
1. RESEARCH (read, no changes): how P2's YamlDotNet decomposer populates `ptask.SourceSpan`
   today; where the decomposer runs relative to the new `ydoc` write on revision cut.
2. When a `ydoc` span map exists for the revision, the P2 decomposer prefers it over its own
   YamlDotNet line/col marks — sidecar spans flow into `ptask.SourceSpan` (and play/handler
   spans where recorded). Fall back to YamlDotNet marks when no `ydoc` is present (old
   revisions, sidecar unavailable at decompose-time).
3. P2's stance is unchanged: YAML text stays the source of truth; records remain a derived
   read model; the decomposer still re-derives records from the new text after every save —
   including saves produced by the structured-edit path.
4. Do not re-parse in consumers: the `ydoc` map is the span source P8 decorations will anchor
   to — keep it authoritative once written.
5. Build: `python3 .cicd/build-cloud-manager.py --target api`

Done when: build passes; a fresh revision's `ptask.SourceSpan` values come from the sidecar
map (verifiably distinct where ruamel and YamlDotNet disagree, e.g. comment-attached nodes);
revisions without a `ydoc` still decompose via the YamlDotNet fallback.
