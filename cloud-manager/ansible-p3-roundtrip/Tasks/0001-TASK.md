# Task 0001 — Round-trip sidecar service (cloud-manager-api repo)

## Steps
1. RESEARCH (read, no changes): how the API repo lays out auxiliary services and Dockerfiles;
   the `PlaybookRevision`/`RoleFileRevision` content shapes the sidecar will receive.
2. Create `roundtrip-service/` in the cloud-manager-api repo — Python 3.12, FastAPI, ruamel.yaml,
   uv + ruff tooling (PLAN.md open question 3 default; zero impact outside the folder):
   - `POST /decompose` — YAML text → structural map with per-node source spans + style metadata
   - `POST /recompose` — original text + list of structured edits (set/insert/delete at record
     paths) → new YAML text, formatting preserved
   - `GET /healthz`
3. Own Dockerfile; service joins the SAME docker-compose stack as the API (PLAN.md open
   question 1 default: one `docker compose up` story). Not network-exposed beyond the API.
4. Recompose conflict handling: if the supplied base text/checksum no longer matches (span
   drift), return 409 — client refetches (PLAN.md open question 2 default).
5. Golden-file fidelity corpus in the sidecar's tests: real-world playbooks covering
   anchors/aliases, block scalars `|`/`>`, deep comments, flow style, weird indentation.
   Assert `recompose(decompose(x)) == x` byte-identical, and each structured edit changes only
   the intended span. CI-runnable with plain pytest.
6. No porch/vorch changes, no new queues or messages — hard constraint from the sidecar
   decision in PLAN.md.

Done when: sidecar container builds and starts; all three endpoints respond; pytest corpus
passes byte-identical round-trip on every golden file.
