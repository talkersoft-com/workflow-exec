# Task 0003 — E2E + build vorch + manual redeploy + ship

## Steps
1. **Build vorch:**
   - `cd vm-infra/cloud-manager/vorch-lib && go build ./... && go test ./provision/...` green.
   - `cd vm-infra/cloud-manager/vorch-service && ./scripts/build-app.sh` green.
   (The `.cicd/build-cloud-manager.py` script targets api/mcp/web only — it does NOT build vorch.)
2. **Manual vorch redeploy on the hypervisor** so the running provisioner emits the new user-data:
   `./scripts/deploy-app.sh` — but note `deploy-app.sh` cannot ssh-to-self, so install/run the
   freshly built binary **locally on the hypervisor**. Record the exact commands in
   `Results/RESULT.md`.
3. **E2E against a real provision (MCP + SSH + db/psql):**
   - **Plain VM:** `cloud_vm_create` + `cloud_orchestration_provision` a vanilla VM → after boot,
     SSH (22) reachable from the operator Mac; **all other inbound ports blocked** (e.g. nmap/`nc`
     to any other port times out). `ufw status` on the VM shows `22/tcp ALLOW Anywhere` + default
     deny incoming.
   - **Postgres VM:** provision a Postgres VM and apply the pg playbook ufw rule →
     - SSH (22) reachable;
     - `5432` reachable **from an on-subnet peer** (`10.0.150.0/24`);
     - `5432` **blocked from the off-subnet Mac** (the original exposure is now closed at the host
       firewall, not just `pg_hba`).
   - **Lockout sanity:** the provisioning SSH session and the post-boot porch/ansible runs were
     never dropped (provision completed; VM reached a healthy state).
4. **Apply the playbook ufw rules** to the running service VM(s) (`cloud_vm_playbook_apply` /
   `cloud_playbook_run`) and confirm the rules land (`ufw status`).
5. **Write outputs BEFORE shipping:**
   - `Results/RESULT.md` — outcome, branch, PR table (filled from hv_integrate), per-phase summary,
     exact vorch build+redeploy commands, the plain/Postgres E2E port-reachability evidence (from
     Mac and on-subnet peer), and the confirmed RabbitMQ port set.
   - `Retro/LESSONS.md` — what would have helped + FIX file list.
6. **Ship:** `hv_integrate` per `../deck.md`. A failed build/redeploy/E2E is a stop condition —
   write a FIX file under `Retro/`, resolve, retry.

## Definition of done
vorch-lib + vorch-service build green; vorch manually redeployed on the hypervisor; plain VM is
SSH-only (all other ports blocked); Postgres VM allows 5432 from the subnet and blocks it
off-subnet; no lockout occurred; playbook ufw rules applied and confirmed; Results + LESSONS written
BEFORE integrate; hv_integrate opened PRs and reported their links.
