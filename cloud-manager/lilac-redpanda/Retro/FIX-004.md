# Fix 004 — always --force and fix double-segmented install_path

## Failed test
Task 0009, install reported `install_status=Installed` but install_path was `/usr/lib/python3/dist-packages/ansible_collections/ansible_collections/community/postgresql` — doubled segment, and pointing at the apt-bundled location instead of `/usr/share/ansible/collections`.

## Root cause
Two issues:
1. `apt install ansible` (from FIX-002) bundles many community collections under `/usr/lib/python3/dist-packages/ansible_collections/`. `ansible-galaxy collection install` without `--force` sees those as already-satisfied and reports the apt path without writing to `-p`.
2. The verify-step parser concatenated `<root>/ansible_collections/<ns>/<name>` but ansible-galaxy's JSON output already includes the `ansible_collections` segment in the root key.

## Fix applied
- `collection_install_handler.go`: always pass `--force` so ansible-galaxy installs to the configured `-p` path regardless of any pre-existing apt-bundled copy.
- `collection_install_handler.go`: fixed verify to use `<root>/<ns>/<name>` (single segment) and to prefer the `/usr/share/ansible/collections/ansible_collections` root over the apt path so the recorded `install_path` always points where our worker writes — which is also what the remove safety guard requires.

## Outcome
Pass — fresh installs land at `/usr/share/ansible/collections/ansible_collections/community/<name>` and the remove flow's safety guard accepts the path.
