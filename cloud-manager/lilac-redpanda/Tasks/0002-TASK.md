# Task 0002 — API: DTOs + AutoMapper + service interfaces + DI

## Goal
DTOs that mirror the entities (with public_ids on the wire, not Guids). AutoMapper profile that maps entity ↔ DTO including FK Guid → public_id projection (same pattern as `PlaybookProfile`). Service interfaces + implementations for catalog, host-collections, and install-runs operations.

## Target repo(s)
`cloud-manager-api`

## Steps

1. **DTOs** under `src/Models/CloudManager.DTO/Models/`:
   - `AnsibleCollection.cs` — `Id` (string), `Namespace`, `Name`, `Description`, `SourceUrl`, `LatestKnownVersion`, audit fields.
   - `HostAnsibleCollection.cs` — `Id`, `HostId` (string public_id), `CollectionId` (string public_id), `Version`, `InstallPath`, `InstallStatus` (string), `LastInstallRunId` (nullable string), `ErrorMessage`, audit.
   - `AnsibleCollectionInstallRun.cs` — `Id`, `HostId`, `CollectionId`, `RequestedVersion`, `Status` (string), `Output`, `ErrorMessage`, `StartedAt`, `CompletedAt`, `Force`, `IsRemoval`, audit.
     - For the worker callback include `CollectionNamespace`, `CollectionName`, and `InstallPath` (sourced via joins) so vorch can fetch one DTO and have everything it needs.
   - `PlaybookCollectionRequirement.cs` — `Id`, `PlaybookId`, `CollectionId`, `MinVersion`, `MaxVersion`, audit.

2. **AutoMapper profile** at `src/Services/CloudManager.Data.Services/MappingProfiles/AnsibleCollectionProfile.cs`:
   - `CreateMap<AnsibleCollection, DtoModel.AnsibleCollection>().ForMember(d => d.Id, o => o.MapFrom(s => s.PublicId));`
   - `HostAnsibleCollection` → DTO: map `Id`, `HostId`, `CollectionId`, `LastInstallRunId` via FK-to-public_id lookups (services do the lookup at query time; mapper itself can ignore those Guid-typed properties and let the service set them, matching the existing `PlaybookRunService.GetByIdAsync` pattern).
   - `AnsibleCollectionInstallRun` → DTO: same pattern. Map `Status` enum to `ToString()`.
   - `PlaybookCollectionRequirement` → DTO: same.

3. **Service interfaces** under `src/Services/CloudManager.Data.Services/Interface/`:
   - `IAnsibleCollectionService` — `ListAsync()`, `GetByIdAsync(string)`, `CreateAsync(CreateRequest)`, `DeleteAsync(string)`.
   - `IHostCollectionService` — `ListByHostAsync(string hostPublicId)`, `RequestInstallAsync(string hostPublicId, string collectionPublicId, string? requestedVersion, bool force)`, `RequestReinstallAsync(string hacollPublicId)`, `RequestRemoveAsync(string hacollPublicId)`.
   - `ICollectionInstallRunService` — `GetByIdAsync(string)`, `PatchFromWorkerAsync(string, PatchRequest)`.

4. **Service implementations** mirror the existing playbook services:
   - DB writes for `RequestInstallAsync` / `RequestRemoveAsync` create both the `host_collections` row update + the `collection_install_runs` row in the same transaction.
   - `PatchFromWorkerAsync` updates the run row, and **if** the patch carries `version`+`installPath` (Succeeded install) or `status=Succeeded` on a removal, also updates the associated `host_collections` row in the same transaction.

5. **DI registration** in `ConfigureServices.cs`: register all three services as scoped.

6. **Build**:
   ```
   cd vm-infra/cloud-manager/cloud-manager-api/src
   dotnet build CloudManager.sln -c Release
   ```
   Expect 0 errors.

7. **Append to `Retro/CHANGE-0002.md`** with one line per file added/modified.

## Definition of done
- Four DTO classes, one mapping profile, three service interfaces + implementations, DI updated.
- `dotnet build` clean.
- `Retro/CHANGE-0002.md` exists.
- No endpoints yet (Phase 3 owns that); compilation proves the surfaces line up.
