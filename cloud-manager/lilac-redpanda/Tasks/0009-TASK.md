# Task 0009 — Playwright e2e verification

## Goal
Walk all six acceptance criteria from PLAN.md against the deployed UI. Capture proof under `Results/`.

## Target repo(s)
None (verification only).

## Steps

Each flow uses Playwright MCP against `https://ubuntu-server.talkersoft.com/ansible/collections`. Capture a screenshot after each major state transition. Verify status against the DB via `psql`.

### Flow A — Add catalog row
1. Navigate to `/ansible/collections`.
2. Click `+ Add collection`.
3. Fill modal: namespace=`community`, name=`general` (a third one not seeded, to prove add flow works), description=`General modules`. Submit.
4. Confirm a new row appears with status "Not installed".
5. Capture `Results/flow-a-catalog-add.png`.
6. DB check: `SELECT public_id, namespace, name FROM ansible.collections WHERE name='general';` → exactly one row.

### Flow B — Install
1. Click `Install` on the `community.postgresql` row.
2. Within 5s, status should show `Pending` or `Installing`.
3. Poll until terminal (~30s). Status should reach `Installed`. Capture `Results/flow-b-installed.png`.
4. DB checks:
   - `SELECT install_status, version, install_path FROM ansible.host_collections WHERE collection_id = (SELECT id FROM ansible.collections WHERE name='postgresql');` → status `Installed`, version populated (e.g. `4.2.0`), install_path under `/usr/share/ansible/collections/ansible_collections/community/postgresql`.
   - `SELECT status, exit_code FROM ansible.collection_install_runs ORDER BY started_at DESC LIMIT 1;` (or via the API) → status `Succeeded`.
5. Filesystem check: `sudo ls /usr/share/ansible/collections/ansible_collections/community/postgresql` → tree exists.

### Flow C — Remove
1. Click `Remove` on the same row.
2. Status flips `Removing`, then `Removed` within 5s.
3. Capture `Results/flow-c-removed.png`.
4. DB check: `SELECT install_status, version FROM ansible.host_collections WHERE …;` → status `Removed`, version cleared.
5. Filesystem check: `sudo ls /usr/share/ansible/collections/ansible_collections/community/postgresql` → "No such file or directory".

### Flow D — Reinstall after Removed
1. Click `Install` on the same (now Removed) row.
2. Status flips Pending → Installing → Installed. Capture `Results/flow-d-reinstalled.png`.

### Flow E — Install the second seeded collection
1. Click `Install` on the `community.hashi_vault` row. Wait for Installed.
2. Filesystem check: both collections present under `/usr/share/ansible/collections/ansible_collections/community/`.

### Flow F — Run `pg` end-to-end
1. Navigate to `/ansible/playbooks`, click pencil to edit `pg`, save the existing draft (no edit needed — just trigger a save so the revision is the latest). Close.
2. Navigate to `/ansible/run`, choose `pg`, choose `postgres-test`, click Run.
3. Wait for the run to either succeed or fail. Capture the resulting status + last 30 lines of output.
4. **Acceptance**: the failure mode is no longer "module not found: community.postgresql.postgresql_db". Whatever happens next (vault-token issues, ssh, etc.) is out of scope for this workflow — but the collection-related failure is gone.

### Capture artifacts
- All screenshots above
- `Results/verify.md` — short bullet list per flow with URLs observed and the matching DB / FS check result
- `Results/pg-final-status.md` — what happened on the `pg` run

## Definition of done
- All flows A–E land at the expected status.
- `pg` run no longer fails with "module not found".
- All screenshots + `verify.md` in `Results/`.
- No source code edits during this phase.
