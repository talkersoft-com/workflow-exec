# Task 0001 ‚ÄĒ vorch-lib: version-aware secret seal (jammy 249 AND noble 255)

Repo: `vorch-lib`. The secret-injection path is `injectSecrets` in `provision/provision.go`, which
generates the on-VM `seal-credentials.sh` that runs at boot. Today that script unconditionally runs
`systemd-creds encrypt` ‚Üí `/etc/credstore.encrypted/<name>`. `systemd-creds` ships only with
**systemd ‚Č•250**; Ubuntu 22.04 has **systemd 249**, so on jammy the encrypt fails **silently**, the
credstore stays empty, and **nothing is injected**. (Locate the seal generation precisely from the
codebase ‚ÄĒ the secret VALUE rides base64 in `write_files` under `/run/cm-seed`; the seal runcmd line
runs the generated script. The cert/`x-secret-path` path is unrelated and must not be disturbed.)

## Root cause (verified ‚ÄĒ do NOT re-investigate)
The API resolves the binding correctly (a `vsb_‚Ä¶` row with `resolvedVaultPath`), the base64 secret +
`seal-credentials.sh` land in `/run/cm-seed`, but `systemd-creds encrypt` is MISSING on systemd 249
‚Üí seal fails silently ‚Üí `/etc/credstore.encrypted` empty ‚Üí no injection. Second defect: on failure
the plaintext password is left in `/run/cm-seed` (the shred never runs) and nothing is surfaced.

## Steps
1. **Make the generated `seal-credentials.sh` version-aware.** At boot it detects the guest's
   systemd version (e.g. `systemctl --version | head -1 | awk '{print $2}'`, or
   `systemd-creds --version`) and branches:
   - **systemd ‚Č•250 (noble):** keep `systemd-creds encrypt` ‚Üí `/etc/credstore.encrypted/<name>`
     (the consuming unit uses `LoadCredentialEncrypted`). Unchanged behaviour.
   - **systemd <250 (jammy):** fall back to a **root-only** credstore file ‚ÄĒ
     `install -m 0600 -o root -g root` the decoded secret to `/etc/credstore/<name>` (the consuming
     unit uses `LoadCredential=<name>:/etc/credstore/<name>`; `LoadCredential` exists since systemd 247).
     The fallback file MUST be `0600 root:root` and live on local disk (not world-readable, not `/run`).
2. **Always shred `/run/cm-seed`** ‚ÄĒ run the shred/cleanup of every seeded plaintext and the script
   itself in a `trap ... EXIT` (or equivalent) so it runs on **success AND failure**, not only the
   happy path. After boot nothing decoded/plaintext remains under `/run/cm-seed`.
3. **Fail loud.** A seal failure (missing tool on an unexpected version, encrypt error, write error)
   must exit **non-zero** and surface on the run (stderr / a marker the run captures) ‚ÄĒ never silent.
   Do **not** print the secret value while doing so (no plaintext in logs or run output).
4. **Scope discipline.** `vorch-lib` only; do not change the API/web/DB schema, the binding
   resolution, or the cert/`x-secret-path` mechanism. The empty-binding user-data path stays
   byte-stable. `vorch-service` source is untouched here (it is rebuilt + redeployed in Task 0003).
5. **Test (golden/unit of the generated user-data / seal script).** Add or extend a provision test
   (e.g. `provision/provision_secrets_test.go`) that renders the seal script/user-data and asserts:
   the ‚Č•250 branch emits `systemd-creds encrypt` ‚Üí `/etc/credstore.encrypted/<name>`; the <250 branch
   emits the root-only `0600` `/etc/credstore/<name>` write with `LoadCredential` consumption; the
   shred of `/run/cm-seed` is unconditional (present in the EXIT/trap path); a seal failure is
   non-zero; and **no secret value** appears in plaintext anywhere in the rendered script except the
   base64 `write_files` blob that is the documented transport. `go build ./...` and
   `go test ./provision/...` green.

## Definition of done
The generated seal script is version-aware (‚Č•250 `systemd-creds encrypt`/`credstore.encrypted`;
<250 root-only `0600` `/etc/credstore/<name>`), always shreds `/run/cm-seed`, and fails loud; the
provision golden/unit test covers both branches + shred + loud-failure + no-plaintext-leak;
`go build ./...` + `go test ./provision/...` green. No API/web/DB changes; cert path untouched.
