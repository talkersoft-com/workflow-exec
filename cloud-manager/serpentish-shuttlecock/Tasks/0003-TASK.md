# Task 0003 ŌĆö E2E verify + build vorch + manual redeploy + ship

## Steps
1. **Build vorch:**
   - `cd vm-infra/cloud-manager/vorch-lib && go build ./... && go test ./provision/...` green.
   - `cd vm-infra/cloud-manager/vorch-service && ./scripts/build-app.sh` green.
   (The `.cicd/build-cloud-manager.py` script targets api/mcp/web only ŌĆö it does NOT build vorch.)
2. **Manual vorch-service redeploy on the host (`ubuntu-server`)** so the running provisioner emits
   the new version-aware seal. The standard deploy script does not cover vorch, and `deploy-app.sh`
   cannot ssh-to-self ŌĆö install/run the freshly built binary **locally on the host** and restart
   `vorch_service` (`sudo systemctl restart vorch_service`). Record the **exact** commands in
   `Results/RESULT.md`.
3. **Prerequisite check (already live in the CM instance):** the published blueprints
   `database-ready-web-server-jammy` (22.04) and `database-ready-web-server-noble` (24.04) each with
   the `haproxy-tls-proxy` playbook + the `DBPASSWD` (`pg-consumer-binding`) secret binding; the
   `wildcard-talkersoft-com` Vault secret; the `vault-token` systemd credential on the host
   (re-seed with `cloud-manager-api/tests/seed-vault-token.sh` if missing).
4. **Run the acceptance test on ubuntu-server** (the sole definition of done):
   `cloud-manager-api/tests/test_secret_injection.py`
   - The token comes via the systemd-sealed `vault-token` credential.
   - **Both** cases ŌĆö jammy (22.04) AND noble (24.04) ŌĆö must print **PASS** and the script must exit
     **0**. Each case provisions from its blueprint with `DBPASSWD` pointed at a Postgres VM, boots
     with the credential injected, and the credential-only app serves `Database connection: SUCCESS`
     through haproxy on 443.
   - Paste the final **RESULT block** (both cases PASS, exit 0) into `Results/RESULT.md`.
5. **Confirm no plaintext remains** in `/run/cm-seed` on **both** VMs after boot (SSH + check the
   directory is empty / no decoded secret files). Record the evidence in `Results/RESULT.md`.
   Confirm the jammy fallback file `/etc/credstore/DBPASSWD` is `0600 root:root`.
6. **Write outputs BEFORE shipping:**
   - `Results/RESULT.md` ŌĆö outcome, branch, PR table (filled from hv_integrate), per-phase summary,
     the exact vorch build + manual redeploy commands, the pasted `test_secret_injection.py` RESULT
     block (both PASS, exit 0), and the `/run/cm-seed` no-plaintext evidence for both VMs.
   - `Retro/LESSONS.md` ŌĆö what would have helped + FIX file list.
7. **Ship:** `hv_integrate` per `../deck.md`. A failed build / redeploy / test is a stop condition ŌĆö
   write a FIX file under `Retro/`, resolve, retry.

## Definition of done
vorch-lib + vorch-service build green; vorch-service manually redeployed on the host (exact commands
recorded); `test_secret_injection.py` exits 0 with **both** jammy and noble PASS, run on
ubuntu-server; no plaintext left in `/run/cm-seed` on either VM and the jammy fallback file is
