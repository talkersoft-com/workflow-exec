# Task 0002 — sample consumption per-OS (hello-db-web)

Repo: `cloud-manager-api`, dir `tests/hello-db-web/`. The sample app stands in for how a real app
consumes the injected credential. Today `hello-db-web.service` hardcodes
`LoadCredentialEncrypted=DBPASSWD` — that ONLY works on systemd ≥250 (noble). On jammy (systemd 249)
there is no encrypted credstore; the unit must instead use
`LoadCredential=DBPASSWD:/etc/credstore/DBPASSWD` (the root-only `0600` fallback that Task 0001 seals).
Either way the app reads `$CREDENTIALS_DIRECTORY/DBPASSWD` — only the unit directive differs.

## Steps
1. **Select the credential directive by the VM's systemd version at deploy time.** In `deploy.sh`,
   detect the target VM's systemd version over SSH (e.g.
   `ssh … 'systemctl --version | head -1 | awk "{print \$2}"'`) and install the matching unit:
   - **≥250 (noble):** `LoadCredentialEncrypted=DBPASSWD` (reads `/etc/credstore.encrypted/DBPASSWD`).
   - **<250 (jammy):** `LoadCredential=DBPASSWD:/etc/credstore/DBPASSWD` (reads the root-only `0600` file).
   Implement this by templating the one directive into `hello-db-web.service` before `scp`/install
   (keep a single source unit with the directive substituted), OR ship two variants and pick one —
   choose the cleaner shape; the rest of the unit is identical.
2. **`server.js` stays credential-only.** It already reads `$CREDENTIALS_DIRECTORY/DBPASSWD` and sets
   `PGPASSWORD` from it; do not add an env-password code path, a Vault fetch, or a hardcoded
   password. Leave the credential-only behaviour intact (it handles both bare-password and JSON forms).
3. **No password on the wire from deploy.** `deploy.sh` must NOT require `PGPASSWORD` when the binding
   injects the credential — keep `PGPASSWORD` strictly optional/fallback (it already is). The `.env`
   it writes stays non-secret connection settings only (`0600 root`), no password unless an explicit
   fallback is passed.
4. **Scope discipline.** `tests/hello-db-web/` only. Do **not** touch `tests/test_secret_injection.py`
   assertions/gate. No API/web/DB-schema changes.
5. **Test (static / shellcheck-level).** Confirm: the deploy path produces
   `LoadCredentialEncrypted=DBPASSWD` for a ≥250 target and
   `LoadCredential=DBPASSWD:/etc/credstore/DBPASSWD` for a <250 target; `server.js` has no
   env-password/hardcoded/Vault path; `deploy.sh` keeps `PGPASSWORD` optional and writes no secret
   into `.env` by default. `bash -n deploy.sh` (and `shellcheck` if available) clean.

## Definition of done
`deploy.sh` installs the unit with `LoadCredentialEncrypted` on systemd ≥250 and `LoadCredential` on
<250, chosen from the VM's detected version; `server.js` remains credential-only; `PGPASSWORD` stays
optional; `tests/test_secret_injection.py` is untouched; `bash -n` clean.
