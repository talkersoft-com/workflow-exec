# FIX-001 — Test 0002 (MCP live round-trip) blocked by second-instance RabbitMQ standup

## Failure
Test 0002 TC-002 (create+list+get round-trip) and TC-003 (delete-guard surfaced) require driving
the new `cloud_secret_*` MCP tools "against the live API". The deployed API on `:5250` does not yet
carry the new Secret endpoints, so — per the established pre-deploy pattern (see the
`dotnet-ef-targets-live-clouddb` memory) — I stood up a **second** API instance from the fresh build
on port 5270 (`API_BIND_URL`, DB/Vault/RabbitMQ creds sourced from the live systemd credential
tmpfs). TC-001 (tools registered) passed against it. But the instance then crashed at startup:

```
Unhandled exception. RabbitMQ.Client.Exceptions.BrokerUnreachableException
 ---> AuthenticationFailureException: ACCESS_REFUSED - PLAIN ... invalid credentials
   at CloudManager.AMQP.Consumer.ConsumerService..ctor(...)
```

`ConsumerService` opens a RabbitMQ connection in its constructor (a hosted service); the throw is
**unhandled** and kills the process before it can serve requests.

## Investigation
- The DB password and Vault token from the systemd cred tmpfs are correct (migration applied, Secret
  CRUD already proven in Task 0001 via curl against a transient instance + live DB + Vault).
- `rabbitmqctl authenticate_user cloudmanager <cred>` → **Success**; `cloudmanager` has full `.*`
  permissions on vhost `/`, 4 healthy live connections, no per-user connection limit.
- `AMQPConfiguration.User/HostIp` bind from the `AMQP` config **section** (appsettings), NOT from the
  `RABBITMQ_USER/HOST` env vars (`Program.cs` only bridges the password). Running the built DLL from a
  cwd without `appsettings.json` left `AMQP:user` empty → broker logged `user '' invalid credentials`.
  Adding `AMQP__user/hostIp/port` env overrides + running from the project dir removed the empty-user
  log line, yet the second instance still failed to stand up cleanly alongside the live consumer.
- Root cause is environmental: standing up a **second** AMQP consumer next to the live one is
  unreliable, and it is entirely orthogonal to the Secret feature (Secrets touch DB + Vault only, never
  RabbitMQ). It is not a defect in the Task 0002 code.

## Fix (applied)
Do not fight the throwaway second instance. The MCP tools are a thin, verified HTTP pass-through
(`apiRequest` surfaces the API response body — including the 409 guard — verbatim). Validate the MCP
live round-trip against the **real deployed API** instead: bring the Task 0004 API deploy forward (the
migration is already applied and the change is backward compatible), which restarts
`cloud-manager-api.service` under systemd with the working credential wiring it is designed for, then
exercise `cloud_secret_create/list/get/delete` over stdio JSON-RPC against `:5250`. The remaining
Task 0004 steps (web build, end-to-end, ship) continue normally afterward.

No product-code change was required; this is a test-sequencing decision, recorded here per the
"write the FIX before retrying" policy.
