# Task 0001 — API: Secret entity, migration, CRUD + state machine + path-lock

## Goal
Add the CM-owned `Secret` entity, the `VmSecretBinding` usage entity, and a nullable `secret_id`
FK on `SecretBinding`, in **one reversible migration**, plus Secret CRUD endpoints driven by the
strict DB-first create/delete state machine and the three-layer path-lock. Mirror the existing
`secret_bindings` service/controller code paths.

## Reference
`../../../workflow-plans/cloud-manager/wheezy-cottonmouth/PLAN.md` — Design section is authoritative.

## Steps

### 1. EF entities
- **`Secret`** (schema `marketplace`, table `secrets`): `id` uuid PK (internal), `public_id`
  varchar(32) `sec_…`, `name`, `description`, `vault_path` text, `origin` varchar(32) (default
  `manual`), `status` varchar(32), `config` jsonb, `created_at`, `updated_at`, `deleted_at`
  (soft-delete; add the global query filter `DeletedAt == null`).
- **`VmSecretBinding`** (schema `vm`, table `vm_secret_bindings`): `id` uuid PK, `public_id`
  `vsb_…`, `virtual_machine_id` uuid FK → `vm.virtual_machines`, `secret_binding_id` uuid FK →
  `marketplace.secret_bindings` **ON DELETE RESTRICT**, `cred_name` varchar, `created_at`.
- **`SecretBinding`**: add `secret_id` uuid **NULL** FK → `marketplace.secrets(id)`
  **ON DELETE RESTRICT**. Do not touch existing columns.

### 2. One reversible migration
A single migration that:
- creates `marketplace.secrets` (incl. the `vault_path` CHECK below + a unique index on `vault_path`),
- creates `vm.vm_secret_bindings`,
- adds `secret_bindings.secret_id` + both RESTRICT FKs.
- **`CHECK (vault_path LIKE 'cloudmanager/data/%')`** on `secrets`.
- `Down()` cleanly reverses all of it. Verify both directions apply with no error.

### 3. Path-lock — single source of truth
Define one constant/helper for the owned prefix (`cloudmanager` mount + `data/`). The API composes
`cloudmanager/data/manual/{slug}` from an operator-supplied slug (normalize: lowercase, `[a-z0-9-]`,
collapse repeats). **Reject** any computed/supplied path not under the owned prefix (defensive —
the DB CHECK is the backstop). The operator never supplies a full path.

### 4. Secret create/delete state machine (strict, DB-first)
- **Create** (`POST`): validate name + slug → compose `vault_path` → `INSERT status=Pending` →
  write value(s) to Vault (KV v2 under the `cloudmanager` mount, reusing the existing controller
  Vault token path) → `UPDATE status=Active`. On Vault write failure → `status=Failed`, return
  502/500 with a clear message (the DB row stays as the record of intent — no unknown orphan).
- **Delete** (`DELETE`): **guard** — if any **live** (`deleted_at IS NULL`) `SecretBinding` has
  `secret_id = this`, return **409** with a message naming the blocking binding(s). Else →
  `status=Deleting` → delete from Vault → `status=Deleted` + stamp `deleted_at`. On Vault delete
  failure → stay `Deleting`, return 502/500 (retryable).
- **List / Get**: standard, camelCase, public_ids only. **Never** return the secret value.

### 5. Conventions
camelCase JSON; public_ids only (use `PublicIdByGuidAsync` / `GuidByPublicIdAsync`); soft-delete
authoritative; secret values never logged or returned. Mirror `secret_bindings` controller/service.

## Build
`python3 .cicd/build-cloud-manager.py --target api` must pass cleanly.

## Definition of done
Entities + one reversible migration (up/down clean) + Secret list/get/create/delete with the state
machine and path-lock, building cleanly. Test 0001 passes.
