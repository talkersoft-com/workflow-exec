# Task 0002 — Diagnose the failure layer + root cause

## Goal
From the captured evidence in `Retro/REPRODUCTION.md`, narrow the failure to a single file:line in a single repo and document why it fails. Output: `Retro/DIAGNOSIS.md`.

## Target repo(s)
None (analysis-only phase — no source changes).

## Steps

1. **Read every artifact** in `Retro/` (REPRODUCTION.md, api.log, vorch.log, rmq.log, network.*, console.txt, curl.txt, screenshots).

2. **Use the symptom→layer table** from `PLAN.md`:

   | Symptom | Likely layer |
   |---|---|
   | 400 with validation message | web (wrong request shape) or API (DTO mismatch) |
   | 404 on the create route | web route misconfig — confirm `/api/v1/virtualmachine/create/{hostId}` |
   | 500 with `Newtonsoft.Json` or `NullReferenceException` | API model binding or controller |
   | 500 with `Npgsql` | DB constraint or schema drift |
   | 200 from API but VM never appears | AMQP publish failed silently OR vorch handler crashed |
   | Vorch journal shows panic / libvirt error | vorch-lib provision or infra |

3. **Walk the flow** from the captured evidence:
   - Did the request reach the API? (look for the route in `api.log`)
   - Did the controller return successfully? (look for the response code in `api.log`)
   - Did RMQ receive the publish? (look in `vorch.log` for a "received message" line — vorch logs this on consume)
   - Did the vorch handler complete? (look for "create VM completed" or a panic / error)
   - Did libvirt error? (look for `virt-install` or `qemu-img` failures in `vorch.log`)

4. **Locate the offending source.** Use `grep` / `Read` to find the file:line that produced the error. Examples:
   - For an API exception, find the throw site in `cloud-manager-api/src/CloudManager.API/` or the service layer.
   - For a vorch panic, find the line in `vorch-service/handlers/vm_handlers.go` or `vorch-lib/provision/`.
   - For a request-shape mismatch, compare:
     - `cloud-manager-web/src/services/api.ts` `vms.create` body
     - `cloud-manager-api/src/CloudManager.API/Controllers/VirtualMachineController.cs` Create action signature + bound DTO
     - The entity model and migration history

5. **Cross-reference recent commits** that touched the offending file:
   ```
   git -C <repo> log -20 --oneline -- <file>
   ```
   If a recent commit introduced the regression, note it.

6. **Write `Retro/DIAGNOSIS.md`** with:
   - Failure layer (web / api / amqp / vorch / vorch-lib / infra / db)
   - Offending file path and line number(s)
   - Why it fails (one paragraph, in plain English)
   - The proposed minimal change (one paragraph; specific edits, not "refactor X")
   - Which repo(s) need PRs
   - Whether this is a regression (and from which commit) or a long-standing issue

## Definition of done
- `Retro/DIAGNOSIS.md` exists and names:
  - The failure layer
  - At least one specific file:line
  - The minimal change to make
  - The list of repos that need PRs (could be a single repo)
- No source code under `vm-infra/cloud-manager/` has been modified.
