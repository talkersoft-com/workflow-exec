# Lessons: cloud-manager/phantom-papaya

## What would have helped

1. **Vorch should log to journal, not a separate file.** Vorch's `app.go:16-31` redirects `log.SetOutput` to `/kvm-automator/provision.log` when `RUNNING_AS_SERVICE=true`. That meant `journalctl -u vorch_service` returned `-- No entries --` for every reproduction window — a huge red herring during diagnosis. Costing 5+ tool calls to discover where the real log lives. Recommendation: when `StandardOutput=journal` is set in the systemd unit (which it already is for vorch_service), don't redirect Go's log output to a file. Let it go to stdout/stderr → journal naturally. The current logic forces operators to know about two log destinations and the env var that toggles between them.

2. **Vorch had no startup-time Vault smoke test.** The token had been revoked for an unknown duration before this bug was reported. Vorch's existing healthprobe loop runs every 5 minutes (visible in `provision-tail.log` checking ansible-runner and a feature-flag endpoint) but does NOT exercise Vault at all. A single `vault PUT cloudmanager/data/_healthcheck/probe` per probe cycle would catch a revoked token in 5 minutes instead of N hours/days. Recommendation (follow-up plan, NOT this bug): add a Vault probe to `vorch-service/healthprobe/probe.go`.

3. **`config_loader.py`'s file-priority order is invisible to operators.** It searches `/etc/cloud-manager/cloud-manager-config.env` first and falls back to `project_root/config/cloud-manager-config.env`. I edited the repo file expecting `install-vorch-service.py` to use it; the installer instead read from `/etc/cloud-manager/`. Wasted one redeploy cycle. Recommendation: have `install-*.py` print which config file it loaded from BEFORE generating component configs, so the operator sees `Reading config from: /etc/cloud-manager/cloud-manager-config.env` and knows where the source of truth is.

4. **`messagepub.PublishSuccessMessage` is misleadingly named.** It publishes the outcome envelope regardless of whether the command succeeded — it just means "the publish call succeeded." The matching log line `"Successfully published success message"` (messagepub.go:34) printed AFTER a `Failed` status caused confusion during diagnosis. Not in scope for this bug but worth a rename — e.g. `PublishOutcomeMessage` and `"Outcome envelope published"`.

5. **No mechanism connects the cloudmanager-policy token's existence on disk to the config env file.** The valid token at `/home/todd/cloudmanager_vault_token.txt` was generated by `vault-cli cloudmanager` (per `cli/cloudmanager.py`) on `2026-05-19`. There's no automation that picks that file up and writes it into `cloud-manager-config.env`. Operators have to remember to copy-paste it manually. Recommendation: `vault-cli cloudmanager` could optionally write the new token into the env file, OR `install-vorch-service.py` could check `/home/todd/cloudmanager_vault_token.txt`'s mtime against the config env's VAULT_TOKEN line and warn on staleness.

## Fix files written
None. The reproduction → diagnosis → fix → verify cycle ran without test failures requiring a `FIX-NNN.md`. The one operational hiccup (initial vorch redeploy didn't pick up the new token because I had only edited the repo file, not `/etc/cloud-manager/`) was recovered inline by running `deploy-config.py`; not logged as a FIX file because no Test failure was on the line — Task 0004 step 1 hadn't been claimed `done` yet when the mistake was caught.

## Regression analysis
Not a code regression. The bug is a stale credential in committed config, introduced when Vault was re-bootstrapped at some point after `cloud-manager-config.env` was last updated. The currently-valid token in `/home/todd/cloudmanager_vault_token.txt` was issued `2026-05-19` and was never propagated back into the env file. Adding a Vault probe to vorch's healthprobe loop (see #2 above) would have surfaced this within 5 minutes of Vault re-init.
