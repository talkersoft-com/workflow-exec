# Task 0004 — vorch: cloud-init injection + first-boot seal

## Goal
In the guest, deliver each secret via cloud-init and seal it into a systemd encrypted credential on
first boot, leaving no plaintext behind.

## Reference
`../../../workflow-plans/cloud-manager/primordial-crystalball/PLAN.md` — build item #4; the
shell-injection guardrail at `vorch-lib/provision/provision.go:31`.

## Steps
- Extend `InitializeHost` (`vorch-lib/provision/provision.go:204`) with the resolved secrets
  (threaded from `vorch-service`'s create-vm consumer) and extend the user-data builder
  (`fmt.Sprintf` template at ~line 313).
- For each secret, **validate `credName`** against a strict charset (mirror `publicIdPattern` /
  `validatePublicId`) before using it in any path.
- Add a `write_files` entry per secret: write the **base64** value to `/run/cm-seed/<credName>`,
  mode `0600`, owner root. (`/run` is tmpfs — never hits the guest disk.)
- Add `runcmd` (or a single first-boot script written via `write_files` + one `runcmd` line) that,
  per secret: `base64 -d /run/cm-seed/<credName>` → `systemd-creds encrypt --with-key=host
  --name=<credName> - /etc/credstore.encrypted/<credName>` (create dir `0700`), then
  `shred -u /run/cm-seed/<credName>`.
- **Secret values never appear in `runcmd` args or logs** — the seal script reads the file; use the
  redact writer if any new log path prints command context.
- **No-secret path unchanged:** when the list is empty, the generated user-data is byte-identical to
  today (no `cm-seed` write_files, no extra runcmd).

## Build
`vorch-lib` + `vorch-service` Go builds clean.

## Definition of done
Generated user-data (unit-level/golden test) contains the write_files + seal runcmd for a sample
secret and is unchanged when empty; credName validation rejects bad names; no value in logs; builds
pass. Test 0004 passes. (Full boot-seal proof happens in Task 0006 E2E.)
