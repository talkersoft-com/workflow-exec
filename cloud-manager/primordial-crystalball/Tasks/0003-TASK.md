# Task 0003 — Contract: create-vm secrets payload (C# + Go lockstep)

## Goal
Carry the resolved secrets from the controller to vorch by extending the create-vm command on both
sides of the wire, in YAML lockstep.

## Reference
`../../../workflow-plans/cloud-manager/primordial-crystalball/PLAN.md` — build item #3.

## Steps
- **C#** (`cloud-manager-api/src/Models/CloudManager.Messages/`): add
  `List<SecretCredData> Secrets` to `VmCommandData`, where `SecretCredData { string CredName;
  string B64Value; }`. Populate it from Task 0002's resolved list at publish.
- **Go** (`vorch-lib/models/templates/createvm/` + `vorch-lib/models/messages/createvmmessage.go`):
  add the matching `Secrets []SecretCred` field to `createvm.VM` (or `CommandData`, matching where
  the C# field lands in the YAML tree) with **identical YAML tags** (`secrets`, `credName`/
  `cred_name`, `b64Value`/`b64_value` — pick one casing and match exactly on both sides).
- **Backward compatible:** an omitted/empty `secrets` deserializes to no secrets on the Go side; a
  VM with none provisions as today.
- Confirm the YAML round-trips: a message serialized by C# deserializes on Go with the secrets
  intact (and vice-versa for the test).
- **Values are base64 (not encrypted)** — sealing is host-bound and happens in the guest (Task 0004).
  Do not log the payload.

## Build
`python3 .cicd/build-cloud-manager.py --target api` clean; `vorch-lib` Go build clean.

## Definition of done
Both structs carry `Secrets` with matching YAML; a populated message round-trips C#↔Go; empty/absent
is backward compatible; builds pass. Test 0003 passes.
