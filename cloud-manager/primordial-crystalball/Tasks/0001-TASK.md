# Task 0001 — API: Vault KV read

## Goal
Give the controller the ability to READ a KV v2 secret value from the `cloudmanager` mount — the
capability that does not exist today (`IVaultClient` has only policy/token methods).

## Reference
`../../../workflow-plans/cloud-manager/primordial-crystalball/PLAN.md` — build item #1.

## Steps
- Add `Task<IReadOnlyDictionary<string,string>> ReadKvSecretAsync(string path)` to
  `IVaultClient` (`cloud-manager-api/src/Services/CloudManager.Vault.Client/IVaultClient.cs`) and
  implement in `VaultClient.cs`.
- KV v2 read semantics against the `cloudmanager` mount (path like
  `cloudmanager/data/...`; return the `data.data` field map). Reuse the existing controller Vault
  token configuration (same auth path the other VaultClient methods use).
- On 404 → a clear typed error (so resolution can distinguish "secret missing" from transport
  failure). On other non-2xx → `VaultException` (existing pattern).
- **The secret value is NEVER logged** — no value (or full response body containing it) in any log
  statement.

## Build
`python3 .cicd/build-cloud-manager.py --target api` must pass cleanly.

## Definition of done
`ReadKvSecretAsync` reads a known `cloudmanager/data/...` path and returns its fields; 404 surfaces
distinctly; no value logged; api builds. Test 0001 passes.
