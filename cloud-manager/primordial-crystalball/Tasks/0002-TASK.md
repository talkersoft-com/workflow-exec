# Task 0002 — API: resolve attached bindings at instantiate

## Goal
At blueprint instantiate, turn a blueprint's attached Secret Bindings into resolved
`{ credName, b64Value }` pairs by computing each concrete Vault path and reading the value.

## Reference
`../../../workflow-plans/cloud-manager/primordial-crystalball/PLAN.md` — build item #2.

## Steps
- In the instantiate path (`MarketplaceController` → `ProvisionService` / `BlueprintRunSequencer` —
  locate where the create-vm command is assembled), load the blueprint's attached
  `marketplace.blueprint_secret_bindings` (with `cred_name`, `position`).
- For each binding compute the concrete path:
  - **static** (`secret_id` set) → the referenced `Secret.vault_path`,
  - **templated** (`secret_id` NULL) → substitute `{vm_public_id}` (and any other `params`) from
    the provision context into `path_template`. (Only `vm_public_id` is expected for the first cut.)
- `ReadKvSecretAsync` each path → select the credential field (default: a single named field, e.g.
  `password`; honor the field convention decided in the plan's Open Questions) → base64-encode.
- Assemble an ordered `[{ credName, b64Value }]` list and hand it to the command-build step
  (Task 0003 carries it on the wire).
- **No bindings → empty list** (must not change today's behavior). A missing/404 secret is a
  provision error surfaced clearly (do not silently skip).
- camelCase; public_ids only; **value never logged**.

## Build
`python3 .cicd/build-cloud-manager.py --target api` must pass cleanly.

## Definition of done
Instantiating a blueprint with an attached binding yields the correct resolved list (verified with
a test secret); no-binding blueprint yields empty; api builds. Test 0002 passes.
